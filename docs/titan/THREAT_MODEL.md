# Threat Model (SEC-1)

STRIDE analysis, a security findings register, and penetration-test preparation for Titan. Surfaces 1-4 (public assessment flow, Worker API, D1, Auth.js) already have a complete, evidenced STRIDE table in `SECURITY_GUIDE.md`'s "Threat model (STRIDE)" section — not duplicated here. This document adds the two surfaces that didn't exist when that table was written (OPS-1's observability/alerting code, and the CI/CD pipeline itself), then a consolidated findings register and pentest-preparation section covering the whole system.

## Surface 5 — Observability & Alerting (OPS-1)

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing | A caller tries to inject fabricated values into `ctx.metrics` to trigger or suppress an alert | `Metrics.increment`/`.recordDuration` are called only from within `router.ts` itself, populated from the real `method`/`path`/`status`/duration of the actual request being handled — no code path accepts a client-supplied tag or count. A caller can only influence metrics by making real requests (already governed by rate limiting and authentication) | Mitigated by construction — no input path exists to forge a metric value |
| Tampering | `alerts`/`requestSummary` are modified in transit | Same TLS-once-deployed + `GET /api/operations/summary`'s existing authentication/authorization boundary as every other field on that response; no new transport | Mitigated (inherited, not new) |
| Repudiation | An alert fires with no record of when/why | A structured `"operational alert evaluated"` log line (`warn`/`error`, matching severity) on every non-empty `evaluateAlerts` result, correlated by `requestId` | Mitigated — real, verified in test output during OPS-1's own final CI run |
| Information disclosure | `Alert.evidence` reveals internal operational state (error rates, config issues, which service is unreachable) | `GET /api/operations/summary` is Platform-Administrator-only (unchanged, EAP-7's own gate) — the same disclosure posture `ServiceStatus.error`'s real exception text already established for this identical trust boundary | Mitigated — no new exposure, same caller who could already see runtime metrics/service errors |
| Denial of service | Computing aggregates/alerts on every call is used to slow the endpoint | `computeLatencyPercentiles`/`computeErrorRate`/`evaluateAlerts` are pure, synchronous, O(n) over data already held in memory — no new I/O beyond the one additional `computeReadiness` call (a single `SELECT 1`) `operationsSummary` now performs, on an already Platform-Administrator-gated, low-frequency endpoint | Mitigated — negligible added cost, gated behind the same privileged caller |
| Elevation of privilege | A caller reaches alert data without the Platform Administrator role | No new authorization concept — `requirePlatformAdministrator`, unchanged | Mitigated (inherited) |

## Surface 6 — CI/CD pipeline

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing | A malicious fork's pull request triggers `titan-ci.yml` to exfiltrate repository secrets | `titan-ci.yml` runs on `pull_request` (not `pull_request_target`) — GitHub's own default behavior withholds repository secrets from a fork's PR runs; the workflow references no secret at all, only the ambient `GITHUB_TOKEN`, now explicitly scoped to `contents: read` (**SEC-1**) | Mitigated — GitHub's own platform guarantee, reinforced by explicit least-privilege permissions this phase added |
| Tampering | A compromised transitive dependency achieves code execution during `npm ci`/build and tampers with build output or exfiltrates CI-scoped credentials | `package-lock.json` is lockfile v3 with full integrity hashes (deterministic installs); explicit least-privilege `permissions: contents: read` (**SEC-1**) limits what a compromised job could do with the ambient token even if it tried; the two jobs that touch real Cloudflare credentials (`migrate-database`, `deploy-worker`) sit behind `workflow_dispatch` (a human must choose to trigger a run) and GitHub's own `environment:` gate | Partially mitigated — the `environment:` gate's real protection (required reviewers, wait timers) depends on a repository *setting* this codebase cannot commit (`SECURITY_GUIDE.md`'s PRD-1 entry, carried forward as finding SEC-1-10 below) |
| Repudiation | A workflow run's real history is altered or hidden | GitHub Actions' own run history and logs — outside this repository's control, a platform guarantee, not something this codebase implements | Mitigated (platform-provided) |
| Information disclosure | A workflow step prints a secret value | `validate-secrets.mjs` (PRD-1) validates secret *names* only, never reads or prints a value; neither workflow file echoes any `secrets.*` reference directly | Mitigated |
| Denial of service | Not a meaningful threat for this repository's own CI (no shared multi-tenant resource this project controls) | N/A | Not applicable |
| Elevation of privilege | Any repository collaborator with `workflow_dispatch` permission can attempt a real deploy | `environment: ${{ inputs.target_environment }}` is meant to require GitHub Environment protection rules (required reviewers) — not yet configured (external, repository-setting action, `SECURITY_GUIDE.md`'s PRD-1 entry) | **Known, accepted, already-documented gap** — carried forward, not new to this phase |

## Security findings register

Every finding this phase evaluated, real-world risk classified in *this system's actual context* (not just a generic severity label), with status and evidence. IDs are stable references for future phases to track against.

| ID | Finding | Severity (this system's real context) | Status | Evidence |
|---|---|---|---|---|
| SEC-1-01 | 3 high-severity CVEs (libvips, via `sharp`) required transitively by `miniflare`→`wrangler` (a **devDependency**) | **Medium** — npm labels this "High" against `sharp`/libvips generically; in this system, `sharp` is 3 levels removed from any application code (confirmed absent from both `apps/web/src` and `packages/platform/src` by direct search), never bundled into the deployed Worker, and reachable only if the local dev/CI toolchain itself is already compromised — a materially narrower exposure than the raw label implies | **Deferred** — `npm audit fix --force` would downgrade `wrangler` from `4.112.0` to `4.15.2` (every version since `4.16.0` carries the identical transitive vulnerability, confirmed via `npm audit fix --force --dry-run`'s exact proposed change), a ~97-minor-version regression of a tool this project's entire local dev/CI/deploy pipeline depends on. Applying it without individually verifying every wrangler-dependent script/workflow still behaves correctly would be reckless, unvetted risk-taking, not a fix. Recommendation: monitor for a `wrangler`/`miniflare` release that resolves the `sharp` dependency without a downgrade; re-run `npm audit` periodically | `npm audit` (run fresh this phase), `npm ls sharp`, `npm audit fix --force --dry-run` |
| SEC-1-02 | `Content-Security-Policy` never set `frame-ancestors` on any route | **Low** — `X-Frame-Options: DENY` already fully mitigated clickjacking on every browser that honors it; this closed the defense-in-depth gap for the (shrinking) set of contexts that check CSP's `frame-ancestors` specifically | **Fixed this phase** — `finalizeResponse.ts`'s `STRICT_CSP` and `authPagesCsp` both now include `frame-ancestors 'none'`; regression test added (`router.test.ts`) | `finalizeResponse.ts`, `router.test.ts` |
| SEC-1-03 | Rate-limiting coverage was inconsistent with its own stated rationale: `POST /api/organizations` (authenticated) was rate-limited; `POST /api/portal/support`, `POST`/`PATCH /api/portal/commercial/subscription` (also authenticated, customer-facing) were not, despite the in-code comment claiming rate limiting exists "specifically for anonymous... routes" | **Low** — every route was still fully authenticated and CSRF-protected regardless; this was an inconsistency in defense-in-depth, not an open authorization gap | **Partially fixed this phase** — the three customer-facing routes now share the same 30/60s limiter `POST /api/organizations` already used (`SECURITY_ARCHITECTURE.md`'s rate limiting table has the full before/after). Remaining Platform-Administrator-only writes (`PATCH /api/leads/:id`, `PATCH /api/organizations/:id`, `/api/users/:id/profiles*`, `PATCH /api/commercial/subscriptions/:id`) deliberately left unrated-limited, consistent with each other now — an already-privileged, already-audited actor is a materially lower real abuse risk than a customer-facing session, and extending to seven more call sites for that marginal benefit was judged out of proportion | `router.ts`, `router.test.ts` (3 new rate-limit tests) |
| SEC-1-04 | Neither `titan-ci.yml` nor `titan-deploy.yml` declared explicit `permissions:` — both ran with the ambient default `GITHUB_TOKEN` scope rather than an explicit least-privilege grant, unlike `deploy.yml` (the marketing site's own pipeline), which already had one | **Low** — no evidence either workflow's default scope was ever exploited; still worth closing, since neither workflow needs anything beyond `contents: read` | **Fixed this phase** — explicit `permissions: contents: read` added to both, matching `deploy.yml`'s own existing convention | `.github/workflows/titan-ci.yml`, `.github/workflows/titan-deploy.yml` |
| SEC-1-05 | `requireOrganizationAccess` (`auth/authorize.ts`) is exported and unit-tested but has zero call sites in the live routing table | **Informational** — not a vulnerability; a real, tested helper built ahead of a need that hasn't arrived yet | **Known, accepted** — ready for the day a real organization-scoped, minimum-role-gated route needs it (`resolvePortalOrganizationId`'s portal routes deliberately don't distinguish member/admin/owner, so they don't call it) | `auth/authorize.ts`, `auth/authorize.test.ts` |
| SEC-1-06 | Rate limiting is per-Worker-isolate, not global (`security/rateLimiter.ts`) | **Low today; Medium once deployed** | **Blocked** — needs a deployed Cloudflare account plus a native Rate Limiting binding or Durable Object (carried forward, `SECURITY_GUIDE.md`) | `SECURITY_GUIDE.md` |
| SEC-1-07 | No bot protection (Turnstile) on the public assessment/lead form | **Low today** (no deployed public form to abuse); **would need real evaluation once deployed** | **Blocked** — needs a deployed Cloudflare account (carried forward, `ROADMAP.md`'s deferral record) | `SECURITY_GUIDE.md`, `ROADMAP.md` |
| SEC-1-08 | 14 cross-organization list/search/export/summary endpoints cannot be delegated to organization-scoped roles, only Platform Administrator (repository query layer has no per-organization filter) | **Low** — deny-by-default holds; no cross-tenant leak exists today, only a real capability gap | **Known, accepted** — needs query-layer organization filtering in each repository, which needs a real consumer to design against (carried forward, `SECURITY_GUIDE.md`) | `SECURITY_GUIDE.md` |
| SEC-1-09 | `wouldRemoveLastPlatformAdministrator`'s check-then-act isn't atomic — two concurrent requests could each remove a different one of exactly two remaining Platform Administrators | **Low** — no concurrent traffic exists against this local-only system today | **Known, accepted** (carried forward, `SECURITY_GUIDE.md`) | `SECURITY_GUIDE.md` |
| SEC-1-10 | GitHub Environment protection rules (required reviewers, wait timers) for `titan-deploy.yml`'s `staging`/`production` environments are not configured | **Medium once a real Cloudflare account exists** — until configured, any collaborator with `workflow_dispatch` permission can attempt a real deploy with no additional approval gate | **Blocked** — a repository *setting* (Settings → Environments), not something this codebase can commit (carried forward, `SECURITY_GUIDE.md`/`DEPLOYMENT_GUIDE.md`) | `SECURITY_GUIDE.md` |
| SEC-1-11 | Nothing in this repository scopes or audits the real `CLOUDFLARE_API_TOKEN`'s actual permission grant | **Medium once a real token exists** | **Blocked** — a human diligence step taken when the token is created in Cloudflare's own dashboard, not verifiable from inside this repository (carried forward, `SECURITY_GUIDE.md`) | `SECURITY_GUIDE.md` |
| SEC-1-12 | OPS-1's real, tested alert-evaluation logic is never delivered to a human (no PagerDuty/Slack/email integration) | **N/A** — no deployed environment produces real traffic to evaluate | **Blocked** — needs a deployed environment and a chosen paging provider, neither of which this project has ever had (carried forward, `SECURITY_GUIDE.md`/`MONITORING_GUIDE.md`) | `SECURITY_GUIDE.md`, `MONITORING_GUIDE.md` |

## Penetration test preparation (Workstream 10)

**No external or internal penetration test has been performed.** Everything below is a real attack scenario this system's own existing, committed, passing test suite already exercises — cited by exact file, not a claim that a human or automated pentester has attacked a running deployment (none exists to attack). Useful as a starting brief the day a real pentest is commissioned, and as the concrete, verifiable evidence behind every "mitigated" claim in this document and `SECURITY_GUIDE.md`.

| # | Attack scenario | Expected result | Verified by |
|---|---|---|---|
| 1 | Reach any Platform-Administrator-gated route with no session | 401 | `router.test.ts` — every gated `describe` block's own "returns 401 for an anonymous caller" case |
| 2 | Reach a Platform-Administrator-gated route as an authenticated non-admin | 403 | `router.test.ts` — every gated `describe` block's own "returns 403 for an authenticated caller who is not a Platform Administrator" case |
| 3 | An organization member supplies another organization's id as a query/body parameter on any `/api/portal/*` route | Ignored — own organization resolved server-side regardless | `router.test.ts`'s "ignores any organizationId the client tries to supply and always resolves the caller's own" |
| 4 | Navigate directly to another organization's assessment by id (not through any UI affordance) | Honest "not available to your organization" message, never the data | `apps/web/e2e/customer-portal.spec.ts` (real browser, real local D1, two real seeded organizations) |
| 5 | Forge a cross-origin write with a mismatched `Origin` header (CSRF) | 403, on all 12 CSRF-checked write routes | `router.test.ts` — a "returns 403 for a ... mismatched Origin" case per write route |
| 6 | Submit a fabricated risk score inconsistent with the submitted answers | Server recomputes and persists the real score; the tampered value never reaches D1 | `router.test.ts`'s "recomputes the risk result server-side... discarding a tampered client result" (Security Release Blocker Sprint) |
| 7 | Exceed the rate limit on any rate-limited route | 429 after the threshold, on all 7 rate-limited routes (4 pre-existing + 3 added this phase) | `router.test.ts` — a dedicated rate-limit test per route |
| 8 | A Platform Administrator attempts to demote/revoke the system's last remaining Platform Administrator | 409, rejected | `router.test.ts`'s `wouldRemoveLastPlatformAdministrator` cases |
| 9 | Inject SQL via any search/filter/text parameter | Structurally prevented — every query is parameterized (`db.prepare(...).bind(...)`), confirmed by direct code audit of all 9 `*.d1.ts` repositories; no dynamic test fuzzes this specifically, since the architecture makes the injection point not exist, not merely defended | Code audit (this phase's own research pass); no raw string concatenation into SQL found anywhere in first-party code |
| 10 | Inject a script via any text field (lead name, organization notes, assessment answers) | Structurally prevented — React's default escaping, zero `dangerouslySetInnerHTML`/`eval`/`new Function` anywhere in first-party code (confirmed by direct grep this phase) | Code audit; `ARCHITECTURE.md`'s own prior audit reached the same conclusion |
| 11 | Craft a `callbackUrl` pointing to an attacker-controlled origin on sign-in/sign-out | Falls back to the Worker's own origin — never redirects to an unrecognized origin | `auth/config.ts`'s `redirect` callback, exercised by real-browser E2E (`admin-dashboard.spec.ts`) |
| 12 | Attempt CSV/formula injection (a cell beginning with `=`/`+`/`-`/`@`) via Audit/Reports export | Not present in this data model — every exported field is either a known enum value or JSON-wrapped metadata (always begins with `{`) | `router.test.ts`'s CSV export tests (EAP-6) |

**Explicitly not exercisable from this repository, honestly**: anything requiring a real deployed environment (real network-level DDoS resistance, real WAF bypass attempts, real Turnstile evasion, real cross-isolate rate-limit exhaustion) — `SECURITY_ARCHITECTURE.md`'s "WAF & edge security" section names what these would test the day a deployment exists.
