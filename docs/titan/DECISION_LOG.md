## 2026-07-22 — OPS-1: Enterprise Operations & Site Reliability Engineering — real observability aggregation, alert evaluation, and readiness hardening on top of the verified EAP-7/PRD-1 foundation

**The OPS-1 master prompt asked for operations, observability, reliability, monitoring, alerting, incident response, operational automation, and SRE practices only** — explicitly excluding security hardening, compliance programs, a WAF, penetration testing, production release, global launch, payment processing, any business/customer/commercial feature, and redesigning Authentication/Authorization/RBAC/the Assessment or Reporting Engines/the Administration or Customer Platforms/production infrastructure/the database architecture/the Repository Pattern/shared components/business logic. This pass ran in one continuous session from the already-clean, already-pushed PRD-1 baseline (`84ee862`). Before any implementation, two research passes verified exactly what EAP-7/PRD-1 already built (the existing Operations Center: `/health`, `/health/ready`, `GET /api/operations/summary`, structured logging, in-memory metrics) so this phase extends it rather than duplicating or contradicting it — a repository-wide search confirmed zero alerting/incident-management/SLO/capacity-planning code existed anywhere before this phase.

**Decided: aggregation and alert evaluation are two new, pure, zero-I/O leaf modules — not logic added directly to `router.ts`.** `observability/aggregate.ts` (`computeLatencyPercentiles`, nearest-rank method; `computeErrorRate`) and `observability/alerts.ts` (`evaluateAlerts`) each take plain data in and return plain data out, the same discipline `config/validateEnv.ts` already established for `validateProductionConfig`. `alerts.ts` deliberately defines its own narrow `ServiceHealthInput` type rather than importing `router.ts`'s `ServiceStatus`, keeping the dependency direction one-way (router depends on observability, never the reverse).

**Decided: readiness now reflects configuration validity, not only database reachability — via one shared function, not two diverging checks.** `router.ts`'s new `computeReadiness` is used by both `readinessResponse` (`GET /health/ready`) and `operationsSummary` (`GET /api/operations/summary`), so a `staging`/`production` deployment with an invalid configuration (PRD-1's `validateProductionConfig`) fails readiness on its own terms — checked before the database, so the more specific `configuration.invalid` alert never appears alongside a redundant generic `readiness.not_ready` one for the identical root cause (verified by a dedicated test for exactly this suppression). Absent-safe by construction: every pre-OPS-1 test that never set `deps.configValidation` keeps its exact prior behavior.

**Decided: every alert carries the real evidence that triggered it — no severity without a number behind it.** `evaluateAlerts` returns `{id, severity, message, evidence}` per triggered condition (readiness, per-service reachability, configuration validity, 5xx error rate, p95 latency), each evidence object containing the actual value compared against the threshold. `DEFAULT_ALERT_THRESHOLDS` (5%/20% error rate, 500ms/2000ms p95 latency) are named, reasonable SRE starting defaults — explicitly not derived from real production traffic, because none has ever existed (`MONITORING_GUIDE.md`'s "Revisiting these thresholds").

**Decided: real alert delivery to a human is out of scope for this phase, not silently skipped.** No PagerDuty/Slack/email/SMS integration exists. Building one against a provider with no account to test it against, for traffic that doesn't exist, would be exactly the fabricated-but-untestable automation this program's engineering principles rule out — the identical reasoning PRD-1 used to defer R2/Queues bindings. What's built instead is the real detection/evaluation logic, reachable via the existing Platform-Administrator-gated endpoint and dashboard, ready to be wired to a real provider the day a deployment and a provider decision both exist.

**Decided: the Operations Center dashboard gains three additions, each a composition of data the endpoint already returns — no new signal invented at the UI layer.** An Alerts panel (`AlertsPanel.tsx`, an honest "No alerts firing" state when the array is empty, not a blank render indistinguishable from a failed load), a "Request health" section added to the existing Runtime Metrics panel, and an Operational Summary banner deriving one line from `highestSeverity(alerts)` plus the pre-existing System Overview's own version/environment.

**Decided: one new automation script, deliberately independent of the pre-existing `smoke-test.mjs`.** `check-operational-thresholds.mjs` measures one real request's own latency against `/health`/`/health/ready` (no auth needed) and classifies it against the same threshold values `MONITORING_GUIDE.md` documents — a narrower, different measurement than the in-app p95-over-real-traffic computation, not a fabricated substitute for it. Kept separate from `smoke-test.mjs` rather than extending it, the same "avoid touching a stable, already-verified file" discipline `titan-deploy.yml` already applied to `titan-ci.yml`.

**A real bug found and fixed while building this phase, not by review alone**: the first version of `evaluateAlerts` fired both a generic `readiness.not_ready` alert and the more specific `configuration.invalid` alert for the identical root cause (an invalid production config, which `computeReadiness` treats as its own higher-priority reason for non-readiness, checked before the database). Caught by inspecting the real log output of an existing PRD-1 worker-level test after wiring the new code in, not assumed correct from the implementation alone — fixed by suppressing the generic alert specifically when an invalid configuration already explains it, with a dedicated test for both the suppressed and the still-fires cases.

**Verified, not just implemented:** `npm run ci` passes clean across all four packages — 1068 tests (up from 1025, +43). `@titan/platform`: 622 (up from 587, +35 — 12 new `aggregate.test.ts` unit tests, 17 new `alerts.test.ts` unit tests, 6 new `router.test.ts` tests covering readiness+configuration interaction and the new `operationsSummary` fields). `@titan/web`: 359 (up from 351, +8 — 4 new `AlertsPanel.test.tsx` tests, 3 new `RuntimeMetricsPanel.test.tsx` tests, 1 new `OperationsWorkspacePage.test.tsx` test); `@titan/assessment-core`/`@titan/design-system` unchanged (25/62). The new `check-operational-thresholds.mjs` script was actually run against a real local `wrangler dev` Worker (both the passing path and the unreachable-URL failure path), not just written. A fresh production build shows the Worker bundle at 452.05 KiB / 93.13 KiB gzip and `@titan/web`'s main bundle at 834.46 kB / 186.42 kB gzip (`MONITORING_GUIDE.md`).

**A real, investigated finding about this session's own execution environment, not a code regression: the full Playwright E2E suite times out unreliably in this specific remote sandbox, on the pre-OPS-1 baseline too.** Running the full 39-test suite against the OPS-1 branch produced 28 timeouts (`toBeVisible` waiting on a heading, 10s limit) spread across nearly every spec file, including ones this phase never touched (Assessment Center, Audit Center, Commercial Platform, Lead Workspace, Organization Workspace, Reporting, User Management). Rather than assume a regression, the working tree was stashed back to the exact pre-OPS-1 commit (`84ee862`) and a smaller, controlled comparison was run: `admin-dashboard.spec.ts` (a file with zero OPS-1 changes) plus `operations-center.spec.ts`, 4 tests total. **2 of the 4 failed with the identical timeout pattern on the unmodified baseline itself.** This is real, direct evidence — not a full-suite baseline re-run, but a controlled sample large enough to show the same failure mode pre-exists — that this remote sandbox's own resource/timing characteristics, not this phase's code, are the cause. The stash was popped and the working tree restored immediately after. `MONITORING_GUIDE.md`/`OPERATIONAL_RUNBOOK.md` do not claim a fresh 39/39 E2E run for OPS-1 as a result — Vitest's 1068/1068 (environment-independent, no browser/webServer timing involved) is this phase's authoritative test evidence; full-suite Playwright E2E execution in this specific remote sandbox is a known, investigated limitation, not silently glossed over.

**Explicitly deferred, not forgotten:** real delivery of an alert to a human (see above). A durable, cross-isolate metrics/alert history (needs Workers Analytics Engine, needs a deployed Cloudflare account — the same limitation `SECURITY_GUIDE.md` already names for EAP-7's own runtime metrics). Tuning alert thresholds or SLO targets against real production traffic (none has ever existed). A real on-call rotation, paging tool, or public status page. Route-based code-splitting for `@titan/web`'s main bundle (PRD-1's own finding, re-confirmed, still out of an operations-only phase's scope). A retention/archival policy for `audit_events` (a new, real finding from this phase's own capacity-planning review — not urgent at today's real volume of 373 rows, but named for the day it matters). Enterprise Security & Compliance (SEC-1), Release Candidate, and General Availability work — per this program's own standing instruction to complete OPS-1 first.

## 2026-07-22 — PRD-1: Enterprise Production Infrastructure — real, verified deployment automation; still never deployed anywhere

**The PRD-1 master prompt asked for production infrastructure only** — environment architecture, Cloudflare production foundation, configuration management, secrets management, CI/CD, database operations, backup/DR, performance, production validation, release automation — explicitly excluding new application features, redesigning any verified system, Enterprise Operations & SRE (OPS-1), Enterprise Security & Compliance (SEC-1), and an actual production deployment/Release Candidate/General Availability. This pass ran in one continuous session with no mid-work container reset, from the already-clean, already-pushed COM-1 baseline (`5c0447a`/`e881006`). This entry records the design decisions made and the real bugs found while building it.

**Decided: named environments live entirely in `wrangler.toml`; nothing about them is invented or fabricated as "real."** `[env.staging]`/`[env.production]` each restate their own `[[d1_databases]]`/`[vars]` — confirmed, not assumed, that named environments don't inherit the default block's bindings, by running `wrangler deploy --dry-run --env staging` before and after adding them and diffing the resolved binding list. Every value a real deploy needs (`database_id`, `ALLOWED_ORIGIN`) is a literal `REPLACE_WITH_REAL_*` placeholder — the exact same honest-placeholder convention the pre-existing default environment's own `database_id` already used since Stage 4, extended rather than invented fresh.

**Decided: configuration validation is a pure function run on every request, not a startup hook — because Workers genuinely has no startup hook.** `env` bindings only exist inside `fetch(request, env, ctx)`, not at module evaluation time (confirmed against the Workers runtime model, not assumed) — so `config/validateEnv.ts`'s `validateProductionConfig` runs cheaply on every request instead, logged once per isolate via a module-scoped flag (`worker.ts`) rather than per-request. Surfaced through `GET /api/operations/summary`'s new `configuration` field — extending EAP-7's existing operational-status endpoint rather than adding a new one, the same "reuse before build" discipline every prior phase has followed.

**Decided: `titan-deploy.yml` is a second, fully independent workflow, not an extension of `titan-ci.yml`.** Duplicates the same five quality-gate steps rather than sharing them via a reusable `workflow_call` — a deliberate choice to avoid touching a stable, working file (`titan-ci.yml`) for a functionally different concern, applying this program's own "never redesign a stable system" instruction to CI configuration itself, not just application code. Manual-only (`workflow_dispatch`), never triggered by push or tag, so an accidental push can never attempt a real deployment.

**Four real bugs found and fixed this phase, each caught by actually running the thing, not by review alone:**
1. `check-wrangler-config.mjs`'s first version matched `[env.staging.vars]` (a sub-table) as if it were a new top-level `[env.production]` header — both start with the literal text `[env.` — silently truncating its own placeholder scan before it ever reached `ALLOWED_ORIGIN`. Found by running it for real and getting only one placeholder back when two were expected.
2. `wrangler secret list` needed `--format json`, not `--json` — assumed from a plausible-sounding flag name, corrected against the real installed CLI's own `--help` output after the first version errored with `Unknown argument: json`.
3. `wrangler d1 execute <name> --local` requires `<name>` to already be a registered `[[d1_databases]]` entry in `wrangler.toml` — it does not create an arbitrary local SQLite file on the fly the way a first attempt at a disaster-recovery drill assumed. Recovery procedure adjusted to restore into the already-registered `titan-platform-db` rather than a throwaway name.
4. Restoring a full `wrangler d1 export` output after first re-running `wrangler d1 migrations apply` causes a real `UNIQUE constraint failed: d1_migrations.id` error — a full export already contains Wrangler's own internal migration-bookkeeping rows, so applying migrations first double-inserts them. The correct, simpler procedure (restore directly, no prior migrations step) was discovered by hitting this failure, not by reading Wrangler's own documentation for it.

**A real, verified disaster-recovery drill — the strongest evidence in this phase.** Captured real baseline row counts across five tables (this session's own accumulated data: 65 leads, 43 assessments, 62 organizations, 373 audit events, 22 subscriptions), exported via `wrangler d1 export --local`, wiped local D1 entirely (`rm -rf .wrangler/state/v3/d1`), and restored using the corrected procedure above. Row counts matched the baseline exactly across all five tables. A live `wrangler dev` against the restored database then passed a full `smoke-test.mjs` run, confirming the recovery was genuinely application-usable, not merely SQL-correct.

**Verified, not just implemented:** `npm run ci` passes clean across all four packages — 1025/1025 tests (up from 1011: `@titan/platform` 587, up from 573 — 9 new `validateEnv.test.ts` unit tests, 3 new `router.test.ts` tests for the `configuration` field, 2 new `worker.test.ts` end-to-end tests proving real `env.ENVIRONMENT` wiring through the actual Worker; `@titan/web`/`@titan/assessment-core`/`@titan/design-system` unchanged — PRD-1 touched no frontend code). The full 39-test Playwright E2E suite was re-run as a regression check after all backend changes — 39/39, unaffected. One real, expected lint gap was found and fixed: the new `packages/platform/scripts/*.mjs` files needed a Node-globals ESLint block (`packages/config/eslint.base.mjs`), since they run outside any browser/Workers context ESLint's existing config accounted for. A fresh production build (not the stale Stage 4 figure) shows the Worker bundle at 445.06 KiB / 91.47 KiB gzip and `@titan/web`'s main bundle at 826.13 kB / 185.49 kB gzip — Vite's own output now warns this exceeds its 500 kB chunk-size guidance, a real, new finding recorded rather than left undiscovered (`ROADMAP.md`'s new follow-up item).

**Explicitly deferred, not forgotten:** any actual Cloudflare deployment (no account exists — every credential-requiring command in this phase's own tooling was run and confirmed to fail with exactly the expected authentication error, never faked as succeeding). R2/Queues/Durable Objects/Turnstile bindings (nothing in this codebase generates a file to store, dispatches an async job, or gates a public form yet — real, evidence-based "not yet justified"). Custom domain/DNS/TLS. A scheduled/automated remote backup job. Route-based code-splitting for `@titan/web`'s main bundle (a real finding, out of an infrastructure-only phase's own scope). No Enterprise Operations & SRE (OPS-1), Enterprise Security & Compliance (SEC-1), Release Candidate, or General Availability work was started, per this program's own standing instruction.

## 2026-07-22 — COM-1: Enterprise Commercial Platform — provider-agnostic subscriptions/licenses/entitlements, a code-defined plan catalog, one deliberate customer-vs-admin asymmetry, and zero changes to any existing route

**The COM-1 master prompt asked for the commercial layer on top of the already-verified EAP-1–EAP-8 and CPP-1 platform**: Subscription Management, Plan Catalog, Tenant Entitlements, License Management, Feature Access, Subscription Lifecycle, Customer Workspace, Commercial Administration, a provider-agnostic Billing Integration Layer, Upgrade/Downgrade Workflows, Trial Management, and Commercial Audit Events — explicitly excluding Payment Gateway Processing, PCI card storage, Accounting, Tax Engines, Marketplace, Affiliate Programs, CRM, ERP, and any production/deployment work. Like CPP-1's own entry, this pass ran in one continuous session with no mid-work container reset, so this entry records the design decisions made while building COM-1 from the already-verified, already-pushed CPP-1 baseline (`ab416f4`), and the verification performed once it was built.

**Decided: Subscription and License are two separate repositories, not one, but License is deliberately not "one row per seat/user."** `SubscriptionRepository` is the organization-level commercial record (plan, status, trial/period dates) — one per organization, the same real 1:1 tenant shape every other entity in this system already has with `organizations`. `LicenseRepository` is the seat *grant* itself (`seatLimit`, `status`, `activatedAt`/`expiresAt`), created alongside its subscription, also 1:1. Seat *usage* is deliberately not a column on either: it's the live count of an organization's own real `UserProfileRecord`s, read via a new `UserProfileRepository.findByOrganizationId` method (the mirror image of the pre-existing `findByUserId`) — a separately maintained seat-usage counter could drift from real membership; a live count never can. This is the identical "compose from the existing repository, don't duplicate its data" discipline CPP-1's own `PortalComplianceSummary` established for assessment counts, applied to a new domain.

**Decided: the Plan catalog is code-defined (`commercial/planCatalog.ts`), not a database table.** Three real plans — Starter, Professional, Enterprise — each with a name, a display-only price string, a trial length, and a small, real entitlement set. This mirrors `@titan/assessment-core`'s own DPDP question bank: real, versioned, typed content that ships with the application rather than being authored at runtime through an admin UI. Nothing in COM-1's brief asked for dynamic plan CRUD (creating a fourth tier, changing an existing tier's price with the proration implications a live price change would carry for existing subscribers) — building that would be real, but different, speculative scope. `priceDisplay` is exactly what its name says: display text, never a chargeable amount any code path processes — the one fact worth stating plainly given how easily "price" could be misread as implying a real billing integration exists.

**Decided: Feature Entitlements are computed and displayed, deliberately not enforced as a gate on any existing route this phase.** `resolveEntitlements(plan, subscription)` (`commercial/entitlements.ts`) is a pure function: the full plan's entitlements while `trialing`/`active` (a trial is genuinely full-featured in this model, not a crippled preview), collapsed to nothing once `canceled`/`expired`. It is real, correct, and shown on both the Customer Portal's own Subscription page and the admin Commercial Detail page. But wiring it as an enforcement check — e.g., blocking CPP-1's own `GET /api/portal/reports/export` unless `entitlements.complianceReportExport` is true — would mean modifying the Enterprise Customer Portal or the Assessment/Reporting Engine, all four explicitly named do-not-redesign systems in this program's own instructions. Enforcement is real, named future work for the day a specific gated feature is identified and the program's scope explicitly includes touching one of those systems — not built speculatively ahead of that decision, and not silently implied to already exist by the presence of the entitlement data itself.

**Decided: `POST`/`PATCH /api/portal/commercial/subscription` refuse a sales-assisted plan; the admin override `PATCH /api/commercial/subscriptions/:id` does not — a deliberate, tested asymmetry, not an inconsistency.** `isSelfServicePlan(plan)` is simply `plan.trialDays > 0`; Enterprise's `trialDays: 0` signals "contact sales" both semantically and functionally. A customer's own self-service path checks this and returns 400 `sales_assisted_plan`; the Customer Portal's own plan-selection UI doesn't even render a start button for Enterprise, so a customer is never shown an action that would fail. The admin path carries no such restriction — a Platform Administrator assigning Enterprise to a real customer's organization on their behalf, after an off-system sales conversation, is exactly the real workflow "contact sales" implies, not a gap. Verified directly: a real router test confirms the customer PATCH 400s attempting `planId: "enterprise"`, and a separate test confirms the admin PATCH succeeds with the identical payload against the identical subscription.

**Decided: one shared internal state machine, `applySubscriptionPatch`, backs both the customer-facing and admin-facing PATCH routes, after each validates the incoming patch against its own, differently-scoped rules.** It diffs the patch against the pre-update record to derive real `subscription.upgraded`/`.downgraded` (by comparing `Plan.tier`), `.canceled`, `.renewed`, and `license.activated`/`.expired` audit events — the same diff-before-recording discipline `updateOrganization`/`updateLead` already established, extended here to also keep the subscription's own license in lockstep (a cancellation expires its license; a renewal reactivates it) rather than letting the two drift apart independently. Renewal always computes a new `currentPeriodEnd` server-side (`Date.now() + 30 days`), never accepting one from the caller — the same "never trust a client value for a business date" discipline `POST /api/leads`'s server-side score recomputation already established for a different field.

**Decided: License Inventory is not its own top-level admin page.** Because License is 1:1 with Subscription in this data model, a separate "Licenses" list would show the identical rows the Subscription Workspace's table already does, just re-sorted — genuinely redundant UI over the same underlying data. Seat limit/usage/status instead surface as columns on the Commercial Workspace table and as a dedicated section on the Commercial Detail page. `GET /api/commercial/licenses/search` still exists as its own real endpoint (COM-1's brief named License Management as its own capability), just not surfaced as a duplicate page.

**Decided: `GET /api/commercial/plans` is open to any authenticated caller — no Platform Administrator or organization-membership gate.** The plan catalog carries no tenant-scoped or otherwise sensitive information (it's the commercial equivalent of public pricing copy), so gating it would just be friction with no real protection behind it — the same deliberate exception `GET /api/me` already is for identity data. Serving one endpoint to both the admin's plan-assignment dropdown and the customer's plan-selection cards avoids a redundant, duplicate route for identical data.

**Verified, not just implemented:** `git diff` against the pre-COM-1 baseline shows every touched pre-existing file (`router.ts`, `router.test.ts`, `worker.ts`, `index.ts`, `repositories/types.ts`, `userProfileRepository.memory.ts`/`.d1.ts`/`.contract.ts`, `App.tsx`, `navItems.ts`, `portalNavItems.ts`/`.test.ts`) as pure addition — 1751 insertions, zero deletions, confirming no existing route, handler, or authorization check changed shape. Full workspace `typecheck`/`lint`/`format`/`build`/`test` passes clean at 1011/1011 tests (`@titan/platform` 573, up from 461: 35 new router tests, 60 new Subscription/License contract tests, 6 new `findByOrganizationId` contract tests, 11 new plan-catalog/entitlement pure-logic tests; `@titan/web` 351, up from 333: 18 new component tests; `@titan/assessment-core` 25 and `@titan/design-system` 62 unchanged). An eleventh Playwright E2E spec (`commercial-platform.spec.ts`, 4 tests) ran clean across 2 consecutive full-suite runs (all 11 specs, 39 tests total), zero flakes — proving a real customer's self-service subscribe/cancel/renew round trip against a real browser and real local D1, and a real Platform Administrator reviewing cross-organization subscriptions and assigning Enterprise directly. Full detail: `ARCHITECTURE.md`'s "Enterprise Commercial Platform architecture (COM-1)" section, `PLATFORM_FOUNDATION.md`'s COM-1 section, `SECURITY_GUIDE.md`'s COM-1 paragraph, `ROADMAP.md`'s COM-1 section.

**Explicitly deferred, not forgotten:** a real payment gateway integration; entitlement enforcement on any existing route; a multi-subscription-per-organization concept; dynamic plan authoring/CRUD; a support-request-style resolution workflow for Commercial Administration beyond the real admin override already built. No Accounting, Tax Engines, Marketplace, Affiliate Programs, CRM, ERP, Cloudflare Production, Release Candidate, or General Availability work was started, per this program's own standing instruction.

## 2026-07-21 — CPP-1: Enterprise Customer Portal — a new, separate customer-facing shell reusing the EAP foundation's services and security model, one genuinely new entity (Support Requests), and zero changes to any existing admin route

**The CPP-1 master prompt asked for a customer-facing portal layered on the already-verified EAP-1–EAP-8 platform**, with explicit, repeated instructions not to redesign the administration platform, not to duplicate admin functionality, and not to begin any Commercial Platform/Cloudflare Production/Release Candidate/General Availability work until CPP-1 was complete. Unlike several prior EAP phases, this pass ran in one continuous session with no container reset mid-work, so there was no "verify what a resume prompt claims against real git state" step of the kind EAP-3/EAP-6/EAP-8's own entries describe — this entry instead records the design decisions made while building CPP-1 from a clean, already-verified EAP-8 baseline (`831bb54`), and the verification performed once it was built.

**Decided: the Customer Portal is a real, separate route tree (`/portal`), not a nested `/admin` route, and not a new authorization role.** A caller reaching `/portal` is a real Organization Member/Admin/Owner — the same `UserProfileRecord` roles `auth/rbac.ts` has had since the Security Release Blocker Sprint — using their own existing Auth.js session. `App.tsx` mounts `<Route path="/portal">` as a sibling to `<Route path="/admin">`, both wrapped in the same `SessionProvider`/`RequireAuth`. This mirrors the user's own stated rationale for sequencing CPP-1 after EAP-8: EAP-1–EAP-8 built the internal administrative platform; CPP-1 exposes a customer-facing subset of the same underlying data, reusing the identity/session/RBAC layer rather than inventing a parallel one.

**Decided: `resolvePortalOrganizationId(caller, requestId): string | Response` is the one mechanism every portal route depends on, and it is the entire security model.** It finds the first of the caller's own `UserProfileRecord[]` with a non-null `organizationId` and returns that id — or a 403 if none exists. Every portal handler (`getPortalOrganization`, `listPortalAssessments`, `buildPortalComplianceSummary`, `getPortalActivity`, `createPortalSupportRequest`) receives that resolved id as a plain function parameter, never reading `organizationId` from a query string, path segment, or request body the way the Platform-Administrator-only admin routes correctly do for a caller who is *allowed* to see any organization. This closes, for this module specifically, the exact class of gap `SECURITY_GUIDE.md`'s "Known, accepted gaps" has named as missing (organization-scoped filtering) since EAP-1 — verified with real seeded data (two real organizations, one assessment each) in both `router.test.ts` and a real-browser Playwright spec, not just asserted.

**Decided: a caller with more than one real organization membership resolves to the first one found — there is no organization switcher in this phase.** A real, named scope limit, not an oversight: building a switcher UI plus deciding how it should interact with Dashboard/Assessments/Reports/Support state was judged out of proportion to how rare multi-organization membership actually is in this data model today (nothing in EAP-1–EAP-8 ever produced one deliberately). Recorded here so a future phase doesn't have to rediscover this by reading code.

**Decided: Assessment Detail reuses the *existing* `GET /api/assessments/:id` unchanged — no new backend route.** That endpoint has delegated to `requireAssessmentAccess` (Security Release Blocker Sprint/EAP-2) since before CPP-1 existed: a Platform Administrator or a real member of the assessment's own organization may read it, anyone else gets a 403. That is exactly the access a portal customer needs, so `usePortalAssessmentDetail` calls the admin feature folder's own `fetchAssessment` directly rather than duplicating the fetch logic or adding a redundant `/api/portal/assessments/:id` route. The frontend maps that 403 to an honest "This assessment is not available to your organization." message — proven against a real browser in `customer-portal.spec.ts` by seeding two organizations and navigating directly to the *other* one's assessment by id, not through any UI affordance.

**Decided: `PortalComplianceSummary` is a new, deliberately smaller type, not `ExecutiveSummary` (EAP-8) reused with a filter.** Most of `ExecutiveSummary`'s sections have no honest single-organization form: leads are never associated with an organization in this data model at all (`SECURITY_GUIDE.md`'s own note on `LeadRecord.organizationId`); "identity" (Platform Administrator counts) is a platform-wide concept with no per-organization analogue; and reusing "audit" as a platform-wide rollup here would leak cross-tenant activity counts through what looks like an organization-scoped endpoint. What does have a real, honest per-organization form is this organization's own assessments — computed via the existing `AssessmentRepository.search({organizationId, ...})`, the same repository call the admin Assessment Workspace already makes, just with the tenant filter forced rather than optional.

**Decided: Support Requests are the one genuinely new entity this phase needed — a new repository, a new migration (`0010_support_requests.sql`), nothing reused or repurposed.** No existing table had a real analogue (leads are pre-signup/anonymous; audit events are a log, not a request-and-response concept). Deliberately minimal: `SUPPORT_REQUEST_STATUSES = ["open"]` — a request is created and stays `"open"`; there is no admin-side endpoint anywhere in this codebase that reads or resolves one. Building a resolution workflow (an admin queue, status transitions, assignment) was judged out of CPP-1's scope — the brief asked for a customer being able to submit a request and see their own history, not a ticketing platform, and a real design for the admin side deserves its own phase rather than being improvised here.

**Decided: `listPortalSupportRequests` scopes to `caller.userId`, not the caller's organization.** A member sees only the support requests *they personally* submitted, not every request their organization's other members have filed — narrower than the organization boundary every other portal route uses. There is no admin-facing or team-visibility concept for support requests yet (the point directly above), so exposing a request to any other member of the same organization would be a real, avoidable information disclosure with no corresponding feature that needs it. Verified directly: two different members of the same seeded organization each see only their own submitted request.

**Decided: `POST /api/portal/support` gets the same `isTrustedOrigin` CSRF check every other authenticated write in this codebase already has, and deliberately no rate limit.** Rate limiting in this codebase has only ever applied to anonymous, unauthenticated POST routes (`POST /api/leads`, `POST /api/assessments`) — an authenticated write already requires a real session, a materially weaker abuse vector than an anonymous one, and every other authenticated write added since EAP-4 (`POST /api/organizations`, the User Directory's grant/change/revoke routes) already follows this same convention. **Decided: the Worker's CORS `Access-Control-Allow-Origin` defaults to `http://localhost:5173` (Vite's default dev port), overridable via the `ALLOWED_ORIGIN` Worker var.** This Worker has never been deployed, so there is no real production frontend origin to allow yet. Whoever deploys this for real must set `ALLOWED_ORIGIN` to the real frontend origin; the current default is not a production-safe value and `wrangler.toml`'s `[vars]` entry says so directly.

**Decided: Auth.js's Email provider runs in a dev-mode logging configuration, not a real email provider, and is built as a plain `EmailConfig` object rather than through `@auth/core/providers/email.js`'s default export.** No email provider has been chosen (`ARCHITECTURE.md`'s "still open" list, unchanged since Phase 2) — Workstream 5 still needed a working Email provider to exercise Auth.js's database-session flow end to end. `providers/email.js` is a deprecated wrapper around `providers/nodemailer.js`, which imports the `nodemailer` package at module load time even when a custom `sendVerificationRequest` never calls its SMTP transport — pulling in a real dependency this project doesn't need yet just to satisfy an unused code path. ## 2026-07-20 — Hosting, database, and auth: Cloudflare-native, Auth.js, Repository Pattern

**Decided:**

- **Hosting/runtime target**: Cloudflare, full platform — Pages (app shell/marketing), Workers (backend/API), D1 (database), R2 (object storage — generated PDF reports), Queues (async work — report/email delivery), Turnstile (bot mitigation on public endpoints), plus Cloudflare's CDN/WAF/Analytics.
- **Database engine**: Cloudflare D1 (SQLite-based), accessed exclusively through a Repository Pattern — business logic never imports a D1 client directly.
- **Auth approach**: self-hosted Auth.js, behind an authentication abstraction layer that application code depends on instead of Auth.js's APIs directly.

**Source:** the Phase 2 master prompt ("CYBERDUDEBIVASH® Enterprise — Project Titan — Phase 2 — Enterprise DPDP Platform Integration & Production Evolution"), which specifies these three under "Architectural Decisions" as directives, not options to weigh. This resolves `ARCHITECTURE.md`'s open decisions #2 (hosting target) and #4 (auth provider) from the Phase 1 planning pass — both had been blocking Workstreams 4–7 since Phase 1 foundation work completed.

**What this closes out:** `PLATFORM_FOUNDATION.md`'s "blocked on architecture decisions" note for authentication, authorization, database, and API foundations no longer applies to *which* hosting/auth/database engine — those three are now known. Building them out is still real work (Phase E/F of the master prompt), not done by virtue of being decided.

**What this does not decide:** payments provider, email provider, and the DPDP legal-review timing remain open — the Phase 2 master prompt doesn't address any of the three. See `ARCHITECTURE.md`'s "Still open" list.

**Rationale for the Repository Pattern specifically (not just "use D1"):** the master prompt states it explicitly ("Business logic must never depend directly on D1"), and it holds up independent of that instruction — D1 is SQLite-based with a narrower feature set than the Postgres `ARCHITECTURE.md`'s original Phase 1 pass recommended. A repository boundary means that if D1's limits become a real problem later, the fix is a new repository implementation, not a rewrite of every place that touches data.
