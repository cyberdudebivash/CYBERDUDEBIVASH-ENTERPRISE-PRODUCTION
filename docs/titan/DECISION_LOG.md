# Decision Log — Project Titan

Chronological record of architecture and product decisions once they're actually made — not proposals (those live in `ARCHITECTURE.md` until someone decides). Newest first.

## 2026-07-21 — EAP-5: Enterprise Identity & User Management — a real continuation this time, a read-only identity repository over Auth.js's own table, this system's first hard delete, and a real environmental finding from real-browser verification

**A resume prompt for this task described EAP-5 as a continuation of verified, already-committed EAP-1 through EAP-4 work, with the explicit instruction to verify that from repository evidence before writing anything** — the same standing instruction, and the same discipline, every prior incident in this log (Security Release Blocker Sprint, EAP-3) already established as mandatory. Unlike those two, this time the verification came back clean on the *prior* phases: `git log`, `git fetch --prune` (which discovered the locally-cached `origin/claude/eap-5-identity-management-1dxc2j` ref was stale and pruned it), and direct inspection all confirmed EAP-1 through EAP-4 genuinely landed on `main` (merge commits for PRs #40–#44, plus one automated CI commit mirroring `dist/` to the repo root for Cloudflare — unrelated to Titan, from this monorepo's other program). **But EAP-5 itself had never been started anywhere** — no branch, no dangling commit, no stash, nothing matching "EAP-5"/"identity"/"user management" in any commit message on any branch. The designated branch existed locally only as a fresh checkout of `main` with zero commits of its own. Proceeded as new implementation from that verified baseline, not a resume — exactly the same conclusion-then-action this log's own precedent already established, just with a clean rather than a lost prior phase this time.

**Decided: a new `UserRepository`, read-only, backed by Auth.js's own pre-existing `users` table (migrations/0001) — not a new table, and deliberately no `save`.** The real identity record (name/email) already exists, written by `@auth/d1-adapter` on every real sign-in; `user_profiles` (RC1/Workstream 6) was already a *separate* table specifically so "application-specific fields live in a table application code owns instead of extending the adapter's schema" (that migration's own comment) — this repository respects the same boundary from the other side, reading identity without ever writing it. Concretely: no "Create User" admin form exists anywhere in this module, and none was built — inventing one would fabricate an identity with no real authenticated principal behind it, directly against this program's own "never fabricate implementation" principle. A person appears in the Enterprise User Directory once, and only once, they've actually signed in for the first time; the Workspace page says this in plain text rather than silently omitting a control that couldn't have worked.

**Decided: `UserRepository.search()` has no unfiltered `list()` counterpart, unlike every other repository in this system.** Every existing repository's `list()` exists because EAP-1's Dashboard needed an unfiltered read before `search()` (paginated/filtered) was added in a later phase. No Dashboard change is in this phase's scope (the brief's own Phase 3 list names the User Directory/Identity Profiles/Role Assignment, not Dashboard metrics), so building an unfiltered method with no caller would be exactly the kind of speculative surface this program's principles rule out. `search({})` with a generous `pageSize` serves any future need for "everyone," the same way it already does for every other search-only consumer in this codebase.

**Decided: `UserProfileRepository` gains `findById`/`list`/`update`/`remove` — Role Assignment's and Administrative User Lifecycle's whole real write surface, since "grant a role" was already `save` (existing since RC1).** `list()` exists for exactly one reason: the last-Platform-Administrator guard below needs to count every platform-wide grant system-wide, not one user's or one organization's. `update()` mirrors every other repository's null-not-throw `update` contract exactly (role reassignment only — `organizationId` isn't patchable, since repointing a grant to a different organization is a different grant, not an edit of this one).

**Decided: `UserProfileRepository.remove()` is this system's first real deletion — every other repository (Lead/Assessment/Organization/Audit) either has no delete at all or is explicitly archive-and-restore, never delete.** Investigated seriously before deciding, not applied by default: a `UserProfileRecord` is a thin `(user, organization, role)` grant with no dependent data of its own — unlike an organization (real linked leads/assessments an archive-not-delete pattern is actually protecting) or a lead (a real captured business record), there is nothing here that pattern would be preserving. The audit trail (`user.role_revoked`, written with the revoked role's own value *before* the row is removed) is what preserves the historical fact that a grant once existed; `user_profiles` itself is left to reflect only current, live grants. A real access-control module that couldn't revoke access at all would be a worse, more dishonest gap than introducing this system's first delete for the one entity shaped to actually need it.

**Decided: `wouldRemoveLastPlatformAdministrator` (router.ts) — a new guard, gating both the role-change and revoke paths — rejects (409) any action that would leave zero Platform Administrators system-wide.** This system has no break-glass account and no self-service bootstrap (Platform Administrator is "provisioned via direct SQL only" per every prior phase's own documentation) — without this guard, a real Platform Administrator could permanently lock every caller, including themselves, out of every Platform-Administrator-gated route in one PATCH or DELETE, with no recovery path short of another direct D1 write. Checked only when the profile being touched is itself a platform-wide grant (`organizationId: null, role: "owner"`) and the action would end that status; every other role change is unaffected and un-slowed. Verified with real scenarios, not just claimed: `router.test.ts` demotes/revokes the *only* remaining Platform Administrator (409, unchanged after) and then the *same* action once a second one exists (200, succeeds) — both directions, not just the happy path. One honest, accepted limitation: the count-then-act isn't wrapped in a single atomic transaction, so two concurrent requests each removing a different one of exactly two remaining grants could theoretically both pass the check before either write lands. Left open for the identical reason `createOrganization`'s slug-conflict check and this same phase's own `grantUserProfile` duplicate-membership check are already accepted unguarded elsewhere in this file — this is a local-only system that has never seen concurrent traffic, and locking one check while leaving every structurally-identical one in the same file unlocked would be inconsistent, speculative hardening, not a real fix. Recorded in `SECURITY_GUIDE.md`, not hidden.

**Decided: `POST /api/users/:id/profiles` is a real, working way to grant Platform Administrator itself, not just an organization role — closing (for every grant after the first) the "no self-service way to become a Platform Administrator" gap named in every phase's `SECURITY_GUIDE.md` since the Security Release Blocker Sprint.** Not a privilege-escalation hole: reaching this route at all requires `requirePlatformAdministrator`, so only a caller who already holds platform-wide authority can grant that same authority to someone else — the expected job of an admin's own role-management console, not a way past it. The *very first* Platform Administrator in a fresh deployment still needs a direct D1 insert, an unavoidable bootstrap constraint (you cannot grant a role through a route that requires already holding it) — `OPERATIONAL_RUNBOOK.md`'s provisioning section is updated to say precisely this, not to overclaim the gap is fully closed.

**Decided: `AssessmentSearchOptions`/`AssessmentRepository.search()` gain a `createdBy` exact-match filter — the same shape and reasoning as EAP-3's `assessmentId` and EAP-4's `organizationId` additions to sibling search options.** Backs User Relationships' "assessments created by this user" panel. Combined with the already-existing `LeadSearchOptions.assignedTo` (EAP-2, previously only ever used for the "assign to me"/"unassigned" sentinel values), User Details' Relationships panel is built entirely from filters that already existed on their own repositories — no new backend surface beyond this one addition.

**Decided: the User Workspace is deliberately smaller than every sibling Workspace — no `FilterPanel`, no saved filters, no column toggle, no create-form.** Each omission has its own real reason, not a shortcut: `UserSearchOptions` has no categorical field the way `OrganizationSearchOptions.status` does (role/membership live on a different repository entirely — see the repository-isolation decision below), so there is nothing for a filter panel to filter by or a saved-filter combination to save; the table has exactly two meaningful identity columns (name, email — plus a verified/unverified indicator), all worth showing always, so there is nothing to hide behind a toggle; and there is no create-form because there is no create capability (`UserRepository`'s own doc comment) — offering one would be a control that couldn't work. Building any of these speculatively, to merely match the other three Workspaces' shape, would be exactly the kind of unrequested complexity this program's principles rule out.

**Decided: `UserRepository.search()` has no `organizationId`/`role` filter, and consequently Organization Details gets no reciprocal "Members" panel this phase.** The identical repository-isolation reasoning `OrganizationSearchOptions`'s missing `riskLevel` filter already established (`DECISION_LOG.md`'s EAP-4 entry): a user's identity (`users`) and their grants (`user_profiles`) are two different repositories with no query-layer dependency on each other anywhere in this codebase, and "list users in organization X" would require exactly the cross-repository join that architecture doesn't support. User Details (this phase) shows a user's *own* memberships via `findByUserId`, already-existing since RC1 — the reciprocal organization-side view is real, valuable, explicitly recommended near-term follow-up (`ROADMAP.md`), not built this pass, to keep this module's own scope to the User Directory/Role Assignment it was asked for rather than also reopening EAP-4's already-shipped `OrganizationDetailPage.tsx`.

**Found and fixed — a real environmental gap, found exactly the way this program's real-browser-verification requirement exists to find this class of thing: this container's `packages/platform` had no `.dev.vars` file, so `AUTH_SECRET` was unset, so `worker.ts` never built an `AuthConfig` at all, so *every* caller — including one with a completely valid session cookie — silently resolved as anonymous.** First symptom looked exactly like a browser/cookie/proxy problem (this environment's outbound traffic runs through a configured proxy, and Playwright's own webServer startup logs printed a proxy-detected warning): every authenticated Playwright scenario failed to reach past a real Auth.js sign-in redirect, for this new module *and*, reproduced deliberately before assuming this was EAP-5-specific, for the already-shipped, previously-verified `organization-workspace.spec.ts` (EAP-4) too — proof this wasn't a regression in new code. Root-caused with a direct `curl -H "Cookie: ..." -H "Origin: ..."` request bypassing the browser entirely (ruling out SameSite/proxy/cookie-jar behavior in one step): a real seeded session, presented with the exact header shape the SPA's own `fetch` sends, still came back 401. That pointed straight at server-side session resolution rather than anything browser-side, and `OPERATIONAL_RUNBOOK.md`'s own documented troubleshooting entry ("`/api/auth/*` returns 404` → `AUTH_SECRET` not set") confirmed it on inspection: a fresh container, cloned new for this session, had simply never had its local secret provisioned — `.dev.vars.example` existed, `.dev.vars` didn't. Fixed by following the runbook's own documented setup step (`cp .dev.vars.example .dev.vars` + a real generated secret) — a container-provisioning gap, not a code defect, and not particular to this phase's own routes; every E2E spec in this suite needed the identical fix to run authenticated at all in this environment.

**Verified, not just implemented:** full workspace `typecheck`/`lint`/`format`/`build`/`test` — **713 tests, up from EAP-4's 609** (`@titan/web` 258, `@titan/platform` 372, `@titan/design-system` 58 and `@titan/assessment-core` 25 unchanged) — plus real operational verification against `wrangler dev` + real local D1 (real search/detail/grant/change/revoke actions through the actual UI, a real Chromium session confirming the Workspace, User Details' Identity/Role Assignment/Relationships/Audit history all rendering real data with zero console errors, and a direct D1 query confirming `user.*` events persist with the real caller as actor) and a new Playwright E2E spec (`apps/web/e2e/user-management.spec.ts`, 4 tests — including this codebase's first-ever real `DELETE` request, the exact class of change that silently broke `PATCH` in EAP-2 until real-browser verification caught it, verified proactively this time rather than found broken after) run alongside the existing five specs (20 E2E tests total, up from 16), stable across 3 of 4 consecutive full-suite runs — the one exception a `workerd` "broken pipe" flake from concurrent `wrangler d1 execute` CLI contention during test seeding (the same shared risk every sibling spec's identical seeding helper already carries), immediately reproduced-clean on retry, not a deterministic failure. Full detail: `PLATFORM_FOUNDATION.md`'s EAP-5 section.

## 2026-07-21 — EAP-4: Enterprise Organization Management Platform — a genuine clean continuation, a real administrative write surface, and a repository-isolation boundary that shaped the search API

**Unlike the two prior phases' resume prompts, this one followed real, verified, already-committed work.** Before writing any code, `git log`/`git fetch` confirmed EAP-3's commit (`d71d0d1`) was genuinely present on both local `HEAD` and `origin/claude/session-oi9iu3`, working tree clean — a real baseline to build on, not another lost-work incident (`ARCHITECTURE.md`'s audit, this log's own EAP-3 and Security Release Blocker Sprint entries, both precedent for exactly this failure mode occurring). Verifying that conclusion came first anyway, per this program's own standing instruction ("repository evidence overrides memory") — the difference from the prior two incidents is the answer, not whether the question was asked.

**The EAP-4 brief explicitly restricted scope to Organizations only** — no User Management, no Billing, no Customer Portal, "Stay focused on Organizations." Built exactly to that scope: an Organization Workspace, Organization Details (Metadata/Health/Relationships/Administration/Audit), and, unlike Assessments, a real create/update/archive/restore write surface, since an organization is genuinely something a Platform Administrator manages over time.

**Decided: `OrganizationRepository` gains `findById`/`update`/`search` with the same shape and reasoning as `LeadRepository`'s EAP-2 extension** — `list()` stays exactly as-is (EAP-1's Dashboard depends on it), `search()` is the new paginated entry point for the Workspace, `update()` mirrors `LeadRepository.update()`'s null-not-throw contract and partial-patch semantics exactly.

**Decided: `updated_at` is repository-owned, never caller-supplied — the first timestamp field in this codebase to work that way.** Every other timestamp (`LeadRecord.timestamp`, `AssessmentRecord.createdAt`, `OrganizationRecord.createdAt` itself) is supplied by the caller in the request body and trusted as-is (never a security concern, since none of these are used for authorization decisions). `updatedAt` is different in kind, not just in name: its entire purpose is "when did this record last actually change," and letting a caller set that value directly would make the Organization Workspace's "last activity" column assert whatever the caller claimed rather than what's true. The repository sets it to `createdAt` on `save` and refreshes it to the real current time on every `update` — the first field in this system the repository itself owns end to end, the same way it already owns `id`.

**Decided: no `riskLevel` or `assessmentStatus` filter on `OrganizationSearchOptions`, despite the brief naming both.** Investigated seriously, not dismissed reflexively. `assessmentStatus` has no backing field anywhere in this system — `AssessmentRecord` has no status/lifecycle column at all (`DECISION_LOG.md`'s own EAP-3 entry established this), so a filter on one would be inventing a concept this codebase's data doesn't have, the same discipline that kept EAP-3 from fabricating a "category" taxonomy beyond the DPDP question bank's real `section` field. `riskLevel` is real, but lives on `AssessmentRecord`, a different repository — computing "organizations with at least one assessment at risk level X" is a cross-repository join, and every repository in this codebase (five of them now) is constructed independently with its own private store, in-memory and D1 alike; there is no precedent anywhere in this system for one repository depending on another's data, and threading assessments into organization search would be a real architectural change, not an additive extension, for a query volume (organization counts, likely tens, not the hundreds a lead/assessment list might reach) that doesn't justify the cost. Risk is instead surfaced at the scope Organization Health actually needs it — one organization at a time, via the same new `AssessmentSearchOptions.organizationId` filter search already needed for Relationships.

**Decided: Organization Administration stays Platform-Administrator-only, not delegated to the existing-but-dormant `requireOrganizationAccess` helper (`auth/authorize.ts`).** That helper has existed since RC1, unwired to any route, ready for the day an organization-scoped role needs its own write access. This isn't that day: the EAP-4 brief scopes this as the platform's own tenant-management console, not a self-service customer portal (explicitly out of scope — "NOT... Customer Portal"), and delegating create/archive/restore authority to org-scoped `owner` roles would introduce a materially wider authorization surface than anything asked for, mixing "the platform administers its tenants" with "a tenant administers itself" in one gate. Every other cross-organization list/search endpoint in this codebase (six of them before this phase) already draws this same line for the identical reason (`SECURITY_GUIDE.md`'s "Authorization model") — Organization Administration's writes follow it too, for consistency, not because delegation is technically impossible.

**Decided: "notes" are `audit_events` rows (`organization.note_added`), identical to Lead's EAP-2 precedent — not a new column.** Same reasoning recorded there: a note is an audit-worthy event (who wrote what, when), and `audit_events` already has the right shape and the append-only guarantee a note should have anyway.

**Decided: archive/restore get their own named audit events (`organization.archived`/`organization.restored`), not folded into a generic `organization.updated`.** The brief names archive/restore as their own administrative actions, distinct from metadata edits — `updateOrganization` (router.ts) diffs the patch against the pre-update record exactly like `updateLead` does, but branches into a distinctly-named event specifically for a `status` transition, while every other changed field (name/industry/region/tags) collapses into one `organization.updated` event with a metadata diff. A patch that changes both status and metadata in one call correctly produces two separate, accurately-named events, not one overloaded one.

**Found and fixed — a real, pre-existing bug in EAP-3's own shipped frontend code, surfaced while wiring this phase's equivalent call: `leadApi.ts`'s `searchLeads` accepted `assessmentId` on `LeadSearchOptions` (added EAP-3) but never actually forwarded it into the request's query string.** Every existing component test mocking `fetch` matched on `url.includes("/api/leads/search")` alone — true whether or not the query string is actually correct — so nothing in this codebase's test suite could have caught it, and nothing did: Assessment Details' "Lead linkage" panel has been silently issuing `GET /api/leads/search?pageSize=100` with no scoping filter at all since EAP-3 shipped, returning up to 100 arbitrary leads system-wide rather than the ones actually linked to that assessment. The existing E2E coverage (`assessment-center.spec.ts`) didn't catch it either — it seeds exactly one linked lead and asserts that lead appears, which stays true even with zero filtering applied. Fixed: `searchLeads` now forwards both `assessmentId` and the new `organizationId`. New regression tests (`leadApi.test.ts`, a file that didn't exist before this phase) assert the real URL string built, not just that some request happened — closing the actual gap in test strategy that let this ship undetected, not just the one call site. `assessmentApi.ts`'s `searchAssessments` gained the equivalent `organizationId` forwarding correctly from the start, with the same test discipline (`organizationApi.test.ts`).

**Verified, not just implemented:** full workspace `typecheck`/`lint`/`format`/`build`/`test` — 609 tests, up from EAP-3's 497 (`@titan/web` 219, `@titan/platform` 307, `@titan/design-system` 58 and `@titan/assessment-core` 25 unchanged) — plus real operational verification against `wrangler dev` + real local D1 (real organization creation/archive/restore through the actual UI form, a real Chromium session confirming the Workspace, create-form auto-slug, search/filter, Details' Metadata/Health/Relationships/Administration/Audit all rendering real data with zero console errors, and a direct D1 query confirming `organization.*` events persist with the real caller as actor) and a new Playwright E2E spec (`apps/web/e2e/organization-workspace.spec.ts`, 4 tests) run alongside the existing four specs (16 E2E tests total, up from 12), stable across 2 consecutive clean full-suite runs plus 2 additional clean single-file runs (4 total clean runs after the one real ambiguous-`getByText` fix each in the component and E2E suites — the same "two legitimate real occurrences of similar text" class this project has hit repeatedly, resolved the same way each time: a more specific selector, never a test deleted or loosened past what it actually needed to check). Full detail: `PLATFORM_FOUNDATION.md`'s EAP-4 section.

## 2026-07-21 — EAP-3: Enterprise Assessment Center — a second "prior session's work was never committed" incident, a shared cross-cutting race fixed in both modules, and a read-only module built without inventing a lifecycle

**A resume prompt for this task described EAP-3 as substantially complete** — assessment repository search, router endpoints, an Assessment Workspace, Assessment Details, Assessment Results, Compliance Intelligence, audit integration, multiple production bug fixes, and three consecutive clean E2E runs — with the remaining work framed as documentation and final verification only, ending mid-edit on `ROADMAP.md` when a usage limit was hit.

**Found, before writing any code (per the resume prompt's own "repository evidence overrides memory" instruction, and exactly the same check the Security Release Blocker Sprint entry below already established as this project's standing discipline for a resume prompt):** none of that work existed in the repository. `git log` on `claude/session-oi9iu3` and `origin/main` both stopped at the EAP-2 merge (`ef7cd08`); `git fetch --prune` plus a scan of every remote branch found the described work on none of them; `git reflog`, `git stash list`, and `git fsck --unreachable --dangling` showed no dangling commits belonging to this work (the dangling commits that did exist were unrelated marketing-site history from weeks earlier). Direct inspection confirmed it: `AssessmentRepository` had only `save`/`findById`/`list` (no `search`), `router.ts` had no `/api/assessments/search` route, `LeadSearchOptions` had no `assessmentId` filter, and `features/admin/assessments/` did not exist as a directory at all. Listing every pull request on the repository confirmed EAP-2 (PR #42) was the most recent merge; no EAP-3 pull request existed, open or closed.

**Why, again:** this environment's sessions run in ephemeral containers reclaimed on container end, independent of whether work was committed. As with the Security Release Blocker Sprint, the prior session's design (read from its own transcript, preserved in the task upload) was coherent and genuinely informed this session's implementation — the exact bugs it described finding (a `sortBy=risk` vs `riskScore` mismatch, an `assessmentId` filter gap in lead search, an audit-trail read/write race) turned out to be real and independently reproducible by reading the current code, not hypothetical — but none of the code itself survived. Re-verifying "is this actually built" came first, exactly as both this task's and the prior sprint's own instructions required, and the answer was "no," not "yes, just polish it."

**Decided: proceed as a real implementation pass using the prior transcript as a design reference, not as a "resume."** Every decision below was re-derived and re-verified against the actual repository (types.ts, router.ts, the leads feature's shipped files, the design system's real component APIs) rather than trusted from the transcript's own account of itself — the transcript predicted a total test count that this session's own, actually-executed count did not match, which is exactly why "repository evidence, not memory" applies to a lost session's transcript too, not only to a stale summary.

**Decided: `AssessmentRepository` gains `search()` with the same shape and reasoning as `LeadRepository.search()` (EAP-2)** — `list()` stays exactly as-is (EAP-1's Dashboard and this phase's own Compliance Intelligence panel both depend on that unfiltered shape), `search()` is the new, separate, paginated entry point for the Assessment Workspace. `search`'s free-text filter matches `id`/`organizationId`/`createdBy` substrings rather than `name`/`email`/`company` — an assessment has no human-entered identity fields the way a lead does, so the search matches the identifying fields this record type actually has, not a copy of leads' field list.

**Decided: the Assessment Center is read-only — no lifecycle to PATCH.** Leads have a real CRM-style lifecycle (status/priority/assignment/tags/notes) because a lead is something an admin actively works. An assessment is "one row per completed run" (migrations/0004_assessments.sql's own comment) — there is no draft/in-progress state in this data model, no assessment-level status field, and nothing for an admin to change about a past assessment's own record. Assessment Details' "Status" therefore shows a literal, honest "Completed" — not a fabricated multi-state workflow this data model doesn't have — and no `PATCH /api/assessments/:id` endpoint, CORS method addition, or CSRF check was added, because nothing needs one.

**Decided: `LeadSearchOptions` gains an `assessmentId` exact-match filter, and `searchLeads` (router.ts) now parses it.** Assessment Details' "Lead linkage" panel (which leads, if any, this assessment produced) needs exactly this — reusing `GET /api/leads/search` rather than inventing a new `/api/assessments/:id/leads` endpoint, since `LeadRecord.assessmentId` already exists (EAP-1) and every other lead-listing need already goes through this one endpoint.

**Decided: `RiskBadge` is imported directly from `features/admin/leads/RiskBadge.js` into the assessments feature, not duplicated.** It depends only on `@titan/assessment-core`'s `RiskLevel` and `@titan/design-system`'s `Badge` — nothing lead-specific — and a cross-feature import for a small, genuinely-shared, dependency-appropriate piece already has precedent in this codebase (`useLeadDetail.ts` already imports `SectionState` from `../dashboard/useDashboardData.js`). Duplicating it would have been real, avoidable UI duplication for no reason beyond feature-folder tidiness. `FrameworkBadge`, by contrast, is a new component — no assessment-framework concept existed before this phase — with two real consumers (the Workspace table, the Details metadata panel).

**Decided: "category summaries" and "control coverage" (Compliance Intelligence, Assessment Results) group by each DPDP question's own `section` field (`@titan/assessment-core`'s question bank) — not an invented taxonomy.** The question bank has no separate "category" concept beyond the literal statutory section each control maps to (`"Section 10"`, `"Section 6"`, etc.); grouping by anything else would be fabricating a classification this codebase's own data doesn't have. Coverage counts (`passed / total` per section) come from the static bank's per-section question counts crossed with each assessment's own server-computed `result.gaps` — the bank is used purely as label/count metadata, never as a second scoring implementation; every pass/fail fact still traces to the server's own `gaps` array.

**Decided: no `EvidencePanel` component, despite Workstream 9 naming it as a possible example.** This codebase's data model has no file-upload or attachment concept anywhere — the closest real analogue is the free-text/qualification answers already surfaced in Assessment Results' "Question responses" section. Building a component for evidence attachments that don't exist would be exactly the kind of speculative feature this program's own principles rule out (the same reasoning EAP-2's report gave for not building a recommendations engine).

**Decided: Compliance Intelligence is computed entirely client-side from `GET /api/assessments`** (the existing, unfiltered, already-Dashboard-consumed full-list endpoint) — the same "aggregate view over an already-fetched list" pattern `DashboardPage.tsx`'s own `countByRiskLevel`/`countByAction` already establish, not a new server-side aggregation endpoint this scope doesn't justify. Every number (risk distribution, framework status, per-section finding counts, month-over-month average score) is arithmetic over already-server-computed `AssessmentRecord.result` values — no client-side re-scoring.

**Found and fixed — a real, pre-existing bug in EAP-2's already-shipped Lead Workspace, surfaced while building its Assessment Workspace analogue: `LeadWorkspacePage.tsx`'s "Risk" column had `id: "risk"`, but `DataTable` passes a column's `id` straight through to `onSortChange`, and only `"riskScore"` is a `LeadSearchOptions.sortBy` value `router.ts`'s `LEAD_SORT_FIELDS` accepts.** Clicking "Risk" to sort sent `sortBy=risk`, which the backend correctly rejected with a 400 — breaking the whole table's next fetch. No existing test caught it (nothing exercised clicking that specific header). Verified by reading `DataTable.tsx` + `router.ts` together, not assumed from the old transcript's account — then fixed (`id: "riskScore"`) and covered by a new regression test in `LeadWorkspacePage.test.tsx`, plus built correctly (`id: "riskScore"` from the start) in the new `AssessmentWorkspacePage.tsx`.

**Found and fixed — a second real, pre-existing bug in `useLeadDetail.ts`, and built correctly from the start in the new `useAssessmentDetail.ts`: the lead-fetch and audit-trail-fetch fired in parallel, with no ordering guarantee against the view-triggered audit write.** `getLead`/`getAssessment` (router.ts) each await their own `X.viewed` audit-event write before returning — a real guarantee, but only about that one request's own response. A second, independently-dispatched request (the audit-trail read) has no ordering relationship to it at all; it can reach the server and complete before the view-write does, silently missing the caller's own just-created event on that same page load. Fixed by sequencing the audit-trail fetch to start only after the record fetch resolves, in both hooks — closing a real race, not a hypothetical one, though its user-visible cost is small (the missed event appears on the next reload).

**Found and understood, not "fixed" — real-browser E2E verification recorded two `assessment.viewed` (and, on inspection, presumably `lead.viewed` too) audit events per single page view, in this dev-server environment specifically.** Root cause: `main.tsx` wraps the app in React 19 `<StrictMode>`, which double-invokes effects in development builds only (by design, to surface exactly the class of non-idempotent-effect bug the previous finding describes) — two real `GET /api/assessments/:id` requests genuinely reach the real Worker per dev-mode page load, and it correctly records a real event for each. This is not fixed, because there is nothing to fix: it does not occur in a production build (StrictMode's double-invocation is dev-only), both recorded events are true statements about real requests that occurred, and de-duplicating "recent identical audit events" would be a heuristic with its own real failure mode (masking a legitimately fast, real second view). The new E2E test asserts on the event's presence, not its exact count, with this reasoning recorded inline.

## 2026-07-20 — Stage 4: real repository test doubles, CORS default, and Auth.js provider scope

**Decided: repository tests run against real SQLite (sql.js), not a hand-rolled fake, as the standing pattern for every repository added in Stage 4.** Stage 3's `fakeD1.ts` proved `LeadRepository`'s SQL/binding/row-mapping logic, but as a hand-written double it couldn't rule out being wrong in the same way the code under test was wrong — it only understood the exact query shapes its author anticipated. `titan/packages/platform/src/repositories/testUtils/testD1.ts` instead runs this package's actual `migrations/*.sql` against `sql.js` (SQLite compiled to WASM) and wraps it as a `D1Database`. This is real SQL parsing, constraints, and joins — closing most of the "is this fake even correct" gap — without requiring the Vitest 3→4 upgrade the previous entry deferred. It is still not `workerd`/real D1 itself; that gap remains open (see below). `fakeD1.ts` was deleted, not kept alongside, to avoid two competing test-double patterns in the same package.

**Decided (still deferred, explicitly): real Cloudflare Workers runtime verification (`@cloudflare/vitest-pool-workers`, real `workerd`) still needs the Vitest 3→4 decision.** Nothing in Stage 4 changed the facts behind Stage 3's original deferral — every other package in this workspace is still on Vitest 3. What Stage 4 changed is how much this gap actually matters day to day: sql.js closes the "wrong SQL" class of bug locally, and `wrangler dev` against real local D1 (Workstream 10 verification, not a test) closes the "wrong Workers-runtime behavior" class manually. A real automated `workerd` test suite is still the right eventual target, not an urgent blocker.

**Decided: the Worker's CORS `Access-Control-Allow-Origin` defaults to `http://localhost:5173` (Vite's default dev port), overridable via the `ALLOWED_ORIGIN` Worker var.** This Worker has never been deployed, so there is no real production frontend origin to allow yet. The default exists purely so `titan/apps/web`'s local dev server (a different origin/port than `wrangler dev`'s `:8787`) can call it during local development — Workstream 4 doesn't work at all otherwise. Whoever deploys this for real must set `ALLOWED_ORIGIN` to the real frontend origin; the current default is not a production-safe value and `wrangler.toml`'s `[vars]` entry says so directly.

**Decided: Auth.js's Email provider runs in a dev-mode logging configuration, not a real email provider, and is built as a plain `EmailConfig` object rather than through `@auth/core/providers/email.js`'s default export.** No email provider has been chosen (`ARCHITECTURE.md`'s "still open" list, unchanged since Phase 2) — Workstream 5 still needed a working Email provider to exercise Auth.js's database-session flow end to end. `providers/email.js` is a deprecated wrapper around `providers/nodemailer.js`, which imports the `nodemailer` package at module load time even when a custom `sendVerificationRequest` never calls its SMTP transport — pulling in a real dependency this project doesn't need yet just to satisfy an unused code path. A Credentials provider was deliberately not added: the Stage 4 master prompt's Workstream 5 names exactly three providers (Email, Google, GitHub), and inventing a fourth not asked for risks exactly the kind of parallel, unrequested implementation the program's own rules warn against.

## 2026-07-20 — Hosting, database, and auth: Cloudflare-native, Auth.js, Repository Pattern

**Decided:**

- **Hosting/runtime target**: Cloudflare, full platform — Pages (app shell/marketing), Workers (backend/API), D1 (database), R2 (object storage — generated PDF reports), Queues (async work — report/email delivery), Turnstile (bot mitigation on public endpoints), plus Cloudflare's CDN/WAF/Analytics.

**Source:** the Phase 2 master prompt ("CYBERDUDEBIVASH® Enterprise — Project Titan — Phase 2 — Enterprise DPDP Platform Integration & Production Evolution"), which specifies these three under "Architectural Decisions" as directives, not options to weigh. **What this closes out:** `PLATFORM_FOUNDATION.md`'s "blocked on architecture decisions" note for authentication, authorization, database, and API foundations no longer applies to *which* hosting/auth/database engine — those three are now known. **What this does not decide:** payments provider, email provider, and the DPDP legal-review timing remain open — the Phase 2 master prompt doesn't address any of the three. See `ARCHITECTURE.md`'s "Still open" list.

**Rationale for the Repository Pattern specifically (not just "use D1"):** the master prompt states it explicitly ("Business logic must never depend directly on D1"), and it holds up independent of that instruction — D1 is SQLite-based with a narrower feature set than the Postgres `ARCHITECTURE.md`'s original Phase 1 pass recommended.
