# Architecture — CyberDudeBivash Compliance Platform (Project Titan)

Planning-phase document. Most choices below are recommendations to review, not decisions already executed. Four rows in the Tech Stack table (Backend, Database, Auth, Deployment target) are now **Decided**, per the Phase 2 master prompt's explicit architectural directives — see `DECISION_LOG.md` for the record of that and everything else decided since. Rows still marked as recommendations remain open.

## Where this lives

Chosen: inside this repository (`CYBERDUDEBIVASH-ENTERPRISE-PRODUCTION`), not a new repo. Recommendation for *how*: as a **separate top-level application directory**, e.g. `titan/`, not woven into `src/` (the existing marketing SPA).

Why separate, even in the same repo:

| Dimension | Existing site (`src/`, root `*.html`) | Titan |
|---|---|---|
| State | Stateless static content | Stateful — accounts, assessments, leads, CRM records |
| Data | None (no database) | Needs a real database, multi-tenant |
| Auth | None | Needs real authentication (admin + customer portal) |
| Deploy target | Static hosting (GitHub Pages / Cloudflare Pages) | Needs a real application host (backend + DB) — cannot deploy the same way |
| Release cadence | Content-driven, infrequent structural change | Product-driven, frequent |

A monorepo with `titan/` as its own directory (own `package.json`, own build, own deploy pipeline) keeps one repository, one git history, one PR/review convention — but doesn't force a stateful SaaS product and a static marketing site through the same build or the same deployment target, which the current repo's five-stage audit trail (`docs/audit/`) already shows causes real problems even *within* a single static site's build pipeline.

**Marketing pages for Titan** (landing, pricing, blog, resources — everything in Module 1 that needs SEO) are a different concern from **the application** (questionnaire, admin portal, customer portal — everything behind auth, none of which needs to be indexed). Recommendation: Titan's own marketing pages ship as static/SSG content (consistent with how the existing site already handles SEO-critical pages), and the authenticated app is a separate SPA. This mirrors the split the existing repo already has between its 16 static pages and its `src/` SPA — a pattern this engagement has already spent five stages validating works, including its failure modes.

## Module 1 — existing scanner asset: discovery findings (Phase A)

The Phase 2 master prompt's Phase A requires reviewing the existing scanner (`dpdpriskscan.html`, an uploaded, production-quality asset — single-file HTML/CSS/JS, ~1,660 lines) and documenting evidence before any rewrite. "Production-quality" describes the code as written, not its current runtime state — see below, that turns out to matter. This section is that review, done against the file as uploaded. **Nothing in the scanner itself has been changed as a result of these findings** — the fixes land in `titan/packages/assessment-core` (Phase C/D, built alongside the original file, proven correct with tests), not in the original file, so the working customer-facing scanner is untouched and keeps working exactly as it does today.

### Architecture as it exists today

- Single static HTML file: ~840 lines of inline `<style>`, ~570 lines of inline `<script>`, no build step, no module system, no framework.
- Two CDN dependencies load unconditionally on every page view: jsPDF 2.5.1 and Chart.js 3.9.1 (both from cdnjs). **Chart.js is loaded but never invoked** — no `new Chart(...)` call exists anywhere in the file. Confirmed dead weight, not a guess.
- State is two global mutables (`answers`, `riskData`) plus a `currentQuestion` index — reasonable for one inline script, not something that survives a module split without deliberate state ownership, which is exactly what modularization (Phase B onward) has to design for.
- Question bank: 15 entries, inline array literal (`const questions = [...]`). Matches the "15 Question Assessment" existing-asset description.

### Confirmed: the script does not currently parse — the scanner is non-functional as uploaded

Verified directly against the uploaded file (`/root/.claude/uploads/.../ce22f943-dpdpriskscan.html`, not a copy), not inferred from reading: extracting the inline `<script>` block (lines 1091–1656) and running `node --check` against it fails with a real `SyntaxError`, not a lint warning:

```
scanner_script.js:65
                text: 'Do you process children's data (<18) with verifiable parental consent?',
                                               ^

SyntaxError: Unexpected identifier 's'
```

The `children_data` question's `text` field (and, unreached because parsing already failed, its `penalty` field two lines later — `'₹200 crore (children's data violation)'`) has a bare, unescaped apostrophe inside a single-quoted JS string literal, closing the string early. A JavaScript `<script>` block that fails to parse does not partially execute — **none** of it runs: `createParticles()`, `startScan()`, the question renderer, the risk engine, `submitLead()`, `generatePDF()`. The hero section's "Start Free Risk Scan" button calls `onclick="startScan()"`, an undefined function once parsing fails, so clicking it does nothing observable (a `ReferenceError` lands in the browser console, nothing on screen). As uploaded, this is a static hero section with a non-functional CTA, not a working 15-question flow — that characterization needs correcting before anything else here is read as "review of a working scanner."

Scoped fix, confirmed by patching a scratch copy only (the real uploaded file is untouched, per Phase A's "do not rewrite yet"): escaping that single apostrophe (`children\'s data`, both occurrences) is sufficient — re-running `node --check` on the patched copy exits clean, no further syntax errors found anywhere else in the script. This is a one-character-class bug with total functional impact, not a sprawling one — worth being precise about both halves of that, since overstating scope is as much a violation of "every recommendation must include evidence" as understating it.

**This finding changes what "existing asset" means for the rest of this document and for Phase B.** The question bank data, risk-scoring logic, and PDF layout below are analyzed from reading the source (which is well-formed once the one string is fixed) and from the scratch-patched copy — they're real and were clearly authored to work — but none of it has ever actually run in this repository until now. Nothing here should be read as "this was working in production and has a subtle bug"; it should be read as "this was one syntax error away from working, is now verified logically once that's fixed, and has the further issues below."

### Confirmed correctness bug: risk score denominator is off by one

This is the next bug behind the parse error above — verified logically against the source and confirmed live against the scratch-patched copy, since the unpatched original never reaches this code path at all. The score is `Math.round((riskData.total / maxScore) * 100)`, with `maxScore` **hardcoded to `13`** independently in two places (the on-screen `calculateAndDisplayRisk` function and, duplicated, in `generatePDF`).

Actual count from the question bank: of 15 questions, 1 is free-text (`company_info`, not scored) and 2 are qualification-only (`collects_data`, `any_work` — explicitly excluded from scoring by the code's own `riskField !== 'qualification'` check). That leaves exactly **12** boolean questions that feed the score (`has_dpo`, `consent_mechanism`, `breach_sop`, `dpr_fulfillment`, `cross_border`, `children_data`, `dpa_vendors`, `retention_policy`, `privacy_notice`, `sdf_classification`, `dpia`, `incident_response`) — not 13.

Effect: an organization that fails **every** scored question gets `riskData.total = 12` and a displayed score of `round((12/13)*100) = 92`, never 100. The scanner cannot display its own worst-case score correctly against its own question bank. This is precisely the failure class this document's own testing strategy (below) already names as highest-priority — a scoring bug with real consequences for what an organization is told about its compliance posture — and it existed because the scoring logic has zero test coverage. Fixed in `packages/assessment-core`, with a regression test pinning the correct denominator so it can't silently drift again.

### Security

- **DOM-based XSS today, a stored-XSS path once persistence and an admin UI exist.** `renderQuestion()` interpolates the user's own free-text answer into an HTML attribute via `innerHTML` (`value="${answers[q.id] || ''}"`). Low severity today (self-XSS against the user's own tab only). Not low severity once Phase E (persistence) and Phase I (Admin Portal) land: `submitLead()` already stores unsanitized `name`/`email`/`company` into the lead record, and an admin UI rendering those fields without escaping is a textbook stored-XSS vector against whoever's viewing the dashboard. Needs output encoding (or a non-string-concatenation rendering approach) before Phase I, not after.
- **Email validation isn't real validation.** `submitLead()` only checks that the string contains both `@` and `.` — `@.` passes. Acceptable for a client-only demo; not acceptable once it gates a real lead pipeline (Phase G) or a real account email (Phase F).
- **The "No Data Stored" trust badge contradicts the code.** The hero section shows a trust badge reading "No Data Stored," but `submitLead()` persists the full lead record — name, email, company, every questionnaire answer, timestamp, user agent — to `localStorage` indefinitely. This is a factual-accuracy problem on a page selling data-protection compliance specifically, separate from its security severity. It needs to either become true (don't persist without disclosed consent) or be reworded before this ships as Module 1 of the real product.
- **No compliance disclaimer on the on-page results/landing UI.** `PRODUCT_VISION.md` already requires the "not a compliance guarantee" disclaimer on "every report, every results page, and every marketing page... as a stated fact of what the product is." The generated PDF has one (in `generatePDF`'s footer); the on-screen results card and the hero's "DPDP Act 2023 Compliant" badge don't. A gap against a rule this project already committed to, not a new one invented here.

### Accessibility

- The 15 answer options are `<div onclick>` elements with custom `.option`/`.option-radio` styling — not `<input type="radio">`/`<label>` or `role="radio"` + `aria-checked`. No accessible name/role/state reaches assistive tech beyond the visible text.
- Keyboard support is one global `keydown` listener with hardcoded key mappings (`1`/`y`/`Y`, `2`/`n`/`N`, arrow keys) over non-focusable divs, instead of native interactive elements that get keyboard operability for free. A keyboard-only or screen-reader user gets no signal these are the controls, or which one is currently selected.
- The progress bar and risk-score circle are purely visual — no `role="progressbar"`, no `aria-valuenow`/`aria-valuemin`/`aria-valuemax`.
- Direct gap against the bar Titan's own design system already holds itself to (`jsx-a11y` + `axe-core` on every interactive component, per `DEVELOPER_GUIDE.md`). Module 17's quality gate applies to Module 1 the same as everywhere else.

### Performance

- Chart.js (unused) and jsPDF (~500KB+, only needed once a user reaches results and downloads/submits) both load unconditionally and render-blocking, instead of being deferred or lazy-loaded until actually needed.
- Google Fonts requests seven weights of Inter (`300;400;500;600;700;800;900`); worth auditing which weights the CSS actually renders before a modularized version ships the request forward unexamined.

### Testing

Zero automated tests exist for this file today, including on the scoring logic this document's own testing strategy already designates the highest-priority target. The `maxScore` bug above is exactly the class of defect that priority exists to catch.

```
Organization ──< User (admin | customer role)
Organization ──< Assessment ──< Response >── Question ──< QuestionCategory
Assessment ──1 RiskScore ──< RiskFinding

```

Every entity above needs a tenant boundary (`organization_id` or equivalent) enforced at the query layer, not just the application layer — this is the highest-severity security requirement for a multi-tenant SaaS handling organizations' self-reported security posture data, and belongs in Module 15's acceptance criteria explicitly, not as an assumed default.

## Security architecture (Module 15, elaborated)

- **Tenant isolation**: every data access path scoped by organization ID, enforced with row-level security or an equivalent query-layer guarantee — not just checked in application code, which is one missed check away from a cross-tenant leak.
- **Auth**: see Tech Stack table — a vetted library, not custom.
- **Secrets**: environment-variable-injected at the hosting layer, never committed (this repo already has a clean track record here — `PRODUCTION_SCORECARD.md` confirms zero secrets found across five audit stages — the same discipline carries forward).
- **Input validation**: schema validation (e.g., Zod) at every API boundary, particularly the questionnaire-response endpoints, which are the highest-volume, most user-controlled input surface.
- **Rate limiting**: required on the free-assessment start endpoint specifically (abuse/scraping vector) and the AI Insights endpoint (cost-abuse vector — the existing repo's own architecture audit already flagged this exact risk class on its current Gemini-backed endpoint).
- **Audit logging**: every admin action, every access to another organization's data (support access), append-only.
- **Encryption**: TLS in transit (standard for any modern host); at-rest encryption via the database host's native support (Postgres providers generally offer this by default) rather than custom application-layer encryption, which is a common source of key-management bugs.

## Observability & testing strategy (approach, not implementation)

- **Logging**: structured (JSON) application logs, correlation IDs per request.
- **Monitoring/health checks**: a `/health` endpoint from Phase 0 onward — this repo's own `PROGRAM_BACKLOG.md` #9 already flags the *current* site's total absence of this as a real gap; Titan should not repeat it.
- **Error reporting**: a hosted error-tracking service (Sentry-class) from Phase 0 — same rationale.
- **Testing pyramid**: unit tests for the risk-scoring engine specifically get the highest priority and coverage bar, since a scoring bug is a correctness bug with real consequences for what an organization is told about their compliance posture. Integration tests for API boundaries. E2E tests for the critical path (landing → questionnaire → results, and admin login → view assessment). Accessibility and performance tests run in CI, not manually, from Phase 0 — retrofitting them later, as this repo's own architecture audit found for the *existing* site, is far more expensive than building them in from the start.

## Enterprise architecture audit (RC1 pass)

A real audit of `titan/` as it stands after Stage 4, not a self-assessment — every claim below was checked with an actual command, not asserted from memory.

**Package dependency graph — clean, no cycles.** Checked directly against each package's `package.json`:

```
@titan/design-system   (no @titan deps — leaf)
@titan/assessment-core (no @titan deps — leaf)
@titan/platform        → @titan/assessment-core
@titan/web              → @titan/assessment-core, @titan/design-system, @titan/platform
```

`@titan/web` is the only consumer of all three others; nothing depends back on `web`. Confirmed no package under `assessment-core`/`design-system` imports `@titan/platform` or `@titan/web` (`grep` across both packages' `src/`, zero matches).

**Repository Pattern boundary — held.** `router.ts` never imports a concrete `*.d1.ts`/`*.memory.ts` repository module — only the interfaces from `repositories/types.ts`, with concrete implementations injected via `Dependencies` from `worker.ts`. `grep` for `D1Database` across `packages/platform/src` returns matches only in the expected files (`*.d1.ts`, `worker.ts`, `testD1.ts`, `auth/config.ts`) — no business logic anywhere touches D1 directly.

**Shared models — single source of truth, no duplication.** `LeadRecord`/`AssessmentRecord`/`OrganizationRecord`/etc. are defined exactly once, in `@titan/platform/src/repositories/types.ts`; `grep` for a second definition anywhere in `apps/web` or `packages/assessment-core` returns nothing. `titan/apps/web`'s `leadStore.ts` re-exports `NewLead` under the name its callers use, rather than redefining the shape.

**API contract — consistent, not accidentally.** The Worker's error envelope (`http/responses.ts`'s `jsonError`: `{ error: { code, message }, requestId }`) and the frontend's parsing of it (`apiClient.ts`'s `ErrorEnvelope`) agree on shape. The client intentionally does not import the server's internal `ApiError` type — an HTTP client parsing a wire contract shouldn't need to import the server's internal response-building types, and a minimal local shape for parsing purposes is standard practice, not duplication of business logic.

**Dependency health.** `npm audit` (root workspace, all packages): 0 known vulnerabilities. `vitest`/`@vitest/coverage-v8` versions are consistent (`^3.0.0`) across all four packages — no accidental drift from the deliberate Vitest-4-deferral decision (`DECISION_LOG.md`).

**Scaling risk identified, deliberately not acted on:** `router.ts`'s route dispatch is a hand-written if/else chain. This is the right amount of structure for today's 6 routes (easy to read top-to-bottom, no framework overhead) but will not scale gracefully if an Admin/Customer Portal (`ROADMAP.md`, deferred this pass) adds the dozens of routes those need. **Recommendation, not action taken:** revisit with a real route-table/matcher once route count materially grows — introducing that abstraction now, for 6 routes, would be solving a problem that doesn't exist yet (this repository's own stated engineering principle: refactor only where measurable value exists).

**No XSS surface found.** `grep` for `dangerouslySetInnerHTML` across `apps/web/src`: zero matches, confirming Stage 3's original finding still holds after all of Stage 4's additions.

**Tooling gap found and fixed:** `packages/config/eslint.base.mjs` ignored `dist/`, `coverage/`, `node_modules/` but not `.wrangler/` — `wrangler dev`'s own local build cache (gitignored, but not excluded from ESLint's scan). This was invisible until Stage 4's real local operational verification actually ran `wrangler dev` for the first time in this workspace, which generates that directory; anyone following `OPERATIONAL_RUNBOOK.md`'s local dev instructions would hit the same `npm run lint` failure. Fixed by adding `**/.wrangler/**` to the shared ignore list.

## Admin Application architecture (EAP-1 Phase 1)

The first real, authenticated product surface built on top of the RBAC/API foundation above — an "Enterprise Administration Platform" (EAP) built as an explicitly phased program, not one pass. Phase 1's scope, set directly by the user rather than the EAP-1 master prompt's own (much larger) request: the authenticated application shell, session lifecycle, protected-route middleware, role-aware navigation, a reusable admin UI framework, and exactly one complete module (Dashboard) — architected so Lead/Assessment/Organization/User Management, an Audit Center, and an Operations Center can plug in as later phases, not built speculatively ahead of them. This section documents what that shell actually is, evidenced against the real code, the same way the RC1 audit section above does for the rest of the platform.

**Where it lives:** `titan/apps/web/src/features/admin/` (`auth/`, `layout/`, `dashboard/`), mounted at `/admin` in `App.tsx` as a sibling route tree to the public DPDP assessment flow (Stage 3's decision, `DECISION_LOG.md`) — not nested inside it, and not sharing its unauthenticated, non-`@titan/design-system` visual system. `/admin` is the first real consumer of `@titan/design-system`'s app-shell components (`Header`/`Sidebar`/`Footer`, already built in Phase 1/Stage 3 but unused by the public flow) and of the two new components this phase added (`MetricCard`, `Panel`).

**Session lifecycle — real, cross-origin, cookie-based, no parallel auth system:**

- `SessionContext`/`useSession()` (`features/admin/auth/SessionContext.tsx`) calls `GET /api/me` once on mount and exposes a `loading | authenticated | unauthenticated | error` state — no client-side session storage of its own; the Auth.js session cookie remains the single source of truth, this is only a cache of what it currently says.
- `RequireAuth` (`features/admin/auth/RequireAuth.tsx`) redirects an unauthenticated visit via `window.location.href = signInUrl(callbackPath)` — a real, full-page navigation to Auth.js's own hosted `/api/auth/signin` page (real HTML, real CSRF token, real provider list), not a custom login form duplicating what `@auth/core` already does correctly. Same for sign-out (`signOutUrl`) — `AdminLayout`'s header renders it as a real link to `/api/auth/signout`, Auth.js's own confirm-and-redirect flow.
- This only works because the SPA (`http://localhost:5173` in local dev) and the Worker (`http://localhost:8787`) are different origins, and a database-session-strategy cookie has to survive that round trip in both directions. Three changes made that real, not simulated:
  1. `http/cors.ts`'s `corsHeaders()` now sets `Access-Control-Allow-Credentials: true` (paired with `apiClient.ts`'s `fetch(..., { credentials: "include" })`) — required by the Fetch spec before a browser will expose a credentialed cross-origin response to page JS at all.
  2. `auth/config.ts`'s `redirect` callback allowlists one additional origin (`AuthConfigOptions.allowedOrigin`, threaded from the same `resolveAllowedOrigin(env.ALLOWED_ORIGIN)` CORS already uses in `worker.ts` — CORS and the redirect allowlist can't drift apart because they're the same value). `@auth/core`'s *default* redirect callback (traced directly against `lib/init.js`, not assumed) only allows a `callbackUrl` that's relative or shares the Worker's own origin — without this, every sign-in/sign-out would silently collapse back to the bare Worker root instead of returning the user to the app.
  3. **`http/finalizeResponse.ts`'s `authPagesCsp` allowlists the same origin in its `form-action` directive — found necessary only by real-browser verification, not by any unit test or curl check.** Auth.js's real sign-out confirmation page POSTs to its own origin (satisfying a bare `form-action 'self'`) and then 302-redirects to the cross-origin SPA's `callbackUrl`. The CSP spec's `form-action` directive also restricts *redirects resulting from* a form submission, not just its immediate POST target — Chromium correctly refused to complete that redirect under the previous, narrower policy, silently breaking cross-origin sign-out. Confirmed via an isolated repro (the identical form submits successfully with no `callbackUrl`, and fails only once one pointing off-origin is present) before the fix, and via a full real-browser sign-in → dashboard → sign-out run afterward. This is the single most significant finding of Phase 1 — the kind of bug that is structurally invisible to jsdom-based component tests or a curl script, and exactly why this phase's own verification requirements insist on a real browser driving a real sign-in.

**Protected-route / layout composition:** `App.tsx` nests `<SessionProvider><RequireAuth><AdminLayout /></RequireAuth></SessionProvider>` around every `/admin/*` route. `AdminLayout` is documented (and tested) as requiring a `RequireAuth` ancestor — it reads `useSession()` assuming a resolved, authenticated session rather than re-deriving loading/redirect logic itself, so that concern exists in exactly one place. `AdminLayout` reuses `@titan/design-system`'s `Header`/`Sidebar`/`Footer` (passing the real signed-in email and a real sign-out link into `Header`'s new optional `session` prop) rather than building parallel chrome, and renders `<Breadcrumbs />` (derived from `useLocation().pathname`, title-casing hyphenated segments) plus `<Outlet />` inside a landmark `<main id="main-content">` — the mount point every future module's own route nests under.

**Role-aware navigation, built for pluggability, not just for Phase 1's one module:** `adminNavItems(me: MeResponse): SidebarItem[]` (`features/admin/layout/navItems.ts`) is the single place that decides what a signed-in caller can navigate to. It returns exactly one entry today (`Dashboard`) — Phase 1 has exactly one module — but takes the caller's real `MeResponse` as its signature specifically so a later phase adding Lead/Assessment/Organization/User Management/Audit Center/Operations Center entries (some of them plausibly Platform-Administrator-only) is a change inside this one function, not a per-page decision repeated at every new route.

**Reusable admin UI framework additions (`@titan/design-system`):** `MetricCard` (label/value/hint, `isLoading` state, `aria-live="polite"`) and `Panel` (a labeled `<section>` using `useId()` for its accessible name) — the two primitives the Dashboard's metrics/sections are built from, added to the design system rather than the `admin/dashboard/` feature folder specifically so later modules reuse them instead of building similar-but-different cards/sections independently.

**Dashboard — the one complete Phase 1 module:** `features/admin/dashboard/` (`useDashboardData.ts` + `DashboardPage.tsx`) renders Executive overview, Organization/Lead/Assessment metrics, a risk-level breakdown, recent activity, an audit summary, and platform health/system status. Two real design decisions, not incidental implementation detail:

- **Per-section data loading, not one page-level gate.** `useDashboardData`'s `SectionState<T>` (`loading | ready | forbidden | error`) is fetched independently per section. `GET /health`/`GET /health/ready` are role-agnostic and always fetched; the four Platform-Administrator-gated sections (leads, assessments, organizations, audit — the same `requirePlatformAdministrator` policy `GET /api/leads` already used) are fetched only when `me.isPlatformAdministrator` is true, and resolve straight to `"forbidden"` for anyone else without firing a request that would just 403 predictably. A non-admin who reaches `/admin` (any authenticated user can — `RequireAuth` checks for a session, not a role) sees an honest "Platform Administrator role required to view this" message per section, and real platform health/status alongside it — never a fabricated metric, never a blank page.
- **No new authorization concept.** Every gated section reuses the existing Platform Administrator role (`auth/rbac.ts`'s `isPlatformAdministrator`, unchanged since the Security Release Blocker Sprint) via `GET /api/me`'s `isPlatformAdministrator` flag — computed server-side, not inferred client-side from probing a privileged endpoint and reading its status code.

**New read endpoints backing the Dashboard**, all in `router.ts`: `GET /api/me` (any authenticated caller — 401 with no session, otherwise `{userId, email, profiles, isPlatformAdministrator}`; no role gate, since identity itself isn't privileged the way a cross-organization data list is), `GET /api/organizations`, `GET /api/assessments` (list, distinct from the existing `GET /api/assessments/:id`), and `GET /api/audit` — the latter three all Platform-Administrator-only via `auth/authorize.ts`'s `requirePlatformAdministrator` (generalized this phase from `requireLeadsAccess`, which now delegates to it), for the identical reason `GET /api/leads` already was: none of `OrganizationRepository.list()`/`AssessmentRepository.list()`/`AuditRepository.list()` filter by organization, so gating any of them at "any organization member" would be a cross-tenant leak dressed up as a fix. Full policy table: `SECURITY_GUIDE.md`'s "Authorization model."

**Explicitly not built this phase** (deferred to later EAP phases, not forgotten): Lead Management, Assessment Management, Organization Management, User Management, an Audit Center, and an Operations Center as full modules. The Dashboard reads across all four of those domains (leads/assessments/organizations/audit) for metrics, but nothing here is a management UI for any of them — no create/edit/delete, no per-record detail view, no search/filter beyond what the Dashboard's own summary needs. `ROADMAP.md`'s EAP-1 section has the phase-by-phase plan.

## Lead Intelligence Platform architecture (EAP-2)

The first complete business module built on the EAP-1 shell — a production Lead Workspace (search/filter/sort/paginate), Lead Details, Lead Lifecycle management, a Risk Intelligence panel, and audit integration, scoped explicitly to Lead Intelligence only (Assessment/Organization/User Management, an Audit Center, and an Operations Center remain unbuilt — `ROADMAP.md`'s EAP section has the sequencing).

**Data model — additive, not a redesign.** `migrations/0008_lead_lifecycle.sql` adds four columns to the existing `leads` table (`status`, `priority`, `assigned_to`, `tags_json`), each defaulted so a lead captured through the public scan flow (which never sets any of them) lands in a real, sane state (`new`/`medium`/unassigned/no tags) rather than NULL. `LeadRepository` gained three new methods (`findById`, `update`, `search`) additively — `list()`, the method EAP-1's Dashboard already depends on, is completely unchanged, both in signature and behavior. **Internal notes and activity history are not new tables** — both are `audit_events` rows (the same table Workstream 6/7 built in Stage 4), reusing its real append-only guarantee (`AuditRepository`'s interface has no `update`/`delete` at all) rather than inventing a second, parallel "things that happened to this record" concept. `AuditRepository.list()` gained an optional `{entityType, entityId}` filter, additive the same way — every existing no-args caller (EAP-1's Dashboard audit summary) is unaffected, and the filter uses `migrations/0007`'s own `(entity_type, entity_id)` index, which already anticipated exactly this query shape.

**New endpoints, all reusing the existing Platform Administrator role — no new authorization concept:** `GET /api/leads/:id` (records a `lead.viewed` audit event with a real actor, unlike the anonymous `lead.created`), `PATCH /api/leads/:id` (the lifecycle write — diffs the patch against the pre-update record so only fields that actually changed produce an audit event), and `GET /api/leads/search` (server-side search/filter/sort/pagination, a separate endpoint from `GET /api/leads` specifically because its response envelope — `{leads, total, page, pageSize}` — genuinely differs from `GET /api/leads`'s bare array, not the same endpoint branching on query-param presence). `GET /api/audit` gained optional `entityType`/`entityId` filters in place, since its response shape never changes. Full policy table: `SECURITY_GUIDE.md`.

**`PATCH /api/leads/:id` is CSRF-checked the same way the anonymous POST endpoints already are** (`security/csrf.ts`'s `isTrustedOrigin`), reused for a materially different reason: it's a cookie-authenticated route, so a forged cross-origin request would ride along with the caller's real session cookie — Origin validation is more load-bearing here, not less.

**Real finding, real-browser-only: CORS `Access-Control-Allow-Methods` never included `PATCH`.** Every write before this phase went through POST; a cross-origin `PATCH` triggers a real preflight `OPTIONS` request that a browser will silently block the real request over if the response's allowed-methods list doesn't include it — invisible to `router.test.ts`/`worker.test.ts` (call `handleRequest`/`worker.fetch` directly, bypassing the browser fetch stack) and to jsdom-based component tests (no real preflight semantics). Fixed in `http/cors.ts`. Full reasoning: `DECISION_LOG.md`'s EAP-2 entry.

**Frontend architecture, `apps/web/src/features/admin/leads/`:** `leadApi.ts` (the one place this app talks to the new endpoints), `useLeadSearch`/`useLeadDetail` (own the Lead Workspace's and Lead Details' data-loading state respectively, both reusing `useDashboardData`'s `SectionState<T>` loading/ready/forbidden/error convention rather than inventing a new one), `LeadWorkspacePage`/`LeadDetailPage` (thin `useSession()` wrappers exporting a directly-testable `*Content`/`*Body` component, matching `DashboardPage`'s established split), and `StatusBadge`/`PriorityBadge`/`RiskBadge` (domain-aware wrappers around `@titan/design-system`'s generic `Badge` — kept out of the design system itself specifically to preserve its dependency-free leaf-package status, `ARCHITECTURE.md`'s own audit further up this document).

**Design-system additions (`@titan/design-system`), all genuinely generic — no `LeadStatus`/`RiskLevel` knowledge leaks into the package:** `Badge` (tone-based, the primitive the three domain badges above wrap), `SearchBar`, `Pagination`, `EmptyState`, `LoadingSkeleton`, `FilterPanel`, `Timeline`, `DataTable`. `DataTable` renders every row it's given directly — no virtualization, since nothing in this codebase's real (local-only) data volume justifies the complexity yet.

**Explicitly not built this phase** (deferred to later EAP phases): Lead Management is complete, but Assessment Management, Organization Management, User Management, an Audit Center, and an Operations Center remain unbuilt — none of their own backing endpoints exist beyond what EAP-1's Dashboard already reads. `ROADMAP.md`'s EAP section has the recommended next phase and why.

Full reasoning, plus a related React 19 `<StrictMode>` dev-mode-only double-audit-event finding that was investigated and deliberately left as-is (not a defect): `DECISION_LOG.md`'s EAP-3 entry.

## Enterprise Organization Management Platform architecture (EAP-4)

The fourth business module built on the EAP-1 shell — a production Organization Workspace (search/filter/sort/paginate), Organization Details (metadata, Health, Relationships, Administration, audit history), and, unlike Assessments, a **real administrative write surface**: create/update/archive/restore, since an organization is genuinely something a Platform Administrator manages over time, not one immutable row per run. Scoped explicitly to Organizations only — no User Management, no Billing, no Customer Portal.

**Data model — additive, real lifecycle columns, not a redesign.** `migrations/0009_organization_lifecycle.sql` adds five columns to the existing `organizations` table (`status`, `industry`, `region`, `tags_json`, `updated_at`), each defaulted so the small set of organizations already exercised by `organizationRepository.contract.ts`'s own fixtures stay valid rows. `status` is a closed two-value vocabulary (`active`/`archived`) — an organization is archived and restorable, never deleted, matching `audit_events`' own append-only guarantee (no repository anywhere in this codebase has a `delete` method). `updated_at` is **repository-owned**, not caller-supplied — unlike `createdAt` (which every other record type's caller already supplies), letting a caller set their own "last modified" timestamp would make the Organization Workspace's "last activity" column meaningless; the repository sets it to `createdAt` on `save` and refreshes it to "now" on every `update`. `OrganizationRepository` gained `findById`, `update`, and `search` additively — `list()`, EAP-1's Dashboard's own dependency, is unchanged. `LeadSearchOptions`/`AssessmentSearchOptions` each gained an `organizationId` exact-match filter, mirroring EAP-3's `assessmentId` filter pattern exactly, backing Organization Relationships' "associated leads"/"associated assessments" panels.

**New endpoints, reusing the existing Platform Administrator role — no new authorization concept, but a genuinely new write surface:** `POST /api/organizations` (create), `GET /api/organizations/search` (server-side search/filter/sort/paginate), `GET /api/organizations/:id` (records an `organization.viewed` audit event, mirroring `getLead`/`getAssessment`), `PATCH /api/organizations/:id` (metadata edits, archive/restore, notes — diffs the patch against the pre-update record so only fields that actually changed produce an audit event, exactly `updateLead`'s pattern). All four are Platform-Administrator-only, kept deliberately narrower than the existing-but-dormant `requireOrganizationAccess` helper (`auth/authorize.ts`) might suggest — this is the tenant-management console itself, not a self-service portal, so delegating administration to org-scoped `owner` roles would introduce a new, wider authorization surface nobody asked for. `POST`/`PATCH /api/organizations/:id` are CSRF-checked the same way `PATCH /api/leads/:id` already is (`security/csrf.ts`'s `isTrustedOrigin`), for the same reason: a cookie-authenticated write, so a forged cross-origin request would ride along the caller's real session.

**A deliberate scope cut, not an oversight: `OrganizationSearchOptions` has no `riskLevel` or `assessmentStatus` filter.** Both were evaluated against this codebase's actual architecture, not assumed possible. `assessmentStatus` has no backing field anywhere — `AssessmentRecord` has no status/lifecycle (`DECISION_LOG.md`'s EAP-3 entry), so filtering on one would invent a concept, not expose a real one. `riskLevel` is a real field, but only on `AssessmentRecord`, one repository over — computing it for an organization search would require a cross-repository join, and every repository in this system (Lead, Assessment, Organization, Audit, UserProfile) is constructed independently with its own private store, in-memory and D1 alike, with zero precedent anywhere in this codebase for one repository depending on another's data. Threading assessments into organization search would be a real architectural change for a query volume (organization counts, not lead/assessment counts) that doesn't justify it. Risk information is instead surfaced at the single-organization scope Organization Health actually needs, via the same new `AssessmentSearchOptions.organizationId` filter.

**Organization Health — computed client-side from an already-fetched list, the same pattern `ComplianceIntelligencePanel` established for EAP-3.** `OrganizationHealthPanel` takes this organization's own `linkedAssessments`/`linkedLeads` (already fetched by `useOrganizationDetail` for the Relationships panel) and derives current risk (the most recent assessment's real `riskLevel`), average score, a real risk-level distribution, and a real score trend (comparing the two most recent assessments' scores) — every number traces to an already-server-computed `AssessmentResult`, nothing invented, no new aggregation endpoint.

**Frontend architecture, `apps/web/src/features/admin/organizations/`:** `organizationApi.ts` (this app's only caller of the new endpoints, including the write ones — `createOrganization`/`updateOrganization` alongside the read-only pattern EAP-3 established), `useOrganizationSearch`/`useOrganizationDetail` (own the Workspace's and Details' data-loading state, both reusing `SectionState<T>`; `useOrganizationDetail` additionally owns `isSubmitting`/`submitError`/`update`, the same write-state shape `useLeadDetail` established for EAP-2, since Organizations — like Leads, unlike Assessments — have something to write), `OrganizationWorkspacePage` (thin `useSession()` wrapper plus an inline create form — this module's first *creation* UI, since neither Leads nor Assessments have an admin-facing create flow; auto-slugs the identifier field from the name, still directly editable), `OrganizationDetailPage` (composes `OrganizationHealthPanel`/`OrganizationRelationshipsPanel`/`OrganizationAdministrationPanel`/`OrganizationAuditPanel`), and `OrganizationStatusBadge` (the domain-aware wrapper pattern `StatusBadge`/`RiskBadge`/`FrameworkBadge` already established). `OrganizationRelationshipsPanel` deep-links into the existing Lead Intelligence and Assessment Center modules (`/admin/leads/:id`, `/admin/assessments/:id`) rather than duplicating either's own detail view — the brief's explicit instruction ("Reuse existing modules. Never duplicate them").

**Real, pre-existing bug found and fixed while wiring the frontend, in EAP-3's own shipped code: `leadApi.ts`'s `searchLeads` never forwarded `assessmentId` into the request's query string, despite `LeadSearchOptions` accepting it since EAP-3.** Every component test mocking `fetch` matched on `url.includes("/api/leads/search")` alone, which passes whether or not the query string is actually correct — so Assessment Details' "Lead linkage" panel has been silently calling `GET /api/leads/search?pageSize=100` with no scoping filter since EAP-3 shipped, returning up to 100 arbitrary leads instead of the ones actually linked to that assessment. Found only because wiring the equivalent `organizationId` call for this phase required reading the same function closely. Fixed (both `assessmentId` and `organizationId` now forwarded), with new regression tests (`leadApi.test.ts`) asserting the real URL built, not just that some request happened — the same gap in the test strategy that let it ship undetected.

**Explicitly not built this phase** (deferred to later EAP phases, not forgotten): User Management, an Audit Center, and an Operations Center as full modules. No organization-scoped delegation of any cross-organization list/search endpoint — every gate stays Platform-Administrator-only, the same scope line EAP-1 through EAP-3 already drew. `ROADMAP.md`'s EAP section has the recommended next phase and why.

## Enterprise Identity & User Management architecture (EAP-5)

The fifth business module built on the EAP-1 shell — a production Enterprise User Directory (search/sort/paginate over real identities), User Details (identity, Role Assignment, Relationships, audit history), and this platform's real Role Assignment/Administrative User Lifecycle write surface: grant, change, and revoke a user's organization or platform-wide role.

**Data model — no new table, and a deliberate read/write split across two existing repositories rather than one new one.** Identity itself (`name`, `email`, `emailVerified`, `image`) already lives in Auth.js's own `users` table (migrations/0001), written exclusively by `@auth/d1-adapter` on every real sign-in; a new, read-only `UserRepository` (`findById`/`search`, no `save`) reads it without ever writing it — inventing a parallel user-creation path would fabricate an identity with no real authenticated principal behind it. Role/organization membership already lived in `user_profiles` (RC1/Workstream 6), extended this phase with `findById`/`list`/`update`/`remove` — the real write surface behind Role Assignment. `UserProfileRepository.remove()` is this system's **first real deletion**: every other repository either has none or is archive-and-restore only, but a `(user, organization, role)` grant has no dependent data of its own to preserve the way an organization's linked leads/assessments do, so an archive-not-delete pattern would have nothing real to protect here. The revoked role's own value is recorded in `user.role_revoked`'s audit metadata *before* the row is removed, so the historical fact of the grant survives in the append-only log even though the live table no longer reflects it.

**New endpoints, reusing the existing Platform Administrator role — no new authorization concept:** `GET /api/users/search` (server-side search by name/email, sort, paginate), `GET /api/users/:id` (identity composed with the user's own `UserProfileRecord[]`, records a `user.viewed` audit event), `POST /api/users/:id/profiles` (grant — validates the target user and, if given, the target organization both exist; 409 on a duplicate `(user, organization)` grant), `PATCH /api/users/:id/profiles/:profileId` (change role) and `DELETE /api/users/:id/profiles/:profileId` (revoke) — both validate the `:profileId` path segment actually names a profile belonging to the `:id` segment's user before touching anything. All five are Platform-Administrator-only. The three profile-write routes are CSRF-checked the same way `PATCH /api/organizations/:id` already is, and are additionally gated by `wouldRemoveLastPlatformAdministrator` (router.ts) whenever the profile being touched is itself a platform-wide grant — a new guard, not present before this phase, that rejects (409) any change that would leave the system with zero Platform Administrators and no recovery path. `DELETE` is a genuinely new HTTP method for this codebase (`http/cors.ts`'s `ALLOWED_METHODS` gained it in the same change that introduces the route, learning directly from EAP-2's own postmortem — a missing CORS method entry there silently broke `PATCH /api/leads/:id` until real-browser verification caught it after the fact; this phase's own E2E spec verifies `DELETE` works in a real browser proactively instead).

**A deliberate scope cut, not an oversight: `UserSearchOptions` has no `organizationId`/`role` filter.** The identical repository-isolation reasoning `OrganizationSearchOptions`'s missing `riskLevel` filter already established (`DECISION_LOG.md`'s EAP-4 entry): a user's identity and their grants are two different repositories with no query-layer dependency on each other anywhere in this codebase, and filtering users by organization/role would require exactly the cross-repository join this architecture doesn't support. The real consequence: Organization Details gets no reciprocal "Members" panel this phase — User Details shows a user's own memberships (via the already-existing `UserProfileRepository.findByUserId`), not the other direction. Real, recommended near-term follow-up (`ROADMAP.md`), not built this pass.

**`AssessmentSearchOptions` gained a `createdBy` exact-match filter, mirroring EAP-3's `assessmentId` and EAP-4's `organizationId` additions exactly.** Backs User Relationships' "assessments created by this user" panel. Combined with the already-existing `LeadSearchOptions.assignedTo` (EAP-2), User Details' Relationships panel needed no other backend surface — both filters already existed on their own repositories, just never had a real user-directory-driven consumer before this phase.

**Frontend architecture, `apps/web/src/features/admin/users/`:** `userApi.ts` (this app's only caller of the new endpoints), `useUserSearch`/`useUserDetail` (own the Workspace's and Details' data-loading state, `SectionState<T>` throughout; `useUserDetail` additionally owns `isSubmitting`/`submitError`/`grant`/`changeRole`/`revoke` — the same write-state shape `useOrganizationDetail` established, applying its audit-trail-sequencing fix from the start), `UserWorkspacePage` (deliberately smaller than every sibling Workspace — no `FilterPanel`, no saved filters, no column toggle, no create-form; see `DECISION_LOG.md`'s EAP-5 entry for why each omission is a real, reasoned scope cut rather than an oversight), `UserDetailPage` (composes `RoleAssignmentPanel`/`UserRelationshipsPanel`/`UserAuditPanel`), `RoleAssignmentPanel` (the real grant/change/revoke write surface — every control a direct server call through `useUserDetail`, never an optimistic local-only update, same discipline as `OrganizationAdministrationPanel`), and `RoleBadge` (the domain-aware `Badge` wrapper pattern `StatusBadge`/`RiskBadge`/`FrameworkBadge`/`OrganizationStatusBadge` already established — named `userRole`, not `role`, as its prop, since a plain `role` prop on a JSX element is indistinguishable, to both a static a11y linter and a future reader, from the DOM's own ARIA `role` attribute). `UserRelationshipsPanel` deep-links into the existing Lead Intelligence and Assessment Center modules rather than duplicating either's own detail view, the same reuse discipline `OrganizationRelationshipsPanel` established.

**A real environmental finding, not a code defect: this session's container had never had `packages/platform/.dev.vars` provisioned, so `AUTH_SECRET` was unset and every caller — even one with a genuinely valid session cookie — silently resolved as anonymous.** Looked, at first, exactly like a browser/cookie/proxy problem; reproduced deliberately against the already-shipped `organization-workspace.spec.ts` before concluding anything about new code, confirming it wasn't EAP-5-specific; root-caused with a direct `curl` request bypassing the browser entirely, which narrowed it to server-side session resolution; closed by following `OPERATIONAL_RUNBOOK.md`'s own already-documented setup step. Full account: `DECISION_LOG.md`'s EAP-5 entry.

**Explicitly not built this phase** (deferred to later EAP phases, not forgotten): an Audit Center and an Operations Center as full modules. No self-service creation of a user's underlying identity (Auth.js's own sign-in is the only path — see the data-model paragraph above). No Organization-side "Members" panel (see the scope-cut paragraph above). No real assignee picker wired into `LeadLifecyclePanel.tsx` — EAP-5 closes the *reason* that was deferred (no user directory existed), but wiring the picker itself into an already-shipped EAP-2 component is real, recommended near-term follow-up, not this phase's own scope. No organization-scoped delegation of any cross-organization list/search endpoint — every gate stays Platform-Administrator-only, the same line every prior phase already drew. `ROADMAP.md`'s EAP section has the recommended next phase and why.

## Enterprise Audit Center architecture (EAP-6)

The sixth business module built on the EAP-1 shell — a single operational interface for reviewing every audit event any of the prior five modules generates, rather than a seventh place `audit_events` gets read from. Consolidates, it does not replace: every existing per-entity audit panel (Lead/Assessment/Organization/User) is untouched and still reads `GET /api/audit?entityType=...&entityId=...` exactly as before; the Audit Center is a new, additional way to read the same table, not a redesign of how any module produces audit events.

**Data model — no new table, no new column, one new repository method.** `AuditRepository` (`migrations/0007_audit_events.sql`, unchanged) gains `search(options: AuditSearchOptions): Promise<AuditSearchResult>` alongside its existing `record`/`list` — additive, mirroring `OrganizationRepository.search()`'s exact shape (case-insensitive substring `search`, exact-match `actorId`/`organizationId`/`action`/`entityType`/`entityId`, an inclusive `dateFrom`/`dateTo` range, `sortBy`/`sortDirection`, `page`/`pageSize`). `AuditSortField` has exactly one member (`"createdAt"`) — there is no second field on this table anyone has ever had a reason to sort by, unlike Lead/Organization/Assessment/User's own multi-field sort unions. `list()`'s existing no-args/entity-scoped behavior is untouched: the Dashboard's audit summary and every per-entity panel depend on it exactly as before.

**Two new endpoints, both Platform-Administrator-only, sharing one filter parser:** `GET /api/audit/search` (the Workspace's paginated/filtered/sorted view) and `GET /api/audit/export` (the identical filters, unpaginated and capped at 10,000 rows, returned as a real CSV or JSON file download via `Content-Disposition: attachment` rather than a JSON envelope). Both call a single `parseAuditSearchOptions(url)` helper — no separate, potentially-drifting filter validation between the two. Neither route needs a "not configured" 503 branch the way EAP-4/EAP-5's `organizations`/`users`-backed routes do: `Dependencies.audit` has always been a required field, unlike those two optional ones.

**A CSV/formula-injection risk (CWE-1236) was investigated and found not present in this data model, verified rather than assumed.** The only genuinely free-form export column is `metadata`; every other column is a UUID, a fixed enum this codebase's own code writes, or a timestamp. `metadata` is always emitted via `JSON.stringify` first, so an exported cell structurally always begins with `{` (or is empty) — confirmed with a test asserting this real, currently-true property, not a defensive escape added for a scenario ruled impossible by the data shape itself. Recorded in `SECURITY_GUIDE.md` as a verified-safe finding, revisit only if a future phase adds a raw, non-JSON-wrapped free-text export column.

**Found and fixed — a real cross-origin CORS gap, caught only by real Playwright/Chromium E2E verification, the identical class of finding EAP-2's missing `Access-Control-Allow-Methods` entry for `PATCH` already was.** `http/cors.ts` never set `Access-Control-Expose-Headers`, so a real browser's cross-origin `fetch()` silently withheld the `Content-Disposition` response header from JavaScript even though it was present and correct on the wire — every unit/component test (mocked `fetch`, no real CORS enforcement) passed regardless, and the export still "worked" under casual inspection, just under a browser-guessed fallback filename instead of the server's real one. Fixed by adding `Access-Control-Expose-Headers: Content-Disposition`, with a new router-level regression test asserting the header going forward.

**Shared components, consolidated rather than added as a sixth private copy:** `auditActionLabels.ts` — one canonical action/entity-type label map, replacing four near-identical private `ACTION_LABELS` maps the Lead/Assessment/Organization/User audit panels each already carried (a genuine fifth consumer justified the extraction; the four existing panels' own behavior and tests are otherwise unchanged). `AuditEntityBadge` — a `Badge` wrapper linking into an entity's own detail page when one exists, with three real consumers (Workspace table, Investigation grouping, event detail panel).

**Frontend architecture, `apps/web/src/features/admin/audit/`:** `auditApi.ts` (search + export, the latter via `apiClient.ts`'s new `getBlob` — the one response shape in this app that isn't JSON, so it needed its own helper rather than a variant of `getJson`), `useAuditSearch` (owns only a sort *direction*, not a sort *field* — there is nothing else to make configurable, per the data-model note above), `AuditWorkspacePage` (search/filter/sort/paginate/export, a Table/Investigation view toggle, saved filters and column preferences via `auditWorkspacePreferences.ts` — the same `localStorage`-backed pattern every sibling Workspace already uses), `AuditEventDetailPanel` (Audit Details — an inline panel over an already-loaded record, not a new route or backend endpoint; its "related events" section reuses the pre-existing `GET /api/audit?entityType=...&entityId=...` endpoint rather than inventing a new one, and states plainly, rather than omitting silently, that this event log has no request-context or result-status columns to show), and `AuditInvestigationView` (Investigation View — groups the *currently loaded page* of search results by entity/actor/organization, deliberately not a correlation engine that fetches and holds every matching row across every page).

**Explicitly not built this phase:** investigation notes/annotations (the brief's own brief marked this optional, "only if the repository supports them" — it doesn't; a real schema addition, not an additive read, so left for a future phase to decide, not spec-cut here). A truncation signal on a 10,000+-row export (the cap itself is the real control; see "Found and fixed" above and `SECURITY_GUIDE.md`'s known-gaps entry). Organization-scoped delegation of `GET /api/audit/search`/`GET /api/audit/export` — both stay Platform-Administrator-only, the same line every prior phase's own cross-organization endpoints already drew, for the identical repository-isolation reason (`AuditSearchOptions.organizationId` is an optional filter a Platform Administrator can supply, not a scope any caller is restricted to). `ROADMAP.md`'s EAP section has the recommended next phase and why.

## Enterprise Operations Center architecture (EAP-7)

The seventh business module built on the EAP-1 shell — the first to consolidate operational (not business) data: real per-service reachability, real request/repository-operation telemetry, and a static system overview, all composed into one read. No new table, no new writer, no new authorization concept.

**Data model — no schema change at all.** `AuditRepository`/`LeadRepository`/`AssessmentRepository`/`OrganizationRepository`/`UserRepository` all keep their existing `search()` shape unchanged; `UserProfileRepository.list()` (RC1/Workstream 6) is reused as-is. The one real interface change is additive: `Metrics` (`observability/metrics.ts`) gains `getCounts()`/`getDurations()` on the interface itself — the concrete `createInMemoryMetrics` implementation already had both methods, instrumented since an earlier phase's own Workstream 8, but nothing anywhere in this codebase ever read them back until this phase needed to.

**One new endpoint, reusing the existing Platform Administrator gate — no new authorization concept:** `GET /api/operations/summary` composes (1) a real reachability check per service — `leads`/`assessments`/`organizations`/`users`/`userProfiles`/`audit`, each a genuine `search({page:1,pageSize:1})` (or `list()` for `userProfiles`) rather than a fabricated ping, timed through the same `withOperationTiming` every other repository call already uses, so this endpoint's own page loads feed real samples back into the very metrics it displays; (2) `ctx.metrics.getCounts()`/`.getDurations()`, this Worker isolate's own real, accumulating request/repository-operation counters; (3) a static `SystemOverview` (version/environment/registered-module list) — real, verified values, not dynamically computed ones, documented as such in the code rather than implied to be more automated than they are. `GET /health`/`GET /health/ready` are untouched — this phase adds no gate to either and reads no new information out of them; the Operations Workspace calls them directly, the same way the Dashboard (EAP-1) already does.

**A deliberate, evaluated exception to this codebase's general error-handling rule:** `ServiceStatus.error` returns a caught exception's real message text to the caller, rather than the generic `internal_error` `handleRequest`'s own top-level catch returns everywhere else. Accepted because only a Platform Administrator can ever reach this endpoint — the same trust boundary that already sees raw audit metadata and every real user's email — and showing the real failure cause is the endpoint's actual purpose; every failure is also logged server-side, so this is additive to observability, not a replacement for it. Full reasoning: `SECURITY_GUIDE.md`'s "Authorization model" section.

**Frontend architecture, `apps/web/src/features/admin/operations/`:** `operationsApi.ts` (the one new endpoint only — `/health`/`/health/ready` are called directly via `getJson`, the same way `useDashboardData.ts` already does, rather than wrapped a second time), `useOperationsData` (the same `SectionState`/`loadSection` convention `useDashboardData.ts` established — health/readiness role-agnostic and always fetched, the summary Platform-Administrator-gated), `OperationsWorkspacePage` (Platform Health, Readiness, Service Status, Runtime Metrics, Background Operations, System Overview — six `Panel`s, each independently loading/erroring), `ServiceStatusPanel` (a `DataTable` of real per-service reachability), `RuntimeMetricsPanel` (two `DataTable`s — request counts, repository operation durations — with an honest empty state, never a fabricated zero row, when this isolate hasn't served anything yet), and `SystemOverviewPanel` (a definition list of the three real, static overview fields).

**Explicitly not built this phase:** investigation-style background-job/queue infrastructure (none exists anywhere in this codebase — `wrangler.toml` has no `[[queues]]`/`durable_objects`/`[triggers]` block, confirmed by reading it directly — an honest note in the Background Operations panel, not a fabricated queue view). A dynamically-read build hash/timestamp (no CI step injects one anywhere in this repository; `SystemOverview.version`/`.environment` are real but manually-maintained constants instead, documented as such). Organization-scoped delegation of `GET /api/operations/summary` — stays Platform-Administrator-only, the same line every prior cross-organization endpoint already drew. A cross-isolate/durable metrics backend (`Metrics` stays in-memory and per-isolate, the same limitation `security/rateLimiter.ts` already documents for the identical reason — a real fix needs Workers Analytics Engine, which needs a deployed Cloudflare account this project has never had). `ROADMAP.md`'s EAP section has the recommended next phase and why.

## Enterprise Reporting & Analytics architecture (EAP-8)

The eighth business module built on the EAP-1 shell — the first that is pure composition over every prior module's own data rather than a new entity of its own: an Executive Dashboard, Business Reports, and an Analytics panel, all backed by a live, request-time aggregation over the same repository reads Dashboard/Compliance Intelligence/Operations Center already call. No new table, no new repository method beyond what already existed, no new authorization concept.

**Not what `ROADMAP.md` said it would be — worth stating plainly, not silently reinterpreting.** EAP-7's own "Recommended next EAP phase" note described this as "the layer that can finally read from the pre-existing, never-written-to `reports` table (`migrations/0006`)." That table tracks a different concept entirely — one row per generated PDF report artifact for an assessment, the persistence side of Roadmap Module H (server-side reporting/storage/email), still genuinely blocked on the undecided email/storage providers, exactly as before this phase. This module composes real repository reads instead (Roadmap Module 10, business intelligence over already-real data) — see `DECISION_LOG.md`'s EAP-8 entry for the full reasoning on why building a `reports`-table producer here would have meant fabricating a pipeline this phase was never asked to build. The `reports` table remains at zero rows, untouched by this phase.

**Data model — no schema change at all.** `LeadRepository`/`AssessmentRepository`/`OrganizationRepository`/`AuditRepository`/`UserRepository`/`UserProfileRepository` all keep their existing `list()`/`search()` shape unchanged. The one real addition is `auth/rbac.ts`'s `countPlatformAdministrators(profiles)` — a system-wide sibling to the existing per-caller `isPlatformAdministrator`, sharing its same `organizationId === null && role === "owner"` predicate rather than re-deriving the rule.

**Three new endpoints, reusing the existing Platform Administrator gate — no new authorization concept:** `GET /api/reports/summary` (the `ExecutiveSummary` the Executive Dashboard and Business Reports panels both render — organization/lead/assessment/identity/audit counts and breakdowns, one aggregating read composed from `Promise.all`-parallelized `list()`/`search({page:1,pageSize:1})` calls, each timed through the same `withOperationTiming` every other repository call already uses), `GET /api/reports/trends` (a `TrendSeries` — one time-bucketed count per UTC calendar day over a caller-chosen window, for one of five entities: leads/assessments/organizations/audit/identity), and `GET /api/reports/export` (the same `ExecutiveSummary`, as a real CSV or JSON file download via `Content-Disposition`, mirroring `exportAuditEvents`'s shape exactly). All three share `resolveCaller` + `requirePlatformAdministrator`, the same gate every cross-organization list/search/export/summary route has used since EAP-1, for the identical repository-isolation reason: the summary itself composes unfiltered, cross-organization data from six repositories at once.

**A deliberately more conservative export than the one it's modeled on.** `GET /api/audit/export` returns individual, potentially-identifying audit events (capped at 10,000 rows as its real control). `GET /api/reports/export` returns only aggregate counts and breakdowns — no name, email, company, or other individual-record field appears anywhere in `ExecutiveSummary` — so it needs no row cap of its own; there is exactly one CSV row per known enum value or observed free-form key (e.g. `leads,status.new,3`), never one per underlying record. Reuses the same blanket `Access-Control-Expose-Headers: Content-Disposition` CORS policy EAP-6 already established (`http/cors.ts`) rather than needing a new one — verified directly, not assumed, since that header is applied to every response, not scoped to the audit routes specifically.

**`GET /api/reports/trends`'s five entities read three different real data paths, not one uniform one, because the underlying repositories genuinely don't offer the same query shape.** Leads/Assessments/Organizations read the same unfiltered `list()` the Dashboard/Compliance Intelligence already call, filtered to the requested window in memory — none of their `*SearchOptions` has a date-range filter today, and adding one to all three purely to serve this one panel would be the same kind of speculative repository-capability addition `DECISION_LOG.md`'s EAP-4 entry already declined for a similar reason. Audit volume/Identity activity instead read `AuditRepository.search()` with a real `dateFrom` filter (`entityType: "user"` for Identity activity, reusing the same `user.role_*`/`user.viewed` events EAP-5 already writes) — `AuditSearchOptions` already has a real date-range filter (EAP-6), and a bounded `pageSize: 10_000` sample keeps one wide-window request from attempting to bucket this deployment's entire audit history. The `organizations` entity fails closed to 503 when `deps.organizations` isn't wired, the same convention every optional-dependency route has followed since EAP-4.

**Frontend architecture, `apps/web/src/features/admin/reporting/`:** `reportingApi.ts` (this app's only caller of the three new endpoints — `getJson` for summary/trends, `getBlob` for export, the same split `auditApi.ts` established), `useReportingData` (the Executive Dashboard/Business Reports' shared `SectionState<ExecutiveSummary>`, the same `useDashboardData.ts` convention every sibling Workspace's own top-level hook already follows), `useTrendSeries` (the Analytics panel's own `SectionState<TrendSeries>`, re-fetching on entity/window change), `ExecutiveSummaryPanel` (real KPI `MetricCard`s — an unconfigured `organizations`/`identity` renders "—" with a "Not configured" hint, the same idiom `DashboardPage.tsx`'s own `overviewMetricProps` established, never a zero indistinguishable from a real one), `BusinessReportsPanel` (every breakdown a real `DataTable` over a server-computed count, reusing the exact tabular primitive `ServiceStatusPanel`/`RuntimeMetricsPanel` already established rather than a bespoke table per report), `AnalyticsPanel` (an entity/date-range selector — `localStorage`-backed, per-browser, the same saved-view convention every sibling Workspace already uses — rendering `TrendSparkline` alongside a real, accessible `DataTable` of the identical per-day counts, so a screen-reader user is never left with the decorative shape alone), and `ReportingWorkspacePage` (composes all three panels under one `Panel`-per-section layout, plus the Export CSV/JSON buttons, matching `OperationsWorkspacePage.tsx`'s established multi-panel shape).

**New design-system primitive:** `TrendSparkline` — a minimal, dependency-free inline SVG polyline (five to ninety points), not a charting library dependency, matching `DataTable`'s own stated preference for the simplest real implementation over a new build dependency for a shape this simple. Its `aria-label` is a real, data-derived sentence, not a generic "chart" announcement; carries its own `axe-core` check, consistent with every other design-system component.

**Explicitly not built this phase:** a materialized/persisted report record via the `reports` table (see above — a different roadmap item, still blocked on an undecided email/storage pipeline). Organization-scoped delegation of any of the three new endpoints — all three stay Platform-Administrator-only, the same line every prior cross-organization endpoint already drew. A truncation/row-cap signal on Report Export — not needed, since its output is bounded by the number of known enum values and observed free-form keys, not by the number of underlying records the way Audit Export's is. `ROADMAP.md`'s EAP section has the recommended next phase and why.

## Enterprise Customer Portal architecture (CPP-1)

The first business module built for a customer rather than a Platform Administrator — a real, separate authenticated shell (`/portal`) sitting alongside the EAP-1 Admin Application (`/admin`), not nested inside it, reusing the same session/RBAC/repository layer rather than inventing a parallel one. Where every EAP module answers "what can the platform's own operators see across every organization," CPP-1 answers the inverse question: "what can one real organization's own member see about themselves, and never anyone else's." The caller is not a new role — a real Organization Member/Admin/Owner (`auth/rbac.ts`'s existing three-role model, unchanged since the Security Release Blocker Sprint) using their own existing Auth.js session.

**The one new architectural device, `resolvePortalOrganizationId(caller, requestId): string | Response` (`router.ts`), is the load-bearing piece of this entire module.** It finds the first of the caller's own `UserProfileRecord[]` with a non-null `organizationId` and returns it, or a 403 if none exists. Every portal handler receives that resolved id as an ordinary function parameter — never a query string, path segment, or request-body field — so there is no code path by which a caller can widen their own scope. This is architecturally the mirror image of every EAP list/search endpoint's own Platform-Administrator-only gate: those stay cross-organization because their repositories' `list()`/`search()` have no per-organization filter at all (widening them to "any organization member" would leak every other organization's rows); CPP-1's routes instead force that same filter to always be present and always be the caller's own, using the identical repository methods.

**Data model — one new table, everything else reused unchanged.** `AssessmentRepository.search({organizationId, ...})`, `OrganizationRepository.findById()`, and `AuditRepository.search({organizationId, ...})` are the exact same repository methods the admin Assessment Workspace, Organization Details, and Audit Center already call — CPP-1 adds no new method to any of them, only a caller that always supplies the tenant filter. The one genuinely new piece is `SupportRequestRepository` (`save`/`listByUser`, in-memory + D1, contract-tested against real SQLite the same way every repository since Stage 4 has been) and its backing table, `migrations/0010_support_requests.sql` — the one entity in this brief with no existing analogue anywhere in the data model.

**Seven new endpoints, one new authorization primitive, zero changes to any existing route:** `GET /api/portal/organization`, `GET /api/portal/assessments`, `GET /api/portal/reports/summary`, `GET /api/portal/reports/export`, `GET /api/portal/activity`, and `GET`/`POST /api/portal/support`. Verified directly via `git diff` against the pre-CPP-1 baseline, not just asserted: every touched file (`router.ts`, `router.test.ts`, `worker.ts`, `index.ts`, `repositories/types.ts`, `App.tsx`) shows only insertions — 948 lines added, zero removed — confirming no existing admin route, handler, or authorization check changed shape. Assessment Detail is a deliberate non-addition: it reuses `GET /api/assessments/:id` exactly as EAP-2/EAP-3 built it, because `requireAssessmentAccess` already delegates to "a Platform Administrator, or a member of the assessment's own organization" — precisely the access a portal customer needs, with nothing to add.

**`PortalComplianceSummary` is a new, deliberately smaller type, not `ExecutiveSummary` (EAP-8) reused with an organization filter bolted on.** Most of `ExecutiveSummary`'s sections have no honest single-organization form at all: leads carry no `organizationId` in this data model (`SECURITY_GUIDE.md`'s own note), Platform Administrator counts are inherently platform-wide, and reusing "audit" as a rollup here would leak cross-tenant activity shape through what looks like a scoped endpoint. `buildPortalComplianceSummary` instead composes only what has a real per-organization meaning: this organization's own assessments, counted by risk level and framework, with the latest assessment date — computed via the same `AssessmentRepository.search()` the admin Assessment Workspace already calls, just with `organizationId` forced rather than accepted as an optional filter.

**Frontend architecture, `apps/web/src/features/portal/`:** `layout/portalNavItems.ts` (`hasPortalOrganizationMembership`/`portalNavItems` — both return nothing at all for a caller with no real organization profile, not a broken or partially-rendered nav), `layout/PortalLayout.tsx` (reuses `Header`/`Sidebar`/`Footer`/`Breadcrumbs`/`useSession`/`signOutUrl` directly from the admin feature folder rather than duplicating any of them, rendering an honest "No organization membership" `Alert` instead of `<Outlet/>` for a non-member), `portalApi.ts` (this app's only caller of the seven new endpoints, using the same `getJson`/`getBlob`/`postJson` primitives every other feature's own API module already does), `AssessmentSummaryCard` (the one new shared component this module needed — two real consumers, Dashboard and Reports — composing the existing `RiskBadge`/`MetricCard` rather than duplicating either), and five page modules each following the established `*Page`/`use*` `SectionState` convention: `dashboard/` (Organization Overview, Compliance Summary, Recent Activity — three independently-loading sections), `assessments/` (a read-only `DataTable` plus Assessment Detail, which reuses `fetchAssessment`/`AssessmentResultsPanel` from the admin `assessments` feature folder directly rather than rebuilding either), `reports/` (the compliance summary plus real CSV/JSON export downloads, mirroring `AuditWorkspacePage.tsx`/`ReportingWorkspacePage.tsx`'s own established export-button pattern), `account/` (`PortalAccountPage` — a thin `useSession` wrapper — plus the exported, directly-testable `PortalAccountContent({me})`, the same `*Content`-prop-driven testability convention `AssessmentDetailPage`/`ReportingWorkspacePage` already established), and `support/` (a submission form plus the caller's own request history via the shared `Timeline` component).

**Explicitly not built this phase:** an organization switcher for a caller with more than one real organization membership (`resolvePortalOrganizationId` resolves to the first found — a named scope limit, `DECISION_LOG.md`'s CPP-1 entry). A support-request ticketing/resolution workflow — no admin-side endpoint reads or updates `support_requests` anywhere in this codebase; `status` starts and stays `"open"`. Password change, notification preferences, and multi-session management on the Account page — no credentials provider, settings/notification service, or `SessionRepository` beyond Auth.js's own single-session-token model exists to back any of them. Enterprise APIs (Roadmap Module 14) and bookings (Roadmap Module 13's other half) — both explicitly out of CPP-1's own brief. No Commercial Platform, Billing, Subscriptions, Licensing, Payments, Cloudflare Production, Release Candidate, or General Availability work — per this program's own standing instruction to complete CPP-1 first. `ROADMAP.md`'s CPP-1 section has the full detail and recommended next phase.

## Enterprise Commercial Platform architecture (COM-1)

The commercial layer built on top of the verified EAP-1–EAP-8 and CPP-1 foundation: real, provider-agnostic Subscriptions and Licenses, a code-defined Plan catalog, and a real UI on both the admin side (`/admin/commercial`) and the customer side (`/portal/subscription`) — nothing here processes a payment, stores a card, or completes a chargeable transaction. Where CPP-1 answers "what can this organization's own member see about their own data," COM-1 answers a genuinely different question — "what commercial tier is this organization on, and what does that entitle them to" — and needed real new state (Subscription, License) to answer it, unlike CPP-1's own portal routes, which composed entirely from data that already existed.

**The Plan catalog is deliberately not a repository.** `commercial/planCatalog.ts` defines `PLAN_CATALOG` — three real plans (Starter, Professional, Enterprise), each with a name, a display-only price string, a trial length, and an entitlement set — as a plain, versioned, exported constant, the identical pattern `@titan/assessment-core`'s `dpdpV1` question bank already established for real, typed content that ships with the application rather than being authored at runtime. This is a genuine architectural choice, not a shortcut: a real `PlanRepository` would imply real plan CRUD, and with it proration semantics for existing subscribers on a price change — a materially larger, differently-scoped feature this brief never asked for. `findPlan(planId)`/`isSelfServicePlan(plan)` are the catalog's own two pure query functions; nothing in `router.ts` reaches into `PLAN_CATALOG` directly.

**Subscription and License are two separate repositories, both genuinely new, both narrowly 1:1 with an organization.** `SubscriptionRepository` (`findByOrganizationId`/`findById`/`save`/`search`/`update`) is the organization-level commercial record — plan, status, trial/period dates — the same real tenant-scoped shape `OrganizationRepository` itself has. `LicenseRepository` is the seat *grant*, created alongside its subscription (`seatLimit`, `status`, `activatedAt`/`expiresAt`). Seat *usage* is deliberately not a column on either — it's `UserProfileRepository.findByOrganizationId(organizationId).length`, a new, narrowly justified method on the pre-existing repository (the mirror image of `findByUserId`, which finds one user's memberships across organizations rather than one organization's members across users). This is the same "compose from what already exists rather than duplicate its data" discipline CPP-1's own `PortalComplianceSummary` established for assessment counts, applied here to keep seat usage from ever drifting out of sync with real membership.

**Eight new endpoints, one shared internal state machine, zero changes to any existing route.** Verified directly via `git diff` against the pre-COM-1 baseline: every touched file (`router.ts`, `router.test.ts`, `worker.ts`, `index.ts`, `repositories/types.ts`, `userProfileRepository.memory.ts`/`.d1.ts`/`.contract.ts`, `App.tsx`, `navItems.ts`, `portalNavItems.ts`/`.test.ts`) shows only insertions — 1751 lines added, zero removed. `applySubscriptionPatch` is the one shared function both `updatePortalSubscription` (customer-facing, gated by `resolvePortalOrganizationId` + `isSelfServicePlan`) and `updateAdminSubscription` (Platform-Administrator-facing, no such plan restriction) call after each validates the incoming patch against its own, differently-scoped rules — it diffs the patch against the pre-update record to derive real `subscription.upgraded`/`.downgraded`/`.canceled`/`.renewed` and `license.activated`/`.expired` audit events, the same diff-before-recording discipline `updateOrganization`/`updateLead` already established, extended here to keep a subscription's own license status in lockstep (cancel expires it, renew reactivates it) rather than letting the two drift apart.

**Feature Entitlements are a pure function, not a repository or a route gate.** `resolveEntitlements(plan, subscription)` (`commercial/entitlements.ts`) takes a `Plan` and a subscription's own `status`, returning the plan's full entitlements while `trialing`/`active` or a collapsed "nothing" shape once `canceled`/`expired`. It is called exactly once in the request path today — inside `getPortalCommercialSummary`, purely to compose a response the frontend displays — and by nothing that decides whether to serve a request. Wiring it as an enforcement check on an existing route (CPP-1's `GET /api/portal/reports/export`, or the Assessment/Reporting Engine) is real, named future work this phase deliberately didn't attempt, since both are systems this program's own instructions name as do-not-redesign. See `DECISION_LOG.md`'s COM-1 entry.

**Frontend architecture spans both existing shells, not a new one.** `apps/web/src/features/admin/commercial/` owns the shared pieces both consumers need: `commercialApi.ts` (this app's only caller of all eight commercial endpoints), `PlanCard`/`SubscriptionStatusBadge`/`SeatUsageMeter`/`EntitlementBadge` (each with two real consumers — the admin pages that "own" them and the Customer Portal's own Subscription page, which imports them cross-feature, the same established pattern `RiskBadge`/`FrameworkBadge` already set for CPP-1), `CommercialWorkspacePage`/`CommercialSubscriptionDetailPage` (the admin module itself, following the exact `*Content`-exported Workspace+Detail shape EAP-4's Organization Management already established, with no create-form — a subscription is created by self-service or an admin override on an *existing* organization, never authored from a blank form), and `useCommercialSubscriptions`/`useCommercialSubscriptionDetail`. `apps/web/src/features/portal/subscription/` holds only what's genuinely portal-specific: `PortalSubscriptionPage` (combining Commercial Dashboard content and lifecycle actions on one page, the same "dashboard content plus the actions that act on it" shape Organization Details already established for Health+Administration) and `usePortalSubscription`.

**Explicitly not built this phase:** a real payment gateway integration (COM-1 is provider-agnostic by its own brief — no card/invoice data or chargeable transaction exists anywhere in this system). Entitlement enforcement on any existing route (computed and displayed, not yet a gate — see above). A multi-subscription-per-organization concept (`SubscriptionRepository.findByOrganizationId` returns at most one row by construction, the same real 1:1 tenant shape every other entity in this system already has). Dynamic plan authoring/CRUD (the catalog is code-defined, the same reasoning the DPDP question bank has never been admin-editable either). A separate License Inventory page (License is 1:1 with Subscription — its own fields surface as columns/sections on the Commercial Workspace/Detail pages instead of a redundant separate list of the same rows). Accounting, Tax Engines, Marketplace, Affiliate Programs, CRM, and ERP — all explicitly out of COM-1's own brief. No Cloudflare Production, Release Candidate, or General Availability work — per this program's own standing instruction to complete COM-1 first. `ROADMAP.md`'s COM-1 section has the full detail and recommended next phase.

## Enterprise Production Infrastructure architecture (PRD-1)

Where every prior phase (EAP-1 through COM-1) added application capability, PRD-1 adds nothing application-facing at all — it is the deployment/configuration/operational layer underneath everything already built, scoped explicitly to exclude new features, a redesign of any existing system, or an actual production deployment (no Cloudflare account has ever existed to deploy to).

**Named environments live entirely in `wrangler.toml`, not in application code.** `[env.staging]`/`[env.production]` each restate their own `[[d1_databases]]`/`[vars]` blocks in full — Wrangler's named-environment config does not inherit the top-level (local-dev) blocks, a real behavior confirmed by running `wrangler deploy --dry-run --env staging` before and after adding the blocks and comparing the resolved bindings, not assumed from documentation. Every value that would need to be real before an actual deploy (`database_id`, `ALLOWED_ORIGIN`) is a literal, grep-able `REPLACE_WITH_REAL_*` placeholder — the same honest-placeholder convention the pre-existing default environment's own `database_id` already established, extended rather than invented.

**Configuration validation is a pure function, not a startup hook — because Workers has no startup hook to use.** `config/validateEnv.ts`'s `validateProductionConfig(env)` is called on every request (cheap, no I/O) rather than once at boot, because `env` bindings only exist inside `fetch(request, env, ctx)` in the Workers runtime, not at module evaluation time — a real, documented platform constraint, not a design shortcut. Its result is logged once per isolate (a module-scoped flag prevents log-flooding) and surfaced continuously via `GET /api/operations/summary`'s new `configuration` field (EAP-7's existing operational-status endpoint, extended rather than duplicated) — Platform-Administrator-only, and every issue message names a field and a reason, never a secret's actual value.

**`RUNTIME_ENVIRONMENT`, a hardcoded literal since EAP-7, becomes `RUNTIME_ENVIRONMENT_FALLBACK` — still hardcoded, still the honest default, now genuinely overridable.** `SystemOverview.environment` reads `deps.configValidation?.environment` when a deployment supplies it (every real deployment will, via `worker.ts`'s own computation from `env.ENVIRONMENT`) and falls back to the exact same `"local development (never deployed)"` string every prior phase already shipped when it's absent — every existing test, and any future caller that doesn't need this, is unaffected by construction (verified: the full pre-existing test suite passes unmodified).

**Release automation is real, standalone Node scripts, not a rewrite of `wrangler` itself.** `packages/platform/scripts/` (`check-wrangler-config.mjs`, `validate-secrets.mjs`, `db-backup-local.mjs`, `smoke-test.mjs`, `release.mjs`) each wrap one real `wrangler` command or a small, targeted check, chained by `release.mjs` into config-check → secrets-check → deploy → smoke-test. None of these needed a new dependency — Node 22's own `child_process`/`fetch`/`fs` were sufficient, consistent with this workspace's standing preference for the simplest real implementation over a new package. A real bug was caught and fixed while building `check-wrangler-config.mjs`: its first version mis-identified `[env.staging.vars]` as a new top-level environment (both start with the literal text `[env.`), silently truncating its own placeholder scan before it ever reached `ALLOWED_ORIGIN` — caught by running the script for real against the file it was meant to check, not by code review alone.

**CI/CD is a second, independent workflow, not an extension of `titan-ci.yml`.** `titan-deploy.yml` duplicates `titan-ci.yml`'s five quality-gate steps rather than sharing them via a reusable workflow, a deliberate choice to avoid touching a stable, already-working file for this phase's own, functionally different concern (deployment, not PR-gating) — the same "never redesign a stable system" discipline this program's own instructions apply to application code, applied here to CI configuration itself. It triggers only via `workflow_dispatch` — never on push or tag — so an accidental push can never attempt a real deployment.

**Explicitly not built this phase:** any R2/Queues/Durable Objects/Turnstile binding (named in the original Phase 2 master prompt's Cloudflare-platform recommendation, but nothing in this codebase generates a file to store, dispatches an async job, needs cross-request coordination, or gates a public form against bots yet — adding any of the four now would be configuration with nothing to exercise it). Custom domain/DNS/TLS configuration (needs a real Cloudflare account and a real domain, neither of which exists). A scheduled/automated remote backup job (needs a real remote D1 database to schedule backups *of*). An artifact-promotion mechanism distinct from "redeploy the same commit to a different environment" (Cloudflare Workers deploys from source on every call — there is no platform primitive to promote a build artifact the way some other platforms have). Any actual production deployment, Release Candidate, or General Availability work — per this program's own standing instruction to complete PRD-1 first. `DEPLOYMENT_GUIDE.md`/`DISASTER_RECOVERY.md`/`ENVIRONMENT_GUIDE.md` have the full detail.

## Enterprise Operations & Site Reliability Engineering architecture (OPS-1)

Where PRD-1 built the deployment layer, OPS-1 builds the operational layer on top of what EAP-7 and PRD-1 already shipped — extending the existing Enterprise Operations Center (`GET /api/operations/summary`, `/admin/operations`), never re-describing or duplicating it. No new application feature, no redesign of any verified system, explicitly excluding Enterprise Security & Compliance (SEC-1) and Release Candidate/General Availability work, per this program's own standing instruction to complete OPS-1 first.

**Aggregation is a new, pure leaf module, not logic bolted onto the router.** `observability/aggregate.ts`'s `computeLatencyPercentiles` (nearest-rank method) and `computeErrorRate` take the exact `RecordedCount[]`/`RecordedDuration[]` shapes `Metrics.getCounts()`/`.getDurations()` already produce (EAP-7) and return real p50/p95/p99 latency and a real 4xx/5xx rate — nothing in this codebase computed either before this phase, confirmed by a repository-wide search before writing it. Zero I/O, so trivially unit-testable with fixed inputs and exact expected outputs, the same discipline `config/validateEnv.ts` already established.

**Alert evaluation is a second pure leaf module, deliberately kept out of `router.ts`'s own knowledge of Cloudflare bindings.** `observability/alerts.ts`'s `evaluateAlerts` takes a narrow, locally-defined `ServiceHealthInput` shape (not `router.ts`'s own `ServiceStatus`) specifically so this module never imports from `router.ts` — the dependency direction stays one-way, router depends on observability, never the reverse. Every returned `Alert` carries the real evidence (the actual rate, the actual latency, the actual failing service name) that triggered it — never a severity without a number behind it, the same disclosure posture `ServiceStatus.error` already established for this Platform-Administrator-only endpoint.

**Readiness now reflects configuration validity, not only database reachability.** `router.ts`'s `computeReadiness` is a new function shared by `readinessResponse` (`GET /health/ready`) and `operationsSummary`, so both report the exact same real answer to "is this deployment ready to serve traffic" rather than two independently-drifting checks. An invalid `staging`/`production` configuration (PRD-1's `validateProductionConfig`) now fails readiness on its own terms, checked before the database — a real orchestrator honoring readiness should stop routing to a deployment that would reject every real cross-origin request just as surely as it should stop routing to one with an unreachable database. Absent-safe by construction: `deps.configValidation` stays optional, and every pre-OPS-1 test that never set it keeps its exact prior behavior (verified: the full pre-existing test suite passes unmodified).

**The Operations Center dashboard gained three additions, all composed from data the endpoint already gathers — no new signal invented at the UI layer.** An Alerts panel (`AlertsPanel.tsx`) rendering `evaluateAlerts`'s real output, with an honest "No alerts firing" success state when the array is empty rather than rendering nothing (which would be indistinguishable from a failed load). A "Request health" section in the existing Runtime Metrics panel, showing the new error-rate/percentile fields alongside the pre-existing raw count/duration tables it already rendered. An Operational Summary banner, deriving one line (`healthy`/`degraded`/`critical`) from `highestSeverity(alerts)` plus the pre-existing System Overview's own version/environment — a composition of what the panels beneath it already say in more detail, never a new, independent claim.

**Never redesigned:** `router.ts`'s route table, the six existing service-reachability checks, `Metrics`'s existing counter/duration recording, or any pre-existing field on `OperationsSummary` — every OPS-1 change to that interface is a new, additional field (`requestSummary`, `alerts`), verified by every pre-existing test (`router.test.ts`'s `operationsSummary` describe block) passing unmodified.

**Explicitly not built this phase:** real delivery of an alert to a human — no PagerDuty/Slack/email/SMS integration exists, because no deployed environment produces real traffic to evaluate and no paging provider has ever been chosen (the same "no forcing decision yet" reasoning `ARCHITECTURE.md`'s own payments/email provider gaps already have). A durable, cross-isolate metrics/alert history (needs Workers Analytics Engine, needs a deployed Cloudflare account — the identical, already-named limitation `SECURITY_GUIDE.md` has for EAP-7's own runtime metrics). Route-based code-splitting for `@titan/web`'s main bundle (PRD-1's own named finding, re-confirmed this phase at 834.46 kB/186.42 kB gzip — still out of an operations-only phase's scope, since it's frontend-routing restructuring, not an operations capability). Any real incident, SLO measurement, or capacity constraint — `INCIDENT_RESPONSE_GUIDE.md`/`SRE_GUIDE.md` are real, usable frameworks built ahead of the traffic that would exercise them, not evidence any of the three has actually happened. `MONITORING_GUIDE.md`/`INCIDENT_RESPONSE_GUIDE.md`/`SRE_GUIDE.md`/`OPERATIONS_GUIDE.md` have the full detail.

## Enterprise Security & Compliance architecture (SEC-1)

Where OPS-1 built the operational layer, SEC-1 audits and hardens the security posture every prior phase already built — extending, never redesigning, Authentication/Authorization/RBAC/the Assessment or Reporting Engines/the Administration or Customer Platforms/Production or Operations Infrastructure/the database architecture/the Repository Pattern/shared components/business logic, per this program's own standing instruction. Before any change, two research passes (one covering existing security documentation, one reading every security-implementation file directly — `auth/rbac.ts`, `auth/authorize.ts`, `auth/config.ts`, `auth/session.ts`, `security/csrf.ts`, `security/rateLimiter.ts`, `http/cors.ts`, `http/finalizeResponse.ts` — in full) verified precisely what already existed, so this phase closes only genuine, evidenced gaps.

**A real, honestly-classified dependency finding, not a blindly-applied fix.** A fresh `npm audit` (the first run against this exact workspace state) found 3 high-severity CVEs in `sharp`, pulled in transitively by `miniflare`, pulled in by `wrangler` (a devDependency). Confirmed via `npm ls sharp` and a direct search of both `apps/web/src` and `packages/platform/src` that `sharp` is never imported by any application code — unreachable from the deployed Worker or any real request. `npm audit fix --force --dry-run` (a command that, contrary to its name, actually mutated `node_modules` when run — caught immediately via `git status` showing the lockfiles untouched, and recovered with a clean `npm ci`) revealed the "fix" would downgrade `wrangler` from `4.112.0` to `4.15.2` — every version since `4.16.0` carries the identical transitive vulnerability, so there is currently no available fix that doesn't regress this project's own load-bearing deploy tooling by roughly 97 minor versions. Deferred, with the full reasoning in `THREAT_MODEL.md`'s findings register (SEC-1-01) — not silently ignored, not blindly force-applied.

**A real CSP gap closed: `frame-ancestors` was missing from every route's Content-Security-Policy.** Per the CSP3 spec, `frame-ancestors` does not inherit from `default-src` — `default-src 'none'` (`STRICT_CSP`) restricted zero framing on its own. Clickjacking protection had rested entirely on the legacy `X-Frame-Options: DENY` header since this codebase's own RC1 phase. Fixed by adding `frame-ancestors 'none'` to both `STRICT_CSP` and `authPagesCsp` (`http/finalizeResponse.ts`) — genuine defense in depth, not a response to any known exploit, since `X-Frame-Options` already fully mitigated the real risk on every browser that honors it.

**A real rate-limiting inconsistency, partially closed.** `POST /api/organizations` (an authenticated, Platform-Administrator-only write) was already rate-limited, contradicting this codebase's own in-code comment claiming rate limiting exists "specifically for the anonymous, unauthenticated POST routes." Three customer-facing writes named explicitly in this program's own brief — `POST /api/portal/support`, `POST`/`PATCH /api/portal/commercial/subscription` — now share the same general rate limiter, closing the gap where it mattered most (a compromised or malicious organization member's own session, a materially more plausible abuse vector than an already-privileged, already-audited Platform Administrator's). The remaining Platform-Administrator-only writes were deliberately left as they were — extending to seven more admin-only call sites for marginal benefit was judged out of proportion to the real risk, a scope decision recorded in `SECURITY_ARCHITECTURE.md`'s rate limiting section, not an oversight.

**CI/CD hardened with explicit least-privilege permissions, matching an existing convention rather than inventing one.** Neither `titan-ci.yml` nor `titan-deploy.yml` declared a `permissions:` block, unlike `deploy.yml` (the marketing site's own pipeline), which already had one. Both now declare `permissions: contents: read` — the complete real requirement, since neither workflow calls the GitHub API for anything beyond checkout.

**Precision correction for future documentation, not a code change: this system has one role enum with three values (`member`/`admin`/`owner`), not "four roles."** Anonymous/Authenticated are session-presence states, not roles; Platform Administrator is a sentinel convention (`organizationId: null` + `role: "owner"`), not a fourth enum value. `SECURITY_ARCHITECTURE.md`'s authorization flow section uses this precise framing throughout.

**Explicitly not built this phase:** any change to `auth/rbac.ts`, `auth/authorize.ts`, `auth/config.ts`, or the underlying role model — every one was read in full and verified correct, not touched. Rate limiting for the remaining seven Platform-Administrator-only writes (a deliberate scope decision, not an oversight — see above). A forced dependency downgrade to resolve SEC-1-01 (would regress `wrangler` by ~97 minor versions — deferred, not applied). Any real penetration test, security certification, or compliance audit — `THREAT_MODEL.md`'s penetration-test-preparation section and `COMPLIANCE_GUIDE.md` are real preparation material, explicitly not a claim that either has actually occurred. `SECURITY_ARCHITECTURE.md`/`THREAT_MODEL.md`/`COMPLIANCE_GUIDE.md` have the full detail.
