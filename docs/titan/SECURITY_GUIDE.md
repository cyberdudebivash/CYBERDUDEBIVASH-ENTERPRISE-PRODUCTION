# Security Guide — Project Titan

A threat model and control review for `titan/`, written from the system as it actually exists after EAP-3 (Enterprise Assessment Center) — not a generic checklist. Every "mitigated" claim below names the actual code that does it; every "open" item names why it's open and what would close it. See `PLATFORM_FOUNDATION.md` for the broader implementation status and `ARCHITECTURE.md` for the architecture audit this guide's threat model is scoped against.

## System surfaces in scope

1. **Public DPDP assessment flow** (`titan/apps/web`'s `/assessment/dpdp`) — unauthenticated, browser-facing.
2. **Worker API** (`titan/packages/platform`) — `GET /health`, `POST`/`GET /api/leads`, `GET /api/leads/:id` (EAP-2), `PATCH /api/leads/:id` (EAP-2), `GET /api/leads/search` (EAP-2, gained an `assessmentId` filter in EAP-3), `POST /api/assessments`, `GET /api/assessments/:id` (EAP-3 adds an `assessment.viewed` audit event), `GET /api/assessments` (list, EAP-1), `GET /api/assessments/search` (EAP-3), `GET /api/organizations` (EAP-1), `GET /api/audit` (EAP-1; EAP-2 adds optional `entityType`/`entityId` query filters), `GET /api/me` (EAP-1), `/api/auth/*`. `GET /api/leads`, `GET /api/leads/:id`, `PATCH /api/leads/:id`, `GET /api/leads/search`, `GET /api/assessments/:id`, `GET /api/assessments/search`, and (EAP-1) `GET /api/assessments`/`GET /api/organizations`/`GET /api/audit` are authorization-gated — see "Authorization model" below. **The Assessment Center (EAP-3) is read-only: no `PATCH`/`DELETE` endpoint exists for assessments** — an assessment is one immutable row per completed run, with no lifecycle to mutate (`DECISION_LOG.md`'s EAP-3 entry), so there is no new write surface to threat-model here beyond the two read endpoints above.
3. **D1 database** — organizations, users, sessions, user_profiles, assessments, leads, reports, audit_events.
4. **Auth.js authentication** — database sessions, Email (dev-mode)/Google/GitHub providers. Now reached cross-origin, by a real browser session, from the admin SPA (`titan/apps/web`'s `/admin`, EAP-1) — not just via curl/tests, as it was through ERP-1.
5. **Audit trail** — `audit_events`, written on lead/assessment creation.
6. **Admin Application** (`titan/apps/web`'s `/admin`, EAP-1) — authenticated, browser-facing; session lifecycle (sign-in redirect, sign-out) and a Dashboard reading surfaces 2/3/5 above through the caller's own browser session, not a service credential.

## Authorization model (Security Release Blocker Sprint)

Roles, from lowest to highest privilege:

| Role | Definition | Grants |
|---|---|---|
| Anonymous | No session (`GET /api/auth/session` returns no user, or no session cookie at all) | `POST /api/leads`, `POST /api/assessments` (the public assessment flow — deliberately stays anonymous, `PRODUCT_VISION.md`), `GET /health`, `GET /health/ready` |
| Authenticated User | A real session, but no `user_profiles` row anywhere | `GET /api/me` (their own identity — EAP-1). May reach the admin app's `/admin` shell and its Dashboard (`RequireAuth` checks for a session, not a role), but every Platform-Administrator-gated section on it resolves to an honest "forbidden" state, not data |
| Organization Member/Admin/Owner | A `user_profiles` row with a real `organizationId` and `role` of `member`/`admin`/`owner` (`auth/rbac.ts`'s existing three-role model, unchanged) | Everything Authenticated User has, plus read access to their own organization's assessments (`GET /api/assessments/:id` when the assessment's `organizationId` matches) |
| Platform Administrator | A `user_profiles` row with `organizationId: null` and `role: "owner"` — a grant not scoped to any organization (`auth/rbac.ts`'s `isPlatformAdministrator`) | Everything an Organization Owner has, for every organization, plus `GET /api/leads`, `GET /api/leads/:id`, `PATCH /api/leads/:id`, `GET /api/leads/search` (EAP-2), `GET /api/organizations`, `GET /api/assessments` (list), `GET /api/assessments/search` (EAP-3), `GET /api/audit` (see below for why these are Platform-Administrator-only, not delegated to org roles) — i.e. the admin Dashboard's real data (EAP-1), the Lead Workspace/Lead Details modules (EAP-2), and the Assessment Workspace module (EAP-3) |

Endpoint-to-policy map:

| Endpoint | Policy | Enforced by |
|---|---|---|
| `GET /health`, `GET /health/ready` | Anonymous | No check (liveness/readiness only, no sensitive data) |
| `POST /api/leads`, `POST /api/assessments` | Anonymous (+ Origin/CSRF check, rate limit) | `security/csrf.ts`, `security/rateLimiter.ts` — unchanged this sprint |
| `GET /api/me` | Any authenticated caller (401 with no session; no role gate) | `router.ts`'s `resolveCaller` + `meResponse` |
| `GET /api/leads`, `GET /api/leads/:id`, `GET /api/leads/search`, `GET /api/organizations`, `GET /api/assessments` (list), `GET /api/assessments/search` (EAP-3), `GET /api/audit` | Platform Administrator only | `router.ts`'s `resolveCaller` + `auth/authorize.ts`'s `requirePlatformAdministrator` (EAP-1 — `requireLeadsAccess` now delegates to this same function; EAP-2's `GET /api/leads/:id`/`GET /api/leads/search` and EAP-3's `GET /api/assessments/search` route through the same gate, so every cross-organization list/search route shares one policy) |
| `PATCH /api/leads/:id` (EAP-2) | Platform Administrator only, + Origin/CSRF check | `router.ts`'s `resolveCaller` + `requireLeadsAccess`, plus `security/csrf.ts`'s `isTrustedOrigin` — the same forgery check the anonymous `POST /api/leads`/`POST /api/assessments` routes use, extended to this authenticated write because a forged cross-origin request here would also ride along the caller's own session cookie, making the check more important, not less |
| `GET /api/assessments/:id` | Platform Administrator, or a member (any role) of the assessment's own `organizationId` | `router.ts`'s `resolveCaller` + `auth/authorize.ts`'s `requireAssessmentAccess`. EAP-3: also records a real `assessment.viewed` audit event with the caller as actor, on the success path only (mirroring `getLead`'s `lead.viewed`) |
| `/api/auth/*` | Auth.js's own action-level rules | `@auth/core`, unchanged |

**Why `GET /api/leads`/`GET /api/leads/:id`/`GET /api/leads/search`/`GET /api/organizations`/`GET /api/assessments`/`GET /api/assessments/search`/`GET /api/audit` are Platform-Administrator-only, not delegated to organization roles:** none of `leadRepository`/`organizationRepository`/`assessmentRepository`/`auditRepository`'s `list()`/`search()` methods have a per-organization `WHERE` clause — each returns every organization's rows in one unfiltered query (confirmed by reading each `*.d1.ts` directly, including EAP-2's `leadRepository.d1.ts`'s `search()` and EAP-3's new `assessmentRepository.d1.ts`'s `search()`). Gating any of them at "any organization member" would grant a member of Organization A visibility into every other organization's data too — a real cross-tenant leak dressed up as a fix, not a real one. Until each repository supports organization-scoped filtering (tracked below, "Known, accepted gaps"), only a caller with no single organization to be scoped to — a Platform Administrator — may call any of these routes at all. `GET /api/me` is a deliberate exception, not an oversight: it returns only the caller's own identity/profiles, the same data `GET /api/auth/session` plus a repository lookup already exposes to that same caller, so it carries none of the cross-tenant risk the list endpoints do.

**EAP-3: `GET /api/leads/search`'s new `assessmentId` filter (backing Assessment Details' "Lead linkage" panel) inherits the same Platform-Administrator-only gate as every other call to that endpoint — it does not carry its own, narrower policy.** A consequence worth naming explicitly: an Organization Member who can read an assessment directly (via `GET /api/assessments/:id`'s per-record delegation) will still see `{status: "forbidden"}` for that same assessment's own Lead linkage panel, because the underlying lead search remains cross-organization-unfiltered. The frontend (`useAssessmentDetail.ts`) surfaces this honestly as its own independent "forbidden" state, scoped to just that one panel, rather than either hiding the panel or fabricating empty data.

**Why `GET /api/leads/:id`/`PATCH /api/leads/:id` are Platform-Administrator-only rather than delegated like `GET /api/assessments/:id` (EAP-2):** a single lead, unlike a single assessment, doesn't carry a reliable `organizationId` to delegate against — `LeadRecord.organizationId` is nullable, and no code path that creates a lead (`validation/` has no `organizationId` field anywhere in `validateNewLead`) ever sets one, so every lead in this system today has `organizationId: null` (leads are captured anonymously, before any organization relationship exists — `PRODUCT_VISION.md`). "Is the caller a member of this lead's organization" is therefore not an answerable question for any lead yet, unlike assessments. Both routes reuse `requireLeadsAccess` rather than inventing a third policy shape for this one pair of routes.

**Why `GET /api/assessments/:id` *is* delegated to organization roles:** unlike the list endpoints, a single assessment fetch is naturally scoped — the record being read has exactly one `organizationId` (or none), so "is the caller a member of *that* organization" is a correct, answerable question per request, unlike an all-organizations list.

**Provisioning the first Platform Administrator** is a direct D1 insert, not a self-service flow — there is no route that grants this role (by design: an endpoint that lets a caller grant themselves platform-wide access would be its own privilege-escalation vulnerability). See `OPERATIONAL_RUNBOOK.md`'s "Provisioning a local Platform Administrator" section — its verification step now includes signing in through the real admin app UI (`/admin`), not only curl.

**EAP-1: this authorization boundary is now reached from a real, cross-origin browser session (the admin SPA), not only from same-origin tests/curl.** That required the Worker to actively support crossing origins with credentials — `Access-Control-Allow-Credentials: true` (`http/cors.ts`), an Auth.js `redirect` callback allowlisting the SPA's origin (`auth/config.ts`), and a CSP `form-action` allowlist for the same origin (`http/finalizeResponse.ts`) — each scoped to the one configured `allowedOrigin`, never a wildcard, so this doesn't broaden who can call these endpoints, only where the legitimate caller's own browser is allowed to run.

Out of scope for this document: anything requiring a deployed Cloudflare account (Turnstile, WAF rules, Cloudflare Access) — this project has never had one, in this session or any prior stage. See `ROADMAP.md`'s deferral record for why.

## Threat model (STRIDE)

### Surface 1 — Public assessment/lead submission

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing | Automated bot submits fake leads at volume | Per-IP rate limiting (`security/rateLimiter.ts`, 30/min on `POST /api/leads`) | Mitigated, but weak — see "Open: bot protection" below |
| Tampering (CSRF) | A malicious site's page issues a forged cross-origin `POST /api/leads` on a visitor's behalf | Origin-header validation (`security/csrf.ts`) — a present-but-mismatched `Origin` is rejected with 403 | **Mitigated** (RC1) |
| Tampering (data integrity) | A client submits a `result` (risk score) that doesn't match the `answers` it also submitted — e.g. reporting LOW risk while every answer indicates otherwise | `router.ts`'s `createLead`/`createAssessment` recompute `result` server-side via `@titan/assessment-core`'s `scoreAssessment`, discarding whatever the client sent in the request body. `createAssessment` first resolves `framework`/`frameworkVersion` against a small registry (`resolveQuestionBank`) and rejects an unrecognized pair with 400 `unsupported_framework` rather than trusting it | **Mitigated (Security Release Blocker Sprint).** Verified against a real running Worker, not just tests: a `POST /api/leads` submitting `answers` for two failed critical DPDP controls alongside a client-claimed `score: 0`/`riskLevel: "low"` persisted the real computed result (`score: 17`, two real gaps with their actual penalties) — the tampered value never reached D1 |
| Information disclosure | Lead form fields, error messages | React's default escaping (no `dangerouslySetInnerHTML` anywhere — confirmed by `grep`, `ARCHITECTURE.md`); error responses carry a `code`/`message`, never a stack trace or internal detail (`http/responses.ts`) | Mitigated |
| Denial of service | Volumetric abuse of the public form | Per-IP rate limiting — real, but per-Worker-isolate only (see "Open" below) | Partially mitigated |

### Surface 2 — Worker API

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing | Forged `cf-connecting-ip` header used to evade rate limiting | Cloudflare sets `CF-Connecting-IP` itself at the edge and strips/overwrites any client-supplied value before a Worker ever sees it — this only holds once actually deployed behind Cloudflare; a direct `wrangler dev` instance (as used for all local verification so far) has no such guarantee and trusts whatever header value is sent | Mitigated once deployed; **not applicable to local-only verification today** — documented so this isn't mistaken for a deployed guarantee |
| Tampering | Malformed/oversized request bodies | Every write endpoint validates required fields and object shapes (`validation/primitives.ts`) before touching a repository; invalid JSON returns 400, not a 500 | Mitigated |
| Repudiation | A write or a privileged read happens with no record of who/what caused it | Every lead/assessment creation records an `audit_events` row (`router.ts`'s `recordAuditEvent`); `lead.viewed` (EAP-2) and, as of EAP-3, `assessment.viewed` are recorded on every real authorized read of `GET /api/leads/:id`/`GET /api/assessments/:id`, with the real caller as actor; a failed audit write is logged but never silently swallowed | Mitigated for the actions instrumented so far (lead/assessment creation and viewing); login/logout/report-generation audit events are not yet emitted because those flows don't exist yet (no portal, no server-side reports) |
| Information disclosure | Verbose internal errors leak stack traces/paths | Centralized error handling (`router.ts`'s top-level `try/catch`) returns a generic `internal_error` to the client while logging the real error server-side with a correlating `requestId` | Mitigated |
| **Information disclosure (ERP-1 finding)** | **`GET /api/leads` had no authentication at all — any caller could list every captured lead: name, email, company, DPDP answers, risk result.** Originally verified directly against `router.ts`'s route dispatch. | `router.ts`'s `resolveCaller` (a real Auth.js session lookup + `UserProfileRepository.findByUserId`) gates the route via `auth/authorize.ts`'s `requireLeadsAccess`, requiring Platform Administrator | **Fixed (Security Release Blocker Sprint).** Verified against a real running Worker + real local D1, not just tests: anonymous → 401, authenticated-non-admin → 403, real Platform Administrator (provisioned via direct SQL, signed in through the real Auth.js email magic-link flow) → 200 with real data. See "Authorization model" above and `OPERATIONAL_RUNBOOK.md` |
| Denial of service | Sustained abusive traffic against any endpoint | Rate limiting on all state-changing routes, `/api/auth/*` POST actions on a separate, stricter limiter | Partially mitigated — see "Open: rate limiting is per-isolate" |
| Elevation of privilege | A caller reaches an endpoint or performs an action their role shouldn't allow | `GET /api/leads`/`GET /api/leads/:id`/`PATCH /api/leads/:id`/`GET /api/leads/search` (EAP-2) and `GET /api/assessments/:id` are now gated (`auth/authorize.ts`'s `requireLeadsAccess`/`requireAssessmentAccess`) | **Mitigated for the routes that exist.** No Admin/Customer Portal beyond this codebase's own Dashboard and Lead Intelligence modules exists yet, so there is no broader privileged surface to escalate into. Revisit the moment a new module adds routes; the same gate pattern (`resolveCaller` + a `require*` helper) is meant to extend, not be reinvented per route |
| **Availability/correctness (EAP-2 finding — CORS preflight blocked the real `PATCH` route)** | `PATCH /api/leads/:id` was reachable and correctly authorized server-side, but every real cross-origin browser call to it failed before the request even left the browser | `http/cors.ts`'s `ALLOWED_METHODS` response header (used to answer the browser's own `OPTIONS` preflight) listed `"GET, POST, OPTIONS"` only — `PATCH` was never added when the route was built, so Chromium's own preflight check rejected the real request client-side with `Method PATCH is not allowed by Access-Control-Allow-Methods`. Invisible to every test that calls `handleRequest`/`worker.fetch` directly or drives `fetch` through jsdom — neither implements real preflight enforcement, only a genuine browser does. Found during this feature's own required real-browser Playwright verification, the same way EAP-1's CSP `form-action` finding was found. Fixed: `ALLOWED_METHODS` now includes `PATCH` | **Fixed (EAP-2).** Confirmed via a standalone debug script capturing the real Chromium console error before the fix, and a full lifecycle-update-persists-across-reload Playwright run after (`apps/web/e2e/lead-workspace.spec.ts`) |

### Surface 3 — D1 database

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Tampering (SQL injection) | Malicious input reaching a raw SQL string | Every repository uses parameterized `db.prepare(...).bind(...)` — no string interpolation into SQL anywhere (`grep` confirms; every `*.d1.ts` file follows the same pattern) | Mitigated |
| Information disclosure | A repository leaks another tenant's data | `organization_id`/`assessment_id` foreign keys exist on every relevant table. Tenant isolation is enforced at the application layer, not the query layer: `getAssessment` fetches the single record first, then `requireAssessmentAccess` checks the caller's membership against that record's own `organizationId` before returning it (`router.ts`). `listLeads`'s and (EAP-2) `searchLeads`'s underlying queries are still genuinely unfiltered by organization — that's exactly why both stay restricted to Platform Administrator only rather than delegated to any organization role (see "Authorization model" above) | **Mitigated for `GET /api/assessments/:id`** (per-record application-layer check, verified against real cross-org D1 data — a member of `org_2` got 403 reading an `org_1` assessment). **`GET /api/leads`'s and `GET /api/leads/search`'s underlying queries remain unfiltered by design.** The Admin Portal (EAP-1) and its Lead Workspace (EAP-2) are now a real, existing consumer of these routes, not a hypothetical one — but every caller who reaches them today is a Platform Administrator by definition, so query-layer org filtering has no real caller to serve yet. It becomes real work, not speculative work, the moment an organization-scoped role needs its own filtered view |
| Repudiation | Audit events themselves could be altered/deleted | `AuditRepository` exposes `record`/`list` only — no `update`/`delete` method exists on the interface at all, so nothing in this codebase can modify or remove an audit event once written | Mitigated by omission — an append-only interface, not just an append-only convention |

### Surface 4 — Auth.js authentication

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing (session forgery) | Forged/guessed session token | Database session strategy (not JWT-only) — every session is a real row in `sessions`, validated against D1 on each request via the adapter, not just a signature check | Mitigated |
| Spoofing (credential stuffing / brute force) | Repeated sign-in/magic-link requests | Separate, stricter rate limiter on `/api/auth/*` POST actions (10/min/IP — RC1) | Mitigated (per-isolate caveat applies, see below) |
| Tampering (CSRF on auth actions) | Forged sign-in/callback request | Auth.js's own built-in CSRF handling (double-submit cookie token) — not reimplemented here, verified as already present (`@auth/core`'s own `validateCSRF`, observed directly when a test omitted the token and got a real `MissingCSRF` error, not a silent pass) | Mitigated by the library, confirmed not assumed |
| Information disclosure (cookie theft via XSS) | A cookie is readable by injected JS | `httpOnly: true` on every Auth.js cookie by default (verified against `@auth/core`'s own `cookie.js`) — combined with "no XSS surface found" above, this is defense in depth against a vulnerability class that doesn't currently exist in this codebase | Mitigated |
| Elevation of privilege | Sign-in as another user via a provider mix-up | `@auth/d1-adapter`'s account-linking queries match on `(provider, providerAccountId)`, not email alone — the standard Auth.js account-linking safeguard | Mitigated by the library |
| **Open redirect (cross-origin `callbackUrl`)** | A crafted `callbackUrl` param on `/api/auth/signin`/`/api/auth/signout` redirects the browser to an attacker-controlled origin after sign-in/out | `auth/config.ts`'s custom `redirect` callback (EAP-1) allows only `baseUrl` (the Worker's own origin) or one explicitly configured `allowedOrigin` (the admin SPA, threaded from the same `ALLOWED_ORIGIN` Worker var CORS already uses) — any other origin, or a malformed URL, falls back to `baseUrl` | **Mitigated (EAP-1).** A real, closed allowlist of exactly two origins, not a pattern match or a wildcard — verified against `@auth/core`'s own default callback behavior (`lib/init.js`) to confirm the addition is strictly narrower than "any origin" |
| **Availability (real sign-out silently failing)** | A real caller cannot end their own session because the sign-out confirmation page's redirect back to the app is blocked | Found via real-browser verification, not a security scan: the CSP spec's `form-action` directive restricts not just a form's immediate POST target but also *redirects resulting from* that submission — Auth.js's real sign-out page POSTs to its own origin (satisfying a bare `form-action 'self'`) then 302s to the cross-origin SPA's `callbackUrl`, which Chromium correctly refused to follow under the previous, narrower policy. Fixed: `http/finalizeResponse.ts`'s `authPagesCsp(allowedOrigin)` widens `form-action` to include the same single allowlisted origin as the `redirect` callback above — same trust boundary, enforced in two layers (Auth.js's own redirect logic, and the browser's CSP enforcement of what that redirect is allowed to target) | **Fixed (EAP-1).** Confirmed via an isolated repro before the fix (identical form succeeds with no `callbackUrl`, fails only once one pointing off-origin is added) and a full real-browser sign-in → dashboard → sign-out run after, plus a committed Playwright regression test (`apps/web/e2e/admin-dashboard.spec.ts`) that exercises this exact path |

Not a full ASVS Level 1/2 certification — that requires an external audit, not a self-review — but a real, evidenced pass against the controls that apply today.

## Supply-chain review (ERP-1)

- `package-lock.json` is lockfile v3 (npm 7+) — full integrity hashes per package, deterministic installs.
- `npm audit` (root workspace, all four packages): 0 known vulnerabilities — verified fresh this pass, not carried over.
- Postinstall scripts across the full dependency tree: exactly three (`core-js`, `esbuild`, `workerd`) — all well-known packages using postinstall to fetch platform-specific prebuilt binaries (standard practice for these tools, not an anomaly). No unfamiliar package runs a postinstall script.
- No dependency confusion risk identified — all `@titan/*` packages are private, workspace-local, never published to a public registry.

## Known, accepted gaps (not silently dropped)

- **`GET /api/leads`, `GET /api/leads/search` (EAP-2), `GET /api/organizations`, `GET /api/assessments` (list), `GET /api/assessments/search` (EAP-3), and `GET /api/audit` cannot be delegated to organization-scoped roles, only Platform Administrator.** Not a residual version of the ERP-1 finding (that's fixed — see Surface 2's table above) — a real, separate, smaller gap, now shared by six cross-organization list/search endpoints (five added across EAP-1/EAP-2/EAP-3): none of their backing repositories' `list()`/`search()` methods filter by organization, so there is no safe way today to let an organization's own admin see only their own organization's data. Closing this needs query-layer org filtering in each repository, which needs a real consumer (the Organization/User Management modules `ROADMAP.md` recommends next) to design against, not a speculative addition ahead of one. `GET /api/leads/:id`/`PATCH /api/leads/:id` are a related but distinct case — see "Authorization model" above for why those two are Platform-Administrator-only for a different reason (no lead in this system carries a non-null `organizationId` yet, not just "the query is unfiltered"). `GET /api/leads/search`'s new `assessmentId` filter (EAP-3) inherits this same gap — see "Authorization model" above for the specific consequence on Assessment Details' Lead linkage panel.
- **Lead assignment (EAP-2) has no user directory — a lead can only be assigned to the caller's own `userId` ("assign to me") or unassigned, never to a named other user.** Deliberate, not an oversight: this codebase has no endpoint that lists platform users to assign to, and inventing one, or a free-text assignee field, would either add unscoped cross-tenant user enumeration or an unvalidated identifier with no referential integrity to `users`. `LeadLifecyclePatch.assignedTo` is still validated as a real string/`null` server-side (`validateLeadLifecyclePatch`) — the restriction is in the UI's affordance (`LeadLifecyclePanel.tsx`), not a relaxed server-side check. Not closed by EAP-3 (the Assessment Center has no assignment concept at all) — still needs a real user-directory endpoint, tracked as a later EAP phase (`ROADMAP.md`'s recommended next phase, User Management).
- **In this local dev environment specifically, a single real page view of an assessment or lead's detail page records two `X.viewed` audit events, not one — a real, understood characteristic of this environment, not a defect.** `main.tsx` wraps the app in React 19 `<StrictMode>`, which double-invokes effects in development builds only (by design, to surface non-idempotent effects); each invocation fires a real `GET` request, and the server correctly records a real event for each. It does not occur in a production build (StrictMode's double-invocation is dev-only), and both recorded events are true statements about real requests that occurred — deliberately not "fixed" via event deduplication, which would risk masking a legitimately fast, real second view in a real deployment. Investigated and recorded during EAP-3's real-browser E2E verification (`DECISION_LOG.md`'s EAP-3 entry); the new `assessment-center.spec.ts` asserts on the event's presence, not its exact count, for this reason.
- **Any authenticated user, not just a Platform Administrator, can reach the admin app's `/admin` shell and Dashboard page.** Deliberate, not an oversight: `RequireAuth` gates on "has a session," and the Dashboard's own per-section checks (backed by the real, server-computed `isPlatformAdministrator` flag from `GET /api/me`) are what actually withhold privileged data — a non-admin sees the shell and an honest "Platform Administrator role required" message per gated section, never fabricated data. This is the same shape as any authenticated-but-unprivileged area of an admin tool (you can see there's a locked door; you can't open it) and was a deliberate design choice (`DECISION_LOG.md`'s EAP-1 entry), not a gap to close.
- **No self-service way to become a Platform Administrator.** Deliberate, not an oversight — see "Authorization model" above. Provisioning is a direct D1 insert (`OPERATIONAL_RUNBOOK.md`), which is fine for local dev and will need a real decision (an operator tool? a one-time bootstrap script?) before any deployment.
- **Rate limiting is per-Worker-isolate, not global.** `security/rateLimiter.ts` says this directly in its own doc comment — a real, working limiter for local dev and for limiting abuse from any single isolate, but Cloudflare runs many isolates concurrently with no shared memory between them. A real global limit needs Cloudflare's native Rate Limiting binding or a Durable Object — both require a deployed Cloudflare account, which doesn't exist. Closing this is Stage 5+ scope once deployment is real.
- **No bot protection (Turnstile).** Requires a Cloudflare account. Explicitly deferred this session (`ROADMAP.md`'s deferral record) — rate limiting is the only current defense against automated abuse of the public form.
- **No CSP nonce support for Auth.js's own pages.** Investigated honestly (`http/finalizeResponse.ts`'s comment) — Auth.js's built-in sign-in page doesn't expose a hook for injecting one, so a scoped `style-src 'unsafe-inline'` is used instead of a fabricated nonce claim.
