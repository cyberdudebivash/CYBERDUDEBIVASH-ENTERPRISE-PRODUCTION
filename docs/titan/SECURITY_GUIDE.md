# Security Guide — Project Titan

A threat model and control review for `titan/`, written from the system as it actually exists after EAP-4 (Enterprise Organization Management Platform) — not a generic checklist. Every "mitigated" claim below names the actual code that does it; every "open" item names why it's open and what would close it. See `PLATFORM_FOUNDATION.md` for the broader implementation status and `ARCHITECTURE.md` for the architecture audit this guide's threat model is scoped against.

4. **Auth.js authentication** — database sessions, Email (dev-mode)/Google/GitHub providers. CPP-1's Customer Portal reuses this exact same session mechanism for `/portal` — no new authentication concept, no new provider, no new session table. COM-1 introduces no authentication concept either — every commercial route reuses `resolveCaller` unchanged.
5. **Audit trail** — `audit_events`, written on lead/assessment/organization creation, updates, and lifecycle changes. **EAP-6 adds no new writer** — the Enterprise Audit Center (`GET /api/audit/search`, `GET /api/audit/export`) is a read-only, filtered/paginated/exported view over the exact same table every other module already writes to; it changes what can be queried, never what gets recorded. **EAP-8 adds no writer either** — `GET /api/reports/trends`' "Audit volume"/"Identity activity" entities and the Executive Summary's own audit KPIs are additional real *readers* of `audit_events` (via `AuditRepository.list()`/`.search()`), same as every reader before them. **CPP-1 adds no writer either** — `GET /api/portal/activity` is a real, organization-scoped *reader* of the exact same table (`AuditRepository.search({organizationId, ...})`), not a new notifications system with its own writer. **COM-1 adds real new event types to the existing table, not a new table or a parallel log** — `subscription.created`/`.upgraded`/`.downgraded`/`.canceled`/`.renewed` and `license.activated`/`.expired`, all written through the same `recordAuditEvent` helper every prior module already uses.
6. **Admin Application** (`titan/apps/web`'s `/admin`, EAP-1) — authenticated, browser-facing; session lifecycle (sign-in redirect, sign-out) and a Dashboard reading surfaces 2/3/5 above through the caller's own browser session, not a service credential. Unchanged by CPP-1. **COM-1 adds one new module, Commercial (`/admin/commercial`)** — the same Platform-Administrator-gated, real-session-only shape every prior admin module already has.
7. **Runtime metrics** (`titan/packages/platform`'s `observability/metrics.ts`) — in-memory request/repository-operation counters, recorded since an earlier phase's own instrumentation but never read back by anything until EAP-7's `GET /api/operations/summary` exposed them. CPP-1's own reads are timed under `portal.*` operation labels, same convention, also read by nothing new. 8. **Customer Portal Application** (`titan/apps/web`'s `/portal`, CPP-1) — authenticated, browser-facing; a real, separate sibling shell to the Admin Application (item 6), not nested under it. A real Organization Member/Admin/Owner's Dashboard, Assessments, Reports, Support, and Account, each reading only that caller's own organization's data through their own browser session — never a service credential, never another organization's data. **COM-1 adds one new page, Subscription (`/portal/subscription`)** — the identical organization-scoped, own-session-only shape every prior portal page already has.

**EAP-5: `POST /api/users/:id/profiles` is now a real, working way to grant the Platform Administrator role itself (`organizationId: null, role: "owner"`), not just an organization-scoped role — a deliberate, evidenced capability, not an accidental escalation path.** The route is gated by `requirePlatformAdministrator` like every other route in this table, so only a caller who *already* holds platform-wide authority can reach it at all; granting that same authority to another real, already-authenticated person is the expected act of an admin's own role-management console; it does not let anyone reach a privilege they couldn't already exercise themselves (they could, for instance, already read/write everything the grant recipient could). This closes the "no self-service way to grant Platform Administrator" item named in `FEATURE_MATRIX.md`'s prior "explicitly not yet" list — **for every grant after the first.** Bootstrapping the very first Platform Administrator in a fresh deployment still requires a direct D1 insert (`OPERATIONAL_RUNBOOK.md`), because reaching this route at all requires already holding the role it grants — a real, unavoidable chicken-and-egg constraint, not an oversight.

**EAP-5: `wouldRemoveLastPlatformAdministrator` (router.ts) — a new guard, not present before this phase — rejects (409) any role change or revoke that would leave the system with zero Platform Administrators.** Checked on `PATCH`/`DELETE /api/users/:id/profiles/:profileId` only when the profile being touched is itself a Platform Administrator grant (`organizationId: null, role: "owner"`) and the change would end that status (a demotion away from `"owner"`, or any revoke); every other role change is unaffected. Without this, a Platform Administrator could demote or revoke their own — or the system's only other — platform-wide grant and permanently lock every caller out of every Platform-Administrator-gated route, with no recovery path (provisioning is direct SQL, `OPERATIONAL_RUNBOOK.md`; there is no break-glass account). This is a genuine, evidenced control (`router.test.ts`'s "returns 409 when demoting/revoking the only remaining Platform Administrator" cases), not a documentation-only claim — see "Known, accepted gaps" below for the one real limitation it still has (a check-then-act race between two concurrent requests).

**Why `GET /api/leads`/`GET /api/leads/:id`/`GET /api/leads/search`/`GET /api/organizations`/`GET /api/assessments`/`GET /api/assessments/search`/`GET /api/organizations/search`/`GET /api/users/search`/`GET /api/users/:id`/`GET /api/audit`/`GET /api/audit/search`/`GET /api/audit/export`/`GET /api/reports/summary`/`GET /api/reports/trends`/`GET /api/reports/export` are Platform-Administrator-only, not delegated to organization roles:** none of `leadRepository`/`organizationRepository`/`assessmentRepository`/`auditRepository`/`userRepository`'s `list()`/`search()` methods have a per-organization `WHERE` clause — each returns every organization's rows (or, for `userRepository`, every user regardless of organization membership) in one unfiltered query (confirmed by reading each `*.d1.ts` directly, including EAP-2's `leadRepository.d1.ts`'s `search()`, EAP-3's `assessmentRepository.d1.ts`'s `search()`, EAP-4's `organizationRepository.d1.ts`'s `search()`, EAP-5's `userRepository.d1.ts`'s `search()`, and EAP-6's new `auditRepository.d1.ts`'s `search()`). Gating any of them at "any organization member" would grant a member of Organization A visibility into every other organization's data too — a real cross-tenant leak dressed up as a fix, not a real one. `AuditSearchOptions.organizationId` (EAP-6) is a filter a caller can supply, not a scope a caller is restricted to — a Platform Administrator can pass any organization's id, or none, same as `GET /api/organizations/search`'s own `industry`/`region` filters are optional narrowing, not access control. Until each repository supports organization-scoped filtering (tracked below, "Known, accepted gaps"), only a caller with no single organization to be scoped to — a Platform Administrator — may call any of these routes at all. **EAP-8's three reporting endpoints inherit this exact reasoning, one level removed: `buildExecutiveSummary` composes the same unfiltered `list()`/`search()` reads directly, so a summary/trend/export is cross-tenant by construction even though it returns aggregate counts rather than individual rows.** `GET /api/me` is a deliberate exception, not an oversight: it returns only the caller's own identity/profiles, the same data `GET /api/auth/session` plus a repository lookup already exposes to that same caller, so it carries none of the cross-tenant risk the list endpoints do.

**EAP-6: `GET /api/audit/export` is a real file download, not a JSON envelope, but carries the identical authorization gate and the same `parseAuditSearchOptions` filter parsing `GET /api/audit/search` uses — no separate, weaker policy for the export path.** Two additional, export-specific controls: (1) a hard server-side cap (`AUDIT_EXPORT_MAX_ROWS = 10,000`, `router.ts`) — an export has no caller-supplied page size to bound it by, so this is what keeps one request from attempting to serialize this deployment's entire audit history; (2) CSV output is properly quoted per RFC 4180 (`csvField`, `router.ts`) — a comma/quote/newline in any field is escaped, not just assumed absent. A CSV/formula-injection risk (CWE-1236, a `metadata` value starting with `=`/`+`/`-`/`@` that a spreadsheet application would interpret as a formula) was evaluated and found **not present in this data model**: `metadata` is always emitted through `JSON.stringify` first, so an exported cell always begins with `{` (or is empty) — verified directly (`router.test.ts`'s CSV export tests), not merely assumed. Recorded here as a verified-safe finding, not a silently-skipped one; revisit if a future phase ever adds a raw (non-JSON-wrapped) free-text export column.

**EAP-6: real Playwright/Chromium E2E verification found a genuine gap this phase's own security review would not otherwise have caught — `Access-Control-Expose-Headers` was never set (`http/cors.ts`), so a real cross-origin browser `fetch()` silently hid the `Content-Disposition` header from JavaScript, even though the header was present and correct on the wire (confirmed via router tests and curl-equivalent checks the whole time).** The practical effect: the Audit Export feature worked in every unit/component test (which mock `fetch` and don't enforce this CORS restriction) and would have appeared to work under casual manual testing too (the file still downloaded, just under a generic fallback name), but `apiClient.ts`'s `getBlob` could never actually read the server-chosen filename in a real browser. Fixed by adding `Access-Control-Expose-Headers: Content-Disposition` to `corsHeaders()` (`http/cors.ts`) — the same class of "only real-browser E2E catches this" finding as EAP-2's missing `Access-Control-Allow-Methods` entry for `PATCH`, now with a router-level regression test (`router.test.ts`) asserting the header going forward.

**EAP-7: `GET /api/operations/summary` is Platform-Administrator-only for the identical repository-isolation reason every cross-organization list/search route above already is — it composes a per-service check against `leads`/`assessments`/`organizations`/`users`/`userProfiles`/`audit`, each of which independently carries the same unfiltered, cross-tenant read the paragraph above describes.** It adds no new authorization concept: same `resolveCaller` + `requirePlatformAdministrator` gate, no CSRF check (a pure read, same reasoning as `GET /api/audit`/`GET /api/organizations/search`), no rate limiting beyond what every other authenticated GET already lacks (consistent, not a regression). `GET /health`/`GET /health/ready` remain role-agnostic and unauthenticated, exactly as before this phase — EAP-7 adds no gate to either and reads no new information out of them.

**EAP-7: `ServiceStatus.error` returns a caught exception's own real message text directly in the response body, a deliberate departure from `handleRequest`'s own top-level catch (which logs the real error server-side but returns a generic "Internal server error" to the caller).** Evaluated, not applied by accident: every caller who can reach this endpoint at all is already a Platform Administrator — the same trust boundary that already sees raw `audit_events.metadata_json`, every user's real email, and every organization's real internal id — and the entire value of an operations tool is showing *why* a service is unreachable, not hiding it behind a generic message the way an unauthenticated-facing error boundary correctly does. The narrow, accepted residual risk: a D1 driver error could in principle include a local filesystem path (e.g. `SQLITE_CANTOPEN: unable to open database file: /path/...`) — on a machine this project has never deployed anywhere real, visible only to a caller who already administers that same machine. Every failure is also logged server-side (`ctx.logger.error`, `router.ts`) for the same "must still be logged" reason `recordAuditEvent`'s own audit-write failure handling already established, so this isn't a return-to-caller-only path either. Recorded here as a deliberate, bounded exposure, not a silently-accepted one — revisit if this system is ever deployed somewhere a broader set of callers could hold the Platform Administrator role.

**EAP-7: `RuntimeMetricsPanel`'s request-count rows include each real request's `path` (`url.pathname`, never the query string) — every path in this codebase is a fixed route template with opaque ids (UUIDs), never PII embedded directly in the path itself, and a Platform Administrator already has cross-organization read access to every entity these ids reference (`GET /api/leads`, `GET /api/organizations`, etc.), so this exposes no id a Platform Administrator couldn't already look up directly.** No query-string, request body, or header value is ever recorded into a metric tag anywhere in this codebase (confirmed by reading every `metrics.increment`/`recordDuration` call site) — only `method`/`path`/`status` for requests, and an `operation` label for repository timings.

**EAP-8: `GET /api/reports/export` is a real file download, not a JSON envelope, carrying the identical authorization gate as `GET /api/reports/summary` — no separate, weaker policy for the export path, the same pattern EAP-6 already established for `GET /api/audit/export` relative to `GET /api/audit/search`.** Unlike Audit Export, this export is deliberately more conservative, not less: `ExecutiveSummary` (and therefore its CSV/JSON serialization) contains only aggregate counts and breakdowns — every value is either a known enum key (`leads.status.new`, `assessments.riskLevel.critical`) or an observed free-form key (`assessments.framework.dpdp`, `audit.action.lead.created`) paired with an integer count. Verified directly against `toExecutiveSummaryCsv`'s own row-construction code, not merely reasoned about: no name, email, company, organization slug, user id, or any other individual-record field is read into any row anywhere in that function. Consequently `GET /api/reports/export` needs no row cap the way `AUDIT_EXPORT_MAX_ROWS` (EAP-6) is the real control against one request serializing this deployment's entire audit history — the output size here is bounded by the number of known enum values and observed free-form keys, not by the number of underlying leads/assessments/organizations/audit events, however many of those exist.

**EAP-8: despite its name, "Enterprise Reporting & Analytics" neither reads from nor writes to the pre-existing `reports` table (`migrations/0006`) — worth stating for a security reviewer specifically, since a reader skimming table names could otherwise assume this module is the `reports` table's real consumer and go looking for access control on a write path that doesn't exist.** `reports` tracks generated PDF report artifacts (a `POST`/`storage_key`/download-tracking concept for Roadmap Module H, still blocked on an undecided email/storage pipeline); this module's `ExecutiveSummary`/`TrendSeries` are computed live, per request, from `Lead`/`Assessment`/`Organization`/`Audit`/`User`/`UserProfile`'s own existing `list()`/`search()` reads. No new table means no new access-control surface to review here beyond the three endpoint gates already covered above. Full reasoning: `DECISION_LOG.md`'s EAP-8 entry.

**CPP-1: `resolvePortalOrganizationId` is the entire security model for every `/api/portal/*` route, and it is deliberately the opposite shape of every Platform-Administrator-only list/search endpoint above.** Those endpoints stay cross-organization because their repositories' `list()`/`search()` methods have no per-organization filter (see the paragraph above) — gating them at "any organization member" would leak every other organization's data. The portal routes invert this: `organizationId` is never accepted from the request at all (no query parameter, path segment, or body field named `organizationId` is ever read by any portal handler) — it is looked up once, server-side, from the caller's own real `UserProfileRecord[]`, and passed as a plain function parameter into the exact same `AssessmentRepository.search()`/`OrganizationRepository.findById()`/`AuditRepository.search()` calls the admin routes already use. A caller cannot widen their own scope by supplying a different id, because no code path ever looks at one. Verified directly, not just reasoned about: `router.test.ts`'s "ignores any organizationId the client tries to supply and always resolves the caller's own" case sends `?organizationId=<a real other organization's id>` and asserts the response still reflects the caller's own organization.

**CPP-1: a Platform Administrator with only a platform-wide grant (`organizationId: null`) cannot reach the Customer Portal at all — they see the identical "No organization membership" state any other non-member caller does.** `resolvePortalOrganizationId` looks for a profile with a non-null `organizationId`; a pure Platform Administrator's own profile has none, so `.find` returns nothing and the same 403 (backend) / honest empty-nav state (frontend, `hasPortalOrganizationMembership`) applies. This is a deliberate consequence of the design, not a special case coded around: the Customer Portal answers "what does this organization's own member see," and a Platform Administrator with no organization membership of their own isn't one. A Platform Administrator who *also* holds a real organization profile (nothing prevents this) sees the portal as that organization's own member, same as anyone else.

**CPP-1: `listPortalSupportRequests` scopes to `caller.userId`, narrower than the organization boundary every other portal route uses — a real, deliberate design choice, not an inconsistency.** A member sees only support requests they personally submitted, not their organization's other members' requests, because there is no team-visibility or admin-resolution concept for support requests anywhere in this codebase yet (see `DECISION_LOG.md`'s CPP-1 entry) — exposing a request to any co-worker would be a real, avoidable disclosure with no feature depending on it. Verified directly: two different members of the same seeded organization each see only their own submitted request in `router.test.ts`.

**CPP-1: `GET /api/portal/reports/export`'s CSV/JSON output contains only aggregate counts for the caller's own organization — no individual assessment id, creator id, or answer ever appears in it, the same conservatism EAP-8's own Report Export established.** Verified directly against a real seeded assessment carrying a real `createdBy` value: the exported CSV is asserted not to contain that value anywhere in its body (`router.test.ts`). Needs no row cap the way `GET /api/audit/export`'s `AUDIT_EXPORT_MAX_ROWS` does, for the identical reason EAP-8's own export doesn't — output size is bounded by known enum values and observed framework keys, not by record count.

**COM-1: every commercial route follows the identical `resolvePortalOrganizationId` server-side-scoping mechanism CPP-1 already established — no route anywhere in COM-1 reads an `organizationId` from a query string, path segment, or request body.** Verified directly, not just reasoned about: `grep`ing `router.ts` for `record.organizationId`/`body.organizationId`/`value.organizationId` (the shape a client-trusted id would take) returns zero matches across the entire file, including every COM-1 handler. `createPortalSubscription`/`updatePortalSubscription` receive the resolved organization id as a plain function parameter; `updateAdminSubscription` derives it from the already-fetched subscription record's own `organizationId` (`before.organizationId`), never from the `:id` path segment's caller-supplied value directly. Cross-organization isolation on the customer-facing summary is verified two ways: `router.test.ts` seeds two real organizations with subscriptions on each and asserts a caller's summary only ever reflects their own, and `commercial-platform.spec.ts` proves the same property against a real browser.

**COM-1: a pure Platform Administrator (no separate organization profile of their own) cannot reach `/api/portal/commercial/subscription` at all — the identical `resolvePortalOrganizationId` behavior `SECURITY_GUIDE.md`'s CPP-1 paragraph already documents for every other portal route.** Not a new exception, a consequence of reusing the same mechanism unchanged.

**COM-1: `POST`/`PATCH /api/portal/commercial/subscription` refuse the sales-assisted "enterprise" plan; `PATCH /api/commercial/subscriptions/:id` does not — a deliberate, tested authorization asymmetry between the customer-facing and admin-facing write paths, not an inconsistency.** `isSelfServicePlan(plan)` (`plan.trialDays > 0`) is checked only in the two customer-facing handlers; the admin override validates a plan id is *known* (`findPlan`) but not that it's self-service. Both paths still require their own respective gate (`resolvePortalOrganizationId` + CSRF for the customer path; `requirePlatformAdministrator` + CSRF for the admin path) — the asymmetry is in business-rule validation *after* authorization succeeds, not in the authorization check itself. Verified directly: `router.test.ts` has one case asserting the customer PATCH 400s on `planId: "enterprise"` and a separate case asserting the admin PATCH 200s on the identical payload against the identical subscription.

**COM-1: Feature Entitlements are computed and displayed, not enforced as a gate on any route in this codebase — worth stating plainly for a security reviewer, since a reader skimming "Entitlements" in the UI could otherwise assume a real access-control check backs it.** `resolveEntitlements(plan, subscription)` is a pure function with no caller anywhere in `router.ts` that branches on its result before serving a request — it is read by `getPortalCommercialSummary` purely to compose a response the frontend displays. This is a deliberate, named scope boundary, not an oversight: wiring enforcement into `GET /api/portal/reports/export` (CPP-1) or the Assessment/Reporting Engine would mean modifying systems this program's own instructions name as do-not-redesign. See `DECISION_LOG.md`'s COM-1 entry.

**COM-1: `SubscriptionRecord`/`LicenseRecord` carry no payment amount, currency, invoice id, or card/token field — verified directly against both migrations and both repository interfaces, not merely asserted.** `priceDisplay` (on the code-defined `Plan`, never persisted) is the only price-shaped value anywhere in this module, and it is display text a human wrote, never a value any code path computes, charges, or reconciles against a real transaction. There is consequently no PCI-relevant data anywhere in this system to protect, by construction rather than by a control that could be misconfigured.

**EAP-3/EAP-4/EAP-5: `GET /api/leads/search`'s `assessmentId` (EAP-3) and `organizationId` (EAP-4) filters, and `GET /api/assessments/search`'s `organizationId` (EAP-4) and `createdBy` (EAP-5) filters, each inherit the same Platform-Administrator-only gate as every other call to their own endpoint — none carries its own, narrower policy.** A consequence worth naming explicitly, unchanged since EAP-3: an Organization Member who can read an assessment directly (via `GET /api/assessments/:id`'s per-record delegation) will still see `{status: "forbidden"}` for that same assessment's own Lead linkage panel, the same Organization Member reading their own organization's assessment would similarly see `{status: "forbidden"}` for that organization's own Relationships panel (EAP-4), and — new this phase — a user viewing their own User Details page (if they could reach `/admin/users/:id` at all, which requires Platform Administrator to begin with) would see the same for their own Relationships panel, because all three underlying searches remain cross-organization-unfiltered. The frontend (`useAssessmentDetail.ts`, `useOrganizationDetail.ts`, and now `useUserDetail.ts`) surfaces this honestly as its own independent "forbidden" state, scoped to just that one panel, rather than either hiding the panel or fabricating empty data.

**EAP-5: `UserRepository.search()`/`GET /api/users/search` deliberately has no `organizationId` or `role` filter, for the identical repository-isolation reason `OrganizationSearchOptions` has no `riskLevel` filter (`DECISION_LOG.md`'s EAP-4 entry).** A user's identity (`users` table) and their organization/role grants (`user_profiles` table) are two different repositories with no query-layer dependency on each other anywhere in this codebase; "list users in organization X" would require `UserRepository.search()` to reach into `UserProfileRepository`'s data, the exact cross-repository join this system's architecture doesn't support. The real capability this would back — an Organization Detail page showing its own members — is consequently not built this phase (see `FEATURE_MATRIX.md`'s not-yet-features, `DECISION_LOG.md`'s EAP-5 entry) rather than built via a repository-layer shortcut.

**Why `GET /api/leads/:id`/`PATCH /api/leads/:id` are Platform-Administrator-only rather than delegated like `GET /api/assessments/:id` (EAP-2):** a single lead, unlike a single assessment, doesn't carry a reliable `organizationId` to delegate against — `LeadRecord.organizationId` is nullable, and no code path that creates a lead (`validation/` has no `organizationId` field anywhere in `validateNewLead`) ever sets one, so every lead in this system today has `organizationId: null` (leads are captured anonymously, before any organization relationship exists — `PRODUCT_VISION.md`). "Is the caller a member of this lead's organization" is therefore not an answerable question for any lead yet, unlike assessments. Both routes reuse `requireLeadsAccess` rather than inventing a third policy shape for this one pair of routes.

**Why Organization Administration (`POST /api/organizations`, `PATCH /api/organizations/:id`, EAP-4) is Platform-Administrator-only rather than delegated to the existing `requireOrganizationAccess` helper (`auth/authorize.ts`), which exists specifically for exactly this kind of per-organization gate:** this is the platform's own tenant-management console — creating, archiving, and editing the tenant boundary itself — not a self-service action an organization's own members take on themselves (a Customer Portal, explicitly out of scope for EAP-4). Delegating to org-scoped `owner` roles would let an organization archive or rename itself, a materially different and wider authorization surface than "read your own organization's data" (which nothing in this codebase grants yet either — see the gap below). `requireOrganizationAccess` remains unwired, ready for the day a real self-service surface needs it, same as it has since RC1.

**Why `GET /api/assessments/:id` *is* delegated to organization roles:** unlike the list endpoints, a single assessment fetch is naturally scoped — the record being read has exactly one `organizationId` (or none), so "is the caller a member of *that* organization" is a correct, answerable question per request, unlike an all-organizations list.

**Provisioning the very first Platform Administrator in a fresh deployment** is still a direct D1 insert, not a self-service flow — reaching `POST /api/users/:id/profiles` (EAP-5) at all requires already holding the role it can grant, so there is no route a caller with zero privileges can use to grant themselves one (by design: that would be its own privilege-escalation vulnerability). See `OPERATIONAL_RUNBOOK.md`'s "Provisioning a local Platform Administrator" section — its verification step now includes signing in through the real admin app UI (`/admin`), not only curl. **Every subsequent grant, to a different real user, now goes through a real, audited endpoint** (`POST /api/users/:id/profiles`) rather than requiring a second direct SQL insert — see the EAP-5 paragraph above.

**EAP-1: this authorization boundary is now reached from a real, cross-origin browser session (the admin SPA), not only from same-origin tests/curl.** That required the Worker to actively support crossing origins with credentials — `Access-Control-Allow-Credentials: true` (`http/cors.ts`), an Auth.js `redirect` callback allowlisting the SPA's origin (`auth/config.ts`), and a CSP `form-action` allowlist for the same origin (`http/finalizeResponse.ts`) — each scoped to the one configured `allowedOrigin`, never a wildcard, so this doesn't broaden who can call these endpoints, only where the legitimate caller's own browser is allowed to run.

Out of scope for this document: anything requiring a deployed Cloudflare account (Turnstile, WAF rules, Cloudflare Access) — this project has never had one, in this session or any prior stage. See `ROADMAP.md`'s deferral record for why.

### Surface 3 — D1 database

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Tampering (SQL injection) | Malicious input reaching a raw SQL string | Every repository uses parameterized `db.prepare(...).bind(...)` — no string interpolation into SQL anywhere (`grep` confirms; every `*.d1.ts` file follows the same pattern) | Mitigated |
| Information disclosure | A repository leaks another tenant's data | `organization_id`/`assessment_id` foreign keys exist on every relevant table. | Threat | Scenario | Mitigation | Status |

## Secrets management (PRD-1)

- **Where secrets live today**: locally, `.dev.vars` (gitignored, generated from `.dev.vars.example` — a real, generated `AUTH_SECRET` per developer, never committed). Nowhere else — no secret has ever existed in this repository's git history, in `wrangler.toml` (which holds only non-secret `[vars]`), or in any CI workflow's own YAML.
- **Where secrets would live for a real deployment**: `wrangler secret put <NAME> --env <staging|production>` — Cloudflare's own secret store, which is write-only by design (there is no `wrangler secret get`; `wrangler secret list` returns names only, verified directly against the real installed CLI's `--help` output). GitHub Actions' own repository/environment secrets (`CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`) authenticate `titan-deploy.yml` to set those — see `DEPLOYMENT_GUIDE.md`'s "First real deploy checklist."
- **Validation, not storage**: `packages/platform/scripts/validate-secrets.mjs` checks that required secret *names* are present — `--local` mode reads `.dev.vars` directly (real, run this phase); `--env <tier>` mode shells out to `wrangler secret list` (needs real Cloudflare credentials this project has never had — run this phase, confirmed it fails with exactly the expected `CLOUDFLARE_API_TOKEN` authentication error, not a validation bug). Neither mode ever reads or prints a secret's actual value.
- **Rotation**: `AUTH_SECRET` already supported an array value for rotation windows before PRD-1 (RC1) — `auth/config.ts` accepts multiple valid secrets simultaneously so an in-flight session signed with the old value keeps validating while new sessions adopt the new one. Rotating a real deployed secret is `wrangler secret put AUTH_SECRET --env <tier>` with the new value (no separate "rotate" command exists — a `put` simply replaces what the Worker reads on its next invocation). No automated rotation schedule exists or is claimed to.
- **Environment separation**: `staging` and `production` are separate Cloudflare environments with independently-set secrets (`wrangler secret put ... --env staging` vs. `--env production` are two distinct calls) — a staging `AUTH_SECRET` compromise cannot forge a production session, by construction of Cloudflare's own per-environment secret scoping, not by anything this repository's own code enforces.
- **Never hardcoded**: verified by direct search — no string matching a plausible secret shape appears anywhere in `wrangler.toml`, any `*.yml` workflow file, or any committed source file; `.dev.vars`/`.env.local` remain gitignored (`titan/.gitignore`, unchanged by PRD-1).

## Known, accepted gaps (not silently dropped)

- **`GET /api/leads`, `GET /api/leads/search` (EAP-2), `GET /api/organizations`, `GET /api/assessments` (list), `GET /api/assessments/search` (EAP-3), `GET /api/organizations/search` (EAP-4), `GET /api/users/search` (EAP-5), `GET /api/audit`, `GET /api/audit/search`, `GET /api/audit/export` (EAP-6), `GET /api/operations/summary` (EAP-7), and `GET /api/reports/summary`/`GET /api/reports/trends`/`GET /api/reports/export` (EAP-8) cannot be delegated to organization-scoped roles, only Platform Administrator.** Not a residual version of the ERP-1 finding (that's fixed — see Surface 2's table above) — a real, separate, smaller gap, now shared by fourteen cross-organization list/search/export/summary endpoints (thirteen added across EAP-1/EAP-2/EAP-3/EAP-4/EAP-5/EAP-6/EAP-7/EAP-8): none of their backing repositories' `list()`/`search()` methods filter by organization, so there is no safe way today to let an organization's own admin see only their own organization's data (or, for `GET /api/users/search`, only their own organization's members, or for the audit/reporting endpoints, only their own organization's events/counts). Closing this needs query-layer org filtering in each repository, which needs a real consumer to design against, not a speculative addition ahead of one. `GET /api/leads/:id`/`PATCH /api/leads/:id` are a related but distinct case — see "Authorization model" above for why those two are Platform-Administrator-only for a different reason (no lead in this system carries a non-null `organizationId` yet, not just "the query is unfiltered"). `GET /api/leads/search`'s `assessmentId` filter (EAP-3) and `organizationId` filter (EAP-4), and `GET /api/assessments/search`'s `organizationId` (EAP-4) and `createdBy` (EAP-5) filters, each inherit this same gap — see "Authorization model" above for the specific consequence on Assessment Details' Lead linkage panel, Organization Details' Relationships panel, and User Details' Relationships panel. `AuditSearchOptions.organizationId` (EAP-6) is the identical shape of optional filter, not a scope restriction — see "Authorization model" above. `GET /api/operations/summary` (EAP-7) has no organization filter at all — it was never scoped to one organization's own service health to begin with, since a Platform Administrator is the only caller for whom "every service's overall status" is a meaningful question. `GET /api/reports/summary`/`GET /api/reports/trends`/`GET /api/reports/export` (EAP-8) have no organization filter either, for the identical reason — an Executive Summary or trend series is, by construction, a cross-organization rollup; there is no single-organization variant of "how many leads exist" to scope it down to.
- **`wouldRemoveLastPlatformAdministrator` (EAP-5) has a real, narrow check-then-act race, not closed this phase.** It reads every profile (`UserProfileRepository.list()`), decides, then a separate `update`/`remove` call runs — two concurrent requests each touching a *different* one of exactly two remaining Platform Administrator profiles could each observe "2 remain, safe" before either write commits, both proceed, and leave zero. The same unguarded check-then-act shape already exists elsewhere in this codebase (`createOrganization`'s slug-conflict check, `grantUserProfile`'s own duplicate-membership check) and is accepted there for the identical reason: nothing in this system has ever used a D1 transaction/batch across a read and a later write, this is a local-only system that has never seen concurrent traffic, and introducing transactional locking for this one guard alone — while every other check-then-act path in the same file stays unguarded — would be inconsistent, speculative hardening. Revisit if/when this system is deployed somewhere real concurrent requests are actually expected.
- **EAP-5's User Directory has no way to filter by organization or role (`GET /api/users/search` has no `organizationId`/`role` param) — so Organization Details has no "Members" panel showing who belongs to it.** Deliberate, not an oversight: see "Authorization model" above for the repository-isolation reasoning (identical to why `OrganizationSearchOptions` has no `riskLevel` filter). A user's own User Details page (EAP-5) shows their own memberships; the reciprocal view from the organization's own side is real, recommended near-term follow-up work (`ROADMAP.md`), not built this pass.
- **Organization Administration (`POST /api/organizations`, `PATCH /api/organizations/:id`, EAP-4) cannot be delegated to an organization's own members either — creating, archiving, restoring, or editing an organization is Platform-Administrator-only, full stop.** A distinct gap from the list/search one above, not the same gap restated: this isn't about a repository query lacking a `WHERE` clause, it's a deliberate scope decision (`DECISION_LOG.md`'s EAP-4 entry, "Authorization model" above) — the existing-but-dormant `requireOrganizationAccess` helper (`auth/authorize.ts`) could technically gate this at "the organization's own `owner` role" today, but doing so would be building a Customer Portal's worth of self-service authority, which the EAP-4 brief explicitly ruled out. Revisit only alongside a real Customer Portal decision, not as an incidental extension of Organization Management.
- **`OrganizationSearchOptions` has no `riskLevel` filter — a real field, but on a different repository (`AssessmentRecord`), not `OrganizationRecord` itself.** Not a security gap (nothing here is under-authorized), but worth recording alongside the others: computing it would require one repository to depend on another's data, which no repository in this codebase does today (`ARCHITECTURE.md`'s EAP-4 section has the full architectural reasoning). Organization Health surfaces the same underlying risk data at the single-organization scope instead.

- **A Subscription is 1:1 with an Organization — there is no multi-subscription-per-tenant concept.** `SubscriptionRepository.findByOrganizationId` returns at most one row by construction (nothing in this codebase writes a second). If a real multi-subscription need ever emerges, this is a real, load-bearing assumption throughout the commercial handlers (`getPortalCommercialSummary`, `createPortalSubscription`'s own 409-on-conflict check) that would need revisiting, not just a repository-layer change.
- **The Plan catalog (`commercial/planCatalog.ts`) is code-defined — there is no admin-facing way to add a fourth plan or change an existing one's price/entitlements without a code change and a redeploy.** Deliberate, not an oversight: dynamic plan authoring carries real implications (proration for existing subscribers on a price change) this phase was never scoped to solve, the same reasoning `@titan/assessment-core`'s own question bank has never been admin-editable either.
- **`applySubscriptionPatch`'s renewal always extends `currentPeriodEnd` by a fixed 30 days, server-computed — there is no caller-chosen term (monthly vs. annual), and no proration for a mid-cycle plan change.** A real, minimal first pass: the state transitions (trialing→active→canceled→active) are the real control this phase built; billing-cycle mechanics belong to whichever real payment provider integration eventually plugs into this lifecycle.
- **PRD-1: `validateProductionConfig` checks exactly two variables (`AUTH_SECRET`, `ALLOWED_ORIGIN`) — a deliberately narrow, real check, not a general-purpose configuration schema.** These are the only two this codebase has ever required for a real deployment to function correctly (`.dev.vars.example`'s own comments). If a future phase adds a genuinely required production variable (a new provider credential, a new binding), it needs its own check added here — this function does not generically validate "every var wrangler.toml could theoretically need," and was never claimed to.
- **PRD-1: nothing in this repository scopes or audits the real `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID` GitHub Actions secrets `titan-deploy.yml` would use.** `DEPLOYMENT_GUIDE.md` recommends a least-privilege token (Workers/Pages/D1 edit only, not full account access) as part of the "First real deploy checklist," but enforcing that scope is a real action taken in Cloudflare's own dashboard when the token is created — no code or config in this repository can verify a token's actual permission scope from the outside. A real, human diligence step, named directly rather than implied to be automatically enforced.
- **PRD-1: `titan-deploy.yml`'s `environment: ${{ inputs.target_environment }}` references GitHub's own Environment-protection-rules feature (required reviewers, wait timers) but does not and cannot configure it from this repository.** GitHub Environment protection rules are a repository *setting* (Settings → Environments), not a file this codebase can commit — until a repository admin configures them, `staging`/`production` triggers run with no additional approval gate beyond whoever has permission to run `workflow_dispatch` at all. Named directly in `DEPLOYMENT_GUIDE.md`'s checklist as a required, external, one-time action, not assumed to already be in place.
