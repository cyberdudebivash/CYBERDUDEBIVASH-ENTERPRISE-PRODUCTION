# Security Guide тАФ Project Titan

A threat model and control review for `titan/`, written from the system as it actually exists after EAP-4 (Enterprise Organization Management Platform) тАФ not a generic checklist. Every "mitigated" claim below names the actual code that does it; every "open" item names why it's open and what would close it. See `PLATFORM_FOUNDATION.md` for the broader implementation status and `ARCHITECTURE.md` for the architecture audit this guide's threat model is scoped against.

## System surfaces in scope

1. **Public DPDP assessment flow** (`titan/apps/web`'s `/assessment/dpdp`) тАФ unauthenticated, browser-facing.
2. **Worker API** (`titan/packages/platform`) тАФ `GET /health`, `POST`/`GET /api/leads`, `GET /api/leads/:id` (EAP-2), `PATCH /api/leads/:id` (EAP-2), `GET /api/leads/search` (EAP-2, gained an `assessmentId` filter in EAP-3 and an `organizationId` filter in EAP-4), `POST /api/assessments`, `GET /api/assessments/:id` (EAP-3 adds an `assessment.viewed` audit event), `GET /api/assessments` (list, EAP-1), `GET /api/assessments/search` (EAP-3, gained an `organizationId` filter in EAP-4 and a `createdBy` filter in EAP-5), `GET /api/organizations` (EAP-1), `POST /api/organizations` (EAP-4), `GET /api/organizations/search` (EAP-4), `GET /api/organizations/:id` (EAP-4, records an `organization.viewed` audit event), `PATCH /api/organizations/:id` (EAP-4), `GET /api/users/search` (EAP-5), `GET /api/users/:id` (EAP-5, records a `user.viewed` audit event), `POST /api/users/:id/profiles` (EAP-5), `PATCH /api/users/:id/profiles/:profileId` (EAP-5), `DELETE /api/users/:id/profiles/:profileId` (EAP-5), `GET /api/audit` (EAP-1; EAP-2 adds optional `entityType`/`entityId` query filters), `GET /api/audit/search` (EAP-6, the Audit Center's filtered/paginated/sorted search), `GET /api/audit/export` (EAP-6, CSV/JSON file download of the same filtered set), `GET /api/operations/summary` (EAP-7, the Operations Center's one aggregating read), `GET /health/ready`, `GET /api/me` (EAP-1), `/api/auth/*`. `GET /api/leads`, `GET /api/leads/:id`, `PATCH /api/leads/:id`, `GET /api/leads/search`, `GET /api/assessments/:id`, `GET /api/assessments/search`, (EAP-1) `GET /api/assessments`/`GET /api/organizations`/`GET /api/audit`, (EAP-4) `POST /api/organizations`/`GET /api/organizations/search`/`GET /api/organizations/:id`/`PATCH /api/organizations/:id`, (EAP-5) every `/api/users*` route, (EAP-6) `GET /api/audit/search`/`GET /api/audit/export`, and (EAP-7) `GET /api/operations/summary` are authorization-gated тАФ see "Authorization model" below. **The Assessment Center (EAP-3) is read-only: no `PATCH`/`DELETE` endpoint exists for assessments** тАФ an assessment is one immutable row per completed run, with no lifecycle to mutate (`DECISION_LOG.md`'s EAP-3 entry). **Organization Management (EAP-4) is the first module past Leads with a real write surface past creation** тАФ `PATCH /api/organizations/:id` supports metadata edits, archive/restore, and notes, but still no `DELETE`: an organization is archived and restorable, never deleted, matching every repository in this codebase (none has a `delete` method) тАФ except one: **`DELETE /api/users/:id/profiles/:profileId` (EAP-5) is this system's first real deletion**, since a role/membership grant (unlike every other entity here) has no dependent data of its own to preserve (`DECISION_LOG.md`'s EAP-5 entry, `UserProfileRepository.remove`'s doc comment).
3. **D1 database** тАФ organizations, users, sessions, user_profiles, assessments, leads, reports, audit_events. **EAP-5 adds no new table** тАФ the Enterprise User Directory reads Auth.js's own pre-existing `users` table (read-only; `@auth/d1-adapter` remains its sole writer) and extends `user_profiles` (already existed since RC1/Workstream 6) with `update`/`remove` operations, not a schema change. **EAP-7 adds no table either** тАФ the Operations Center reads only what every repository's own `search()`/`list()` already returns.
4. **Auth.js authentication** тАФ database sessions, Email (dev-mode)/Google/GitHub providers. Now reached cross-origin, by a real browser session, from the admin SPA (`titan/apps/web`'s `/admin`, EAP-1) тАФ not just via curl/tests, as it was through ERP-1.
5. **Audit trail** тАФ `audit_events`, written on lead/assessment/organization creation, updates, and lifecycle changes. **EAP-6 adds no new writer** тАФ the Enterprise Audit Center (`GET /api/audit/search`, `GET /api/audit/export`) is a read-only, filtered/paginated/exported view over the exact same table every other module already writes to; it changes what can be queried, never what gets recorded.
6. **Admin Application** (`titan/apps/web`'s `/admin`, EAP-1) тАФ authenticated, browser-facing; session lifecycle (sign-in redirect, sign-out) and a Dashboard reading surfaces 2/3/5 above through the caller's own browser session, not a service credential.
7. **Runtime metrics** (`titan/packages/platform`'s `observability/metrics.ts`) тАФ in-memory request/repository-operation counters, recorded since an earlier phase's own instrumentation but never read back by anything until EAP-7's `GET /api/operations/summary` exposed them.

## Authorization model (Security Release Blocker Sprint)

Roles, from lowest to highest privilege:

| Role | Definition | Grants |
|---|---|---|
| Anonymous | No session (`GET /api/auth/session` returns no user, or no session cookie at all) | `POST /api/leads`, `POST /api/assessments` (the public assessment flow тАФ deliberately stays anonymous, `PRODUCT_VISION.md`), `GET /health`, `GET /health/ready` |
| Authenticated User | A real session, but no `user_profiles` row anywhere | `GET /api/me` (their own identity тАФ EAP-1). **EAP-5: `POST /api/users/:id/profiles` is now a real, working way to grant the Platform Administrator role itself (`organizationId: null, role: "owner"`), not just an organization-scoped role тАФ a deliberate, evidenced capability, not an accidental escalation path.** The route is gated by `requirePlatformAdministrator` like every other route in this table, so only a caller who *already* holds platform-wide authority can reach it at all; granting that same authority to another real, already-authenticated person is the expected act of an admin's own role-management console; it does not let anyone reach a privilege they couldn't already exercise themselves (they could, for instance, already read/write everything the grant recipient could). This closes the "no self-service way to grant Platform Administrator" item named in `FEATURE_MATRIX.md`'s prior "explicitly not yet" list тАФ **for every grant after the first.** Bootstrapping the very first Platform Administrator in a fresh deployment still requires a direct D1 insert (`OPERATIONAL_RUNBOOK.md`), because reaching this route at all requires already holding the role it grants тАФ a real, unavoidable chicken-and-egg constraint, not an oversight.

**EAP-5: `wouldRemoveLastPlatformAdministrator` (router.ts) тАФ a new guard, not present before this phase тАФ rejects (409) any role change or revoke that would leave the system with zero Platform Administrators.** Checked on `PATCH`/`DELETE /api/users/:id/profiles/:profileId` only when the profile being touched is itself a Platform Administrator grant (`organizationId: null, role: "owner"`) and the change would end that status (a demotion away from `"owner"`, or any revoke); every other role change is unaffected. Without this, a Platform Administrator could demote or revoke their own тАФ or the system's only other тАФ platform-wide grant and permanently lock every caller out of every Platform-Administrator-gated route, with no recovery path (provisioning is direct SQL, `OPERATIONAL_RUNBOOK.md`; there is no break-glass account). This is a genuine, evidenced control (`router.test.ts`'s "returns 409 when demoting/revoking the only remaining Platform Administrator" cases), not a documentation-only claim тАФ see "Known, accepted gaps" below for the one real limitation it still has (a check-then-act race between two concurrent requests).

**Why `GET /api/leads`/`GET /api/leads/:id`/`GET /api/leads/search`/`GET /api/organizations`/`GET /api/assessments`/`GET /api/assessments/search`/`GET /api/organizations/search`/`GET /api/users/search`/`GET /api/users/:id`/`GET /api/audit`/`GET /api/audit/search`/`GET /api/audit/export` are Platform-Administrator-only, not delegated to organization roles:** none of `leadRepository`/`organizationRepository`/`assessmentRepository`/`auditRepository`/`userRepository`'s `list()`/`search()` methods have a per-organization `WHERE` clause тАФ each returns every organization's rows (or, for `userRepository`, every user regardless of organization membership) in one unfiltered query (confirmed by reading each `*.d1.ts` directly, including EAP-2's `leadRepository.d1.ts`'s `search()`, EAP-3's `assessmentRepository.d1.ts`'s `search()`, EAP-4's `organizationRepository.d1.ts`'s `search()`, EAP-5's `userRepository.d1.ts`'s `search()`, and EAP-6's new `auditRepository.d1.ts`'s `search()`). Gating any of them at "any organization member" would grant a member of Organization A visibility into every other organization's data too тАФ a real cross-tenant leak dressed up as a fix, not a real one. `AuditSearchOptions.organizationId` (EAP-6) is a filter a caller can supply, not a scope a caller is restricted to тАФ a Platform Administrator can pass any organization's id, or none, same as `GET /api/organizations/search`'s own `industry`/`region` filters are optional narrowing, not access control. Until each repository supports organization-scoped filtering (tracked below, "Known, accepted gaps"), only a caller with no single organization to be scoped to тАФ a Platform Administrator тАФ may call any of these routes at all. `GET /api/me` is a deliberate exception, not an oversight: it returns only the caller's own identity/profiles, the same data `GET /api/auth/session` plus a repository lookup already exposes to that same caller, so it carries none of the cross-tenant risk the list endpoints do.

**EAP-6: `GET /api/audit/export` is a real file download, not a JSON envelope, but carries the identical authorization gate and the same `parseAuditSearchOptions` filter parsing `GET /api/audit/search` uses тАФ no separate, weaker policy for the export path.** Two additional, export-specific controls: (1) a hard server-side cap (`AUDIT_EXPORT_MAX_ROWS = 10,000`, `router.ts`) тАФ an export has no caller-supplied page size to bound it by, so this is what keeps one request from attempting to serialize this deployment's entire audit history; (2) CSV output is properly quoted per RFC 4180 (`csvField`, `router.ts`) тАФ a comma/quote/newline in any field is escaped, not just assumed absent. A CSV/formula-injection risk (CWE-1236, a `metadata` value starting with `=`/`+`/`-`/`@` that a spreadsheet application would interpret as a formula) was evaluated and found **not present in this data model**: `metadata` is always emitted through `JSON.stringify` first, so an exported cell always begins with `{` (or is empty) тАФ verified directly (`router.test.ts`'s CSV export tests), not merely assumed. Recorded here as a verified-safe finding, not a silently-skipped one; revisit if a future phase ever adds a raw (non-JSON-wrapped) free-text export column.

**EAP-6: real Playwright/Chromium E2E verification found a genuine gap this phase's own security review would not otherwise have caught тАФ `Access-Control-Expose-Headers` was never set (`http/cors.ts`), so a real cross-origin browser `fetch()` silently hid the `Content-Disposition` header from JavaScript, even though the header was present and correct on the wire (confirmed via router tests and curl-equivalent checks the whole time).** The practical effect: the Audit Export feature worked in every unit/component test (which mock `fetch` and don't enforce this CORS restriction) and would have appeared to work under casual manual testing too (the file still downloaded, just under a generic fallback name), but `apiClient.ts`'s `getBlob` could never actually read the server-chosen filename in a real browser. Fixed by adding `Access-Control-Expose-Headers: Content-Disposition` to `corsHeaders()` (`http/cors.ts`) тАФ the same class of "only real-browser E2E catches this" finding as EAP-2's missing `Access-Control-Allow-Methods` entry for `PATCH`, now with a router-level regression test (`router.test.ts`) asserting the header going forward.

**EAP-7: `GET /api/operations/summary` is Platform-Administrator-only for the identical repository-isolation reason every cross-organization list/search route above already is тАФ it composes a per-service check against `leads`/`assessments`/`organizations`/`users`/`userProfiles`/`audit`, each of which independently carries the same unfiltered, cross-tenant read the paragraph above describes.** It adds no new authorization concept: same `resolveCaller` + `requirePlatformAdministrator` gate, no CSRF check (a pure read, same reasoning as `GET /api/audit`/`GET /api/organizations/search`), no rate limiting beyond what every other authenticated GET already lacks (consistent, not a regression). `GET /health`/`GET /health/ready` remain role-agnostic and unauthenticated, exactly as before this phase тАФ EAP-7 adds no gate to either and reads no new information out of them.

**EAP-7: `ServiceStatus.error` returns a caught exception's own real message text directly in the response body, a deliberate departure from `handleRequest`'s own top-level catch (which logs the real error server-side but returns a generic "Internal server error" to the caller).** Evaluated, not applied by accident: every caller who can reach this endpoint at all is already a Platform Administrator тАФ the same trust boundary that already sees raw `audit_events.metadata_json`, every user's real email, and every organization's real internal id тАФ and the entire value of an operations tool is showing *why* a service is unreachable, not hiding it behind a generic message the way an unauthenticated-facing error boundary correctly does. The narrow, accepted residual risk: a D1 driver error could in principle include a local filesystem path (e.g. `SQLITE_CANTOPEN: unable to open database file: /path/...`) тАФ on a machine this project has never deployed anywhere real, visible only to a caller who already administers that same machine. Every failure is also logged server-side (`ctx.logger.error`, `router.ts`) for the same "must still be logged" reason `recordAuditEvent`'s own audit-write failure handling already established, so this isn't a return-to-caller-only path either. Recorded here as a deliberate, bounded exposure, not a silently-accepted one тАФ revisit if this system is ever deployed somewhere a broader set of callers could hold the Platform Administrator role.

**EAP-7: `RuntimeMetricsPanel`'s request-count rows include each real request's `path` (`url.pathname`, never the query string) тАФ every path in this codebase is a fixed route template with opaque ids (UUIDs), never PII embedded directly in the path itself, and a Platform Administrator already has cross-organization read access to every entity these ids reference (`GET /api/leads`, `GET /api/organizations`, etc.), so this exposes no id a Platform Administrator couldn't already look up directly.** No query-string, request body, or header value is ever recorded into a metric tag anywhere in this codebase (confirmed by reading every `metrics.increment`/`recordDuration` call site) тАФ only `method`/`path`/`status` for requests, and an `operation` label for repository timings.

**EAP-3/EAP-4/EAP-5: `GET /api/leads/search`'s `assessmentId` (EAP-3) and `organizationId` (EAP-4) filters, and `GET /api/assessments/search`'s `organizationId` (EAP-4) and `createdBy` (EAP-5) filters, each inherit the same Platform-Administrator-only gate as every other call to their own endpoint тАФ none carries its own, narrower policy.** A consequence worth naming explicitly, unchanged since EAP-3: an Organization Member who can read an assessment directly (via `GET /api/assessments/:id`'s per-record delegation) will still see `{status: "forbidden"}` for that same assessment's own Lead linkage panel, the same Organization Member reading their own organization's assessment would similarly see `{status: "forbidden"}` for that organization's own Relationships panel (EAP-4), and тАФ new this phase тАФ a user viewing their own User Details page (if they could reach `/admin/users/:id` at all, which requires Platform Administrator to begin with) would see the same for their own Relationships panel, because all three underlying searches remain cross-organization-unfiltered. The frontend (`useAssessmentDetail.ts`, `useOrganizationDetail.ts`, and now `useUserDetail.ts`) surfaces this honestly as its own independent "forbidden" state, scoped to just that one panel, rather than either hiding the panel or fabricating empty data.

**EAP-5: `UserRepository.search()`/`GET /api/users/search` deliberately has no `organizationId` or `role` filter, for the identical repository-isolation reason `OrganizationSearchOptions` has no `riskLevel` filter (`DECISION_LOG.md`'s EAP-4 entry).** A user's identity (`users` table) and their organization/role grants (`user_profiles` table) are two different repositories with no query-layer dependency on each other anywhere in this codebase; "list users in organization X" would require `UserRepository.search()` to reach into `UserProfileRepository`'s data, the exact cross-repository join this system's architecture doesn't support. The real capability this would back тАФ an Organization Detail page showing its own members тАФ is consequently not built this phase (see `FEATURE_MATRIX.md`'s not-yet-features, `DECISION_LOG.md`'s EAP-5 entry) rather than built via a repository-layer shortcut.

**Why `GET /api/leads/:id`/`PATCH /api/leads/:id` are Platform-Administrator-only rather than delegated like `GET /api/assessments/:id` (EAP-2):** a single lead, unlike a single assessment, doesn't carry a reliable `organizationId` to delegate against тАФ `LeadRecord.organizationId` is nullable, and no code path that creates a lead (`validation/` has no `organizationId` field anywhere in `validateNewLead`) ever sets one, so every lead in this system today has `organizationId: null` (leads are captured anonymously, before any organization relationship exists тАФ `PRODUCT_VISION.md`). "Is the caller a member of this lead's organization" is therefore not an answerable question for any lead yet, unlike assessments. Both routes reuse `requireLeadsAccess` rather than inventing a third policy shape for this one pair of routes.

**Why Organization Administration (`POST /api/organizations`, `PATCH /api/organizations/:id`, EAP-4) is Platform-Administrator-only rather than delegated to the existing `requireOrganizationAccess` helper (`auth/authorize.ts`), which exists specifically for exactly this kind of per-organization gate:** this is the platform's own tenant-management console тАФ creating, archiving, and editing the tenant boundary itself тАФ not a self-service action an organization's own members take on themselves (a Customer Portal, explicitly out of scope for EAP-4). Delegating to org-scoped `owner` roles would let an organization archive or rename itself, a materially different and wider authorization surface than "read your own organization's data" (which nothing in this codebase grants yet either тАФ see the gap below). `requireOrganizationAccess` remains unwired, ready for the day a real self-service surface needs it, same as it has since RC1.

**Why `GET /api/assessments/:id` *is* delegated to organization roles:** unlike the list endpoints, a single assessment fetch is naturally scoped тАФ the record being read has exactly one `organizationId` (or none), so "is the caller a member of *that* organization" is a correct, answerable question per request, unlike an all-organizations list.

**Provisioning the very first Platform Administrator in a fresh deployment** is still a direct D1 insert, not a self-service flow тАФ reaching `POST /api/users/:id/profiles` (EAP-5) at all requires already holding the role it can grant, so there is no route a caller with zero privileges can use to grant themselves one (by design: that would be its own privilege-escalation vulnerability). See `OPERATIONAL_RUNBOOK.md`'s "Provisioning a local Platform Administrator" section тАФ its verification step now includes signing in through the real admin app UI (`/admin`), not only curl. **Every subsequent grant, to a different real user, now goes through a real, audited endpoint** (`POST /api/users/:id/profiles`) rather than requiring a second direct SQL insert тАФ see the EAP-5 paragraph above.

**EAP-1: this authorization boundary is now reached from a real, cross-origin browser session (the admin SPA), not only from same-origin tests/curl.** That required the Worker to actively support crossing origins with credentials тАФ `Access-Control-Allow-Credentials: true` (`http/cors.ts`), an Auth.js `redirect` callback allowlisting the SPA's origin (`auth/config.ts`), and a CSP `form-action` allowlist for the same origin (`http/finalizeResponse.ts`) тАФ each scoped to the one configured `allowedOrigin`, never a wildcard, so this doesn't broaden who can call these endpoints, only where the legitimate caller's own browser is allowed to run.

Out of scope for this document: anything requiring a deployed Cloudflare account (Turnstile, WAF rules, Cloudflare Access) тАФ this project has never had one, in this session or any prior stage. See `ROADMAP.md`'s deferral record for why.

## Threat model (STRIDE)

### Surface 1 тАФ Public assessment/lead submission

| Threat | Scenario | Mitigation | Status |
|---|---|---|---|
| Spoofing | Automated bot submits fake leads at volume | Per-IP rate limiting (`security/rateLimiter.ts`, 30/min on `POST /api/leads`) | Mitigated, but weak тАФ see "Open: bot protection" below |
| Tampering (CSRF) | A malicious site's page issues a forged cross-origin `POST /api/leads` on a visitor's behalf | Origin-header validation (`security/csrf.ts`) тАФ a present-but-mismatched `Origin` is rejected with 403 | **Mitigated** (RC1) |
| Tampering (data integrity) | A client submits a `result` (risk score) that doesn't match the `answers` it also submitted тАФ e.g. reporting LOW risk while every answer indicates otherwise | `router.ts`'s `createLead`/`createAssessment` recompute `result` server-side via `@titan/assessment-core`'s `scoreAssessment`, discarding whatever the client sent in the request body. `createAssessment` first resolves `framework`/`frameworkVersion` against a small registry (`resolveQuestionBank`) and rejects an unrecognized pair with 400 `unsupported_framework` rather than trusting it | **Mitigated (Security Release Blocker Sprint).** Verified against a real running Worker, not just tests: a `POST /api/leads` submitting `answers` for two failed critical DPDP controls alongside a client-claimed `score: 0`/`riskLevel: "low"` persisted the real computed result (`score: 17`, two real gaps with their actual penalties) тАФ the tampered value never reached D1 |
| Information disclosure | Lead form fields, error messages | React's default escaping (no `dangerouslySetInnerHTML` anywhere тАФ confirmed by `grep`, `ARCHITECTURE.md`); error responses carry a `code`/`message`, never a stack trace or internal detail (`http/responses.ts`) | Mitigated |
| Denial of service | Volumetric abuse of the public form | Per-IP rate limiting тАФ real, but per-Worker-isolate only (see "Open" below) | Partially mitigated |

### Surface 2 тАФ Worker API

## Known, accepted gaps (not silently dropped)

- **`GET /api/leads`, `GET /api/leads/search` (EAP-2), `GET /api/organizations`, `GET /api/assessments` (list), `GET /api/assessments/search` (EAP-3), `GET /api/organizations/search` (EAP-4), `GET /api/users/search` (EAP-5), `GET /api/audit`, `GET /api/audit/search`, `GET /api/audit/export` (EAP-6), and `GET /api/operations/summary` (EAP-7) cannot be delegated to organization-scoped roles, only Platform Administrator.** Not a residual version of the ERP-1 finding (that's fixed тАФ see Surface 2's table above) тАФ a real, separate, smaller gap, now shared by eleven cross-organization list/search/export/summary endpoints (ten added across EAP-1/EAP-2/EAP-3/EAP-4/EAP-5/EAP-6/EAP-7): none of their backing repositories' `list()`/`search()` methods filter by organization, so there is no safe way today to let an organization's own admin see only their own organization's data (or, for `GET /api/users/search`, only their own organization's members, or for the audit endpoints, only their own organization's events). Closing this needs query-layer org filtering in each repository, which needs a real consumer to design against, not a speculative addition ahead of one. `GET /api/leads/:id`/`PATCH /api/leads/:id` are a related but distinct case тАФ see "Authorization model" above for why those two are Platform-Administrator-only for a different reason (no lead in this system carries a non-null `organizationId` yet, not just "the query is unfiltered"). `GET /api/leads/search`'s `assessmentId` filter (EAP-3) and `organizationId` filter (EAP-4), and `GET /api/assessments/search`'s `organizationId` (EAP-4) and `createdBy` (EAP-5) filters, each inherit this same gap тАФ see "Authorization model" above for the specific consequence on Assessment Details' Lead linkage panel, Organization Details' Relationships panel, and User Details' Relationships panel. `AuditSearchOptions.organizationId` (EAP-6) is the identical shape of optional filter, not a scope restriction тАФ see "Authorization model" above. `GET /api/operations/summary` (EAP-7) has no organization filter at all тАФ it was never scoped to one organization's own service health to begin with, since a Platform Administrator is the only caller for whom "every service's overall status" is a meaningful question.
- **`wouldRemoveLastPlatformAdministrator` (EAP-5) has a real, narrow check-then-act race, not closed this phase.** It reads every profile (`UserProfileRepository.list()`), decides, then a separate `update`/`remove` call runs тАФ two concurrent requests each touching a *different* one of exactly two remaining Platform Administrator profiles could each observe "2 remain, safe" before either write commits, both proceed, and leave zero. The same unguarded check-then-act shape already exists elsewhere in this codebase (`createOrganization`'s slug-conflict check, `grantUserProfile`'s own duplicate-membership check) and is accepted there for the identical reason: nothing in this system has ever used a D1 transaction/batch across a read and a later write, this is a local-only system that has never seen concurrent traffic, and introducing transactional locking for this one guard alone тАФ while every other check-then-act path in the same file stays unguarded тАФ would be inconsistent, speculative hardening. Revisit if/when this system is deployed somewhere real concurrent requests are actually expected.
- **EAP-5's User Directory has no way to filter by organization or role (`GET /api/users/search` has no `organizationId`/`role` param) тАФ so Organization Details has no "Members" panel showing who belongs to it.** Deliberate, not an oversight: see "Authorization model" above for the repository-isolation reasoning (identical to why `OrganizationSearchOptions` has no `riskLevel` filter). A user's own User Details page (EAP-5) shows their own memberships; the reciprocal view from the organization's own side is real, recommended near-term follow-up work (`ROADMAP.md`), not built this pass.
- **Organization Administration (`POST /api/organizations`, `PATCH /api/organizations/:id`, EAP-4) cannot be delegated to an organization's own members either тАФ creating, archiving, restoring, or editing an organization is Platform-Administrator-only, full stop.** A distinct gap from the list/search one above, not the same gap restated: this isn't about a repository query lacking a `WHERE` clause, it's a deliberate scope decision (`DECISION_LOG.md`'s EAP-4 entry, "Authorization model" above) тАФ the existing-but-dormant `requireOrganizationAccess` helper (`auth/authorize.ts`) could technically gate this at "the organization's own `owner` role" today, but doing so would be building a Customer Portal's worth of self-service authority, which the EAP-4 brief explicitly ruled out. Revisit only alongside a real Customer Portal decision, not as an incidental extension of Organization Management.
- **`OrganizationSearchOptions` has no `riskLevel` filter тАФ a real field, but on a different repository (`AssessmentRecord`), not `OrganizationRecord` itself.** Not a security gap (nothing here is under-authorized), but worth recording alongside the others: computing it would require one repository to depend on another's data, which no repository in this codebase does today (`ARCHITECTURE.md`'s EAP-4 section has the full architectural reasoning). Organization Health surfaces the same underlying risk data at the single-organization scope instead.
- **Lead assignment (EAP-2) still only supports "assign to me" or unassigned in its own UI тАФ not a picker over the real user directory EAP-5 now provides.** The *reason* this restriction existed (`DECISION_LOG.md`'s EAP-2 entry: "no user directory... would either add unscoped cross-tenant user enumeration or an unvalidated identifier with no referential integrity to `users`") is now genuinely closed тАФ `GET /api/users/search` (EAP-5) is exactly the real, referentially-valid, Platform-Administrator-gated directory that was missing. `LeadLifecyclePatch.assignedTo` was, and remains, validated as a real string/`null` server-side (`validateLeadLifecyclePatch`) regardless. Wiring a real assignee picker into `LeadLifecyclePanel.tsx` is real, newly-unblocked, recommended near-term follow-up (`ROADMAP.md`) тАФ not built this pass, to keep EAP-5's own scope to the User Directory/Role Assignment module itself rather than also reopening a previously-shipped EAP-2 component.
- **In this local dev environment specifically, a single real page view of an assessment or lead's detail page records two `X.viewed` audit events, not one тАФ a real, understood characteristic of this environment, not a defect.** `main.tsx` wraps the app in React 19 `<StrictMode>`, which double-invokes effects in development builds only (by design, to surface non-idempotent effects); each invocation fires a real `GET` request, and the server correctly records a real event for each. It does not occur in a production build (StrictMode's double-invocation is dev-only), and both recorded events are true statements about real requests that occurred тАФ deliberately not "fixed" via event deduplication, which would risk masking a legitimately fast, real second view in a real deployment. Investigated and recorded during EAP-3's real-browser E2E verification (`DECISION_LOG.md`'s EAP-3 entry); the new `assessment-center.spec.ts` asserts on the event's presence, not its exact count, for this reason.
- **Any authenticated user, not just a Platform Administrator, can reach the admin app's `/admin` shell and Dashboard page.** Deliberate, not an oversight: `RequireAuth` gates on "has a session," and the Dashboard's own per-section checks (backed by the real, server-computed `isPlatformAdministrator` flag from `GET /api/me`) are what actually withhold privileged data тАФ a non-admin sees the shell and an honest "Platform Administrator role required" message per gated section, never fabricated data. This is the same shape as any authenticated-but-unprivileged area of an admin tool (you can see there's a locked door; you can't open it) and was a deliberate design choice (`DECISION_LOG.md`'s EAP-1 entry), not a gap to close.
- **No self-service way to become the *first* Platform Administrator in a fresh deployment.** Deliberate, not an oversight тАФ see "Authorization model" above; reaching the real grant endpoint (EAP-5's `POST /api/users/:id/profiles`) requires already holding the role, an unavoidable bootstrap constraint. The first grant is still a direct D1 insert (`OPERATIONAL_RUNBOOK.md`), which is fine for local dev and will need a real decision (an operator tool? a one-time bootstrap script?) before any deployment. **Every subsequent grant is now a real, audited endpoint call, not a second SQL insert** (EAP-5) тАФ this bullet is narrower than it was before this phase.
- **Rate limiting is per-Worker-isolate, not global.** `security/rateLimiter.ts` says this directly in its own doc comment тАФ a real, working limiter for local dev and for limiting abuse from any single isolate, but Cloudflare runs many isolates concurrently with no shared memory between them. A real global limit needs Cloudflare's native Rate Limiting binding or a Durable Object тАФ both require a deployed Cloudflare account, which doesn't exist. Closing this is Stage 5+ scope once deployment is real.
- **No bot protection (Turnstile).** Requires a Cloudflare account. Explicitly deferred this session (`ROADMAP.md`'s deferral record) тАФ rate limiting is the only current defense against automated abuse of the public form.
- **No CSP nonce support for Auth.js's own pages.** Investigated honestly (`http/finalizeResponse.ts`'s comment) тАФ Auth.js's built-in sign-in page doesn't expose a hook for injecting one, so a scoped `style-src 'unsafe-inline'` is used instead of a fabricated nonce claim.
- **`GET /api/audit/export` silently returns only the first 10,000 matching rows (`AUDIT_EXPORT_MAX_ROWS`, EAP-6), with no indication in the response that the export was truncated.** A deliberate, minimal first pass тАФ the cap itself is the real control (unbounded export is the actual risk); a truncation signal (e.g. a response header, or an error instead of a silent partial file) is real, recommended follow-up (`ROADMAP.md`) rather than built speculatively for a data volume (10,000+ audit events in one filtered query) this local-dev system has never approached.
- **`GET /api/operations/summary`'s `ServiceStatus.error` returns a caught exception's real message text to the caller (EAP-7), a deliberate, scoped exception to this codebase's general "never leak internal errors to a client" rule.** Full reasoning in "Authorization model" above тАФ accepted because only a Platform Administrator can ever reach it and showing the real failure cause is this endpoint's actual purpose, not an incidental leak.
- **Runtime metrics (`GET /api/operations/summary`'s `requestCounts`/`repositoryOperations`, EAP-7) are per-Worker-isolate, not global or durable тАФ the same limitation `security/rateLimiter.ts` already documents for the identical reason.** Counters reset to empty whenever an isolate restarts; the Operations Center's own empty-state copy says this rather than implying "zero requests, ever." A real cross-isolate metrics backend needs Workers Analytics Engine, which needs a deployed Cloudflare account this project has never had.
- **`SystemOverview.version` (EAP-7) is a manually-maintained literal (`PLATFORM_VERSION`, `router.ts`), not read from `package.json` at runtime.** Not a security gap, but worth recording alongside the others: this codebase has no build pipeline that injects real build metadata (a commit hash, a build timestamp) anywhere, so "build information" is deliberately not claimed by the Operations Center at all rather than fabricated. Revisit only alongside a real CI/build-info-injection decision.
- **The Audit Center has no "result status"/outcome column to filter or display, because `audit_events` has none.** Not an oversight тАФ every row this table has ever recorded is, by construction, a successful, authorized action; a denied, failed, or errored request is never written here (confirmed by reading every `recordAuditEvent` call site in `router.ts`). Surfacing a "status" column would fabricate data this schema doesn't capture. Same reasoning covers "request context" (no request id/IP/user-agent column) тАФ `AuditEventDetailPanel.tsx` says this honestly rather than leaving the field blank with no explanation.
- **Investigation View has no "investigation notes" тАФ a workstream item the EAP-6 brief itself marked optional, "only if the repository supports them."** It doesn't: `audit_events` has no notes/annotations column, and adding one would be new schema, not an additive read over existing data (unlike the Investigation View's grouping/correlation, which is a pure client-side transform of already-fetched search results). Not built this phase; real follow-up if a future phase decides this system needs annotatable investigations (`ROADMAP.md`).
