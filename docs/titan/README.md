## What's still blocked

Nothing is deployed anywhere — no Cloudflare account/credentials have existed in any environment this project has run in, so "verified" here always means "verified locally," never "verified in production." **PRD-1 built and structurally verified the deployment automation itself (`wrangler.toml` environments, CI/CD pipeline, release scripts) but did not and could not change this fact** — every command in that automation requiring real Cloudflare credentials was run and confirmed to fail with exactly the expected authentication error (`DEPLOYMENT_GUIDE.md`). R2 (object storage) and Queues (async work), named in the original Phase 2 master prompt's Cloudflare-platform recommendation, remain deliberately unwired — nothing in this codebase generates a server-side file to store or an async job to dispatch yet, so binding either now would be dead configuration; the same is true of Turnstile (bot mitigation), which no public form here has had a real evaluated need for yet. Payments provider and email provider remain genuinely undecided (`ARCHITECTURE.md`'s "Still open" list) — Auth.js's Email sign-in logs its link instead of sending real mail as a result, and server-side generated PDF reports (the pre-existing `reports` table, migrations/0006 — a different concept from EAP-8's own reporting layer, see below) remain blocked on the same undecided email/storage pipeline. Enterprise SSO remains unbuilt. A customer portal is now real (CPP-1) — the admin application has a real shell and all eight modules the original EAP-1 brief named plus this roadmap's own recommended eighth one (Dashboard, Lead Intelligence, Assessment Center, Organization Management, Enterprise Identity & User Management, Enterprise Audit Center, Enterprise Operations Center, Enterprise Reporting & Analytics), and a separate customer-facing shell now exposes a scoped subset of that same data to a real organization member. CPP-1 has its own named, honest gaps, not silent ones: a caller with more than one real organization membership resolves to the first one found — there is no organization switcher yet, a real scope limit for a later phase (`DECISION_LOG.md`'s CPP-1 entry); Support Requests have no ticketing workflow — `status` starts and, today, stays `"open"`, since there is no admin-side resolution endpoint anywhere in this codebase yet; the Account page shows profile and session/sign-out only — password change, notification preferences, and multi-session management all require services this platform doesn't have yet (a credentials provider, a settings/notification service, a `SessionRepository` beyond Auth.js's own single-session-token model) and are not fabricated as if they existed. **COM-1's Commercial Platform is real but explicitly provider-agnostic — there is no payment gateway integration anywhere in this system, no card/invoice data, and no real chargeable transaction has ever occurred**: `priceDisplay` on every plan is display text, never a processed amount. Its own named, honest gaps: entitlements are computed and displayed (a real, correct answer to "what does this organization's plan grant") but not yet enforced as a gate on any existing route — wiring enforcement into the Assessment/Reporting Engine or the pre-existing Customer Portal routes is out of this phase's scope (both are named do-not-redesign systems), so this is real information, not yet a real restriction; a Subscription is 1:1 with an Organization (no multi-subscription-per-tenant concept); the sales-assisted "enterprise" plan cannot be self-service-subscribed (`isSelfServicePlan`) — only a Platform Administrator can assign it, via the admin override; and renewal always extends the period by a fixed 30 days server-side, not a caller-chosen term. **Enterprise Reporting & Analytics (EAP-8) does not read from or write to the pre-existing `reports` table** despite the similar name — that table tracks generated PDF report artifacts (blocked on the undecided email/storage pipeline above); EAP-8 instead composes a live Executive Dashboard/Business Reports/Analytics view directly from every prior module's own existing data (`DECISION_LOG.md`'s EAP-8 entry). Lead assignment still only supports "assign to me"/unassign in its own UI — the user directory it needed is now real (EAP-5), but a real assignee picker isn't wired into `LeadLifecyclePanel.tsx` yet, a named near-term follow-up, not a silent gap (`SECURITY_GUIDE.md`). The Assessment Center has no lifecycle/status workflow — an assessment is one immutable row per completed run, not a gap, a deliberate reflection of the real data model (`DECISION_LOG.md`'s EAP-3 entry). The Audit Center's export is capped at 10,000 rows with no truncation signal yet — the cap itself is the real control, a named near-term follow-up (`SECURITY_GUIDE.md`'s known gaps); Report Export (EAP-8) needs no equivalent cap, since its output is bounded by known enum values and observed free-form keys, not by record count. The Operations Center's runtime metrics are per-Worker-isolate only, not a durable global history — the same limitation the rate limiter already has, for the identical reason (no deployed Cloudflare account to back a real Analytics Engine binding). Both of ERP-1's named findings (`GET /api/leads` authentication, server-side risk-score recomputation) are fixed — see above; the current release-classification gates are Gate C (Operations, nothing deployed anywhere) and Gate E (Commercial, no payments/billing) rather than any known, unfixed security finding. **OPS-1's own alert evaluation (`observability/alerts.ts`) is real and tested but never paged to a human** — no deployed environment produces real traffic to evaluate, and no paging provider (PagerDuty/Slack/email) has ever been chosen, the identical "no forcing decision yet" reasoning `ARCHITECTURE.md`'s payments/email provider gaps already have (`MONITORING_GUIDE.md`). Incident response, SLOs, and capacity planning (`INCIDENT_RESPONSE_GUIDE.md`, `SRE_GUIDE.md`) are real, usable frameworks, not evidence of a real incident, a measured SLO, or a real capacity constraint ever having occurred — because none has. **SEC-1's own real, deferred finding**: a fresh `npm audit` found 3 high-severity CVEs in `sharp`, required transitively by `miniflare`/`wrangler` (a devDependency, never bundled into the deployed Worker or reachable from any application code — confirmed absent from both `apps/web/src` and `packages/platform/src`). No fix is available without downgrading `wrangler` by roughly 97 minor versions (`npm audit fix --force`'s own proposed change, `THREAT_MODEL.md`'s findings register) — deferred with real reasoning, not silently ignored. GitHub Environment protection rules for `titan-deploy.yml` remain unconfigured (a repository setting, carried forward from PRD-1, still real and still actionable).

## Why this is eight documents plus a matrix, not the nine-folder structure originally requested

Each folder in the original brief (`/docs/product`, `/docs/api`, `/docs/security`, `/docs/testing`, `/docs/operations`, `/docs/analytics`, `/docs/growth`) needs real content to justify existing. Right now there's enough real material for product vision, architecture, a decision log, roadmap, implementation status, a feature matrix, and a developer guide — not yet for API docs (no API exists), security baseline (no backend to secure), or a testing guide beyond what `DEVELOPER_GUIDE.md` already covers. New documents get added when the phase that needs them starts, not preemptively.
