# SEO Validation Report

Phase 1.0.5 deliverable. Generated by actually running the validation
engine (`src/seo/reports/generateReport.ts`) against the real Phase 1.0
data model (`src/seo/config/`) — every number below is genuine output, not
an estimate. The validation engine only **reads** `src/seo/config/` and
`src/seo/types/`; nothing in Phase 1.0 was modified to produce this report
(see `SEO_DATA_MODEL.md`, still accurate and untouched).

## Executive Summary

16 validators examined every config file, 119 findings total: **0
errors, 106 warnings, 13 info**. Zero errors means the data model has full
structural integrity — no duplicate ids, no broken cross-references, no
mismatched canonicals, no malformed CTA targets, no circular category
chains, anywhere in the model. Every one of the 119 findings is a
**completeness gap or an observation**, not a defect: fields Phase 1.0
correctly left unset because no real content existed yet (documented at
the time in `SEO_DATA_MODEL.md`'s "Known risks" section), now
systematically inventoried with exact counts instead of prose memory.

The single most consequential finding: of the seven commercial fields
Phase 1.0.5 was asked to check (Primary Keyword, Secondary Keywords,
Intent, Audience, Business Objective, Commercial Priority, Primary CTA),
**Business Objective and Commercial Priority are at 0% coverage across
every single entity in the entire model** — 17 pages, 5 products, 6
services, 5 solutions, zero exceptions. Audience and Primary CTA are 0%
at the page level specifically. This is the clearest, most actionable
signal in the whole report for whoever plans Phase 1.4 (Commercial SEO).

## Pass/Fail Summary

**PASS** — no error-severity finding exists anywhere in the model. The
entity-identity layer (ids, urls, canonicals, KG entities/relationships,
CTA target schemes, category hierarchies) is internally consistent across
all 17 pages, 5 products, 6 services, 5 solutions, 3 authors, 3 articles,
7 categories, and the derived 24-node Knowledge Graph. This was verified,
not assumed: `validateRelationships` re-resolves every `relatedX` /
`authorId` / `categoryIds` / `parentCategory` / `brand` reference in the
model against its real target collection, and `validateKnowledgeGraph`
independently re-resolves every derived relationship endpoint — both
report zero unresolved references.

The 106 warnings and 13 info findings do not block Phase 1.1 — they are
the honest content/coverage backlog Phase 1.0 always intended to surface
once tooling existed to measure it.

## Migration Readiness

**7 of 17 static pages (41%) have no navigation path leading to them** in
either `PRIMARY_NAVIGATION` or `FOOTER_NAVIGATION`: `contact`,
`dark-web-monitor`, `item`, `platforms`, `pricing`, `status`,
`threat-intel`. (`validateRelationships` normalizes `/index.html` ↔ `/`
before computing this — the home page's nav entry uses the literal string
`/index.html` while the page registry's root path is `/`; without that
normalization the home page itself would have misreported as orphaned.
That representational mismatch is itself worth fixing whenever
`navigation.config.ts` is next touched, even though this report doesn't
modify it.)

This doesn't mean these 7 pages are unreachable on the live site — users
may reach them via in-page buttons/links this SEO-facing nav model doesn't
capture — but it does mean Phase 1.3 (Internal Linking Engine) has a
concrete, evidenced worklist rather than a guess.

`item.html` remains the one page with no `description` and no
`canonical` — carried forward unchanged from Phase 1.0, now
mechanically re-verified rather than remembered.

## Knowledge Graph Health

24 entities (1 organization, 1 brand, 5 products, 6 services, 5 solutions,
3 authors, 3 articles), 0 duplicate entity ids, 0 duplicate
(type, refId) pairs, 0 unresolved relationship endpoints.

- **3 orphaned entities** (zero relationships touching them in either
  direction): `kg-product-blog` ("Research Blog & Advisories"),
  `kg-solution-iso_tool` ("Compliance Starter Pack ISO 27001"),
  `kg-solution-zt_tool` ("Zero Trust Blueprint NIST 800-207"). None of
  the three populate `relatedServices`/`relatedProducts`/`relatedSolutions`
  in their own config entry, and nothing else references them either.
- **`Solution` never appears as a relationship source** — confirmed by
  reading `knowledge-graph.config.ts`'s own relationship-builder code,
  not inferred: it constructs edges from `PRODUCTS[].relatedServices`,
  `SERVICES[].relatedProducts`, `SERVICES[].relatedSolutions`, and
  `BLOG_ARTICLES[].authorId` — never from `SOLUTIONS[].relatedProducts`
  or `SOLUTIONS[].relatedServices`, even though both fields exist on
  `SEOSolution` and are genuinely populated on 3 of the 5 solutions.
- **3 asymmetric edges** — `ai_tool` (→ `owasp`), `red_tool`
  (→ `pentest`), and `dpdp_tool` (→ `dpdp`) each set their own
  `relatedServices`, but the corresponding graph edge only exists because
  the *Service* side happens to independently record the same
  relationship via `relatedSolutions`. If a future edit ever changed one
  side without the other, the graph would silently reflect only the
  Service's version — invisible today only because both sides currently
  agree.
- **No `Category` Knowledge Graph entities exist** — `KnowledgeGraphEntityType`
  includes `"Category"` and 7 real `SEOCategory` records exist across
  `blog.config.ts` (3) and `research.config.ts` (4), but
  `knowledge-graph.config.ts` never constructs Category nodes or the
  `SEOArticle.categoryIds` relationships that would connect to them.
- **`Organization` and `Author` also never appear as a relationship
  source** — expected and benign (an Organization/Author is conceptually
  something things relate *to*, never something that relates *outward*),
  listed for completeness since the check is generic and type-agnostic,
  not because it indicates a problem the way the `Solution` finding does.

None of the above are modifications to `knowledge-graph.config.ts` —
Phase 1.0.5's own rule is to validate the data model, not change it. They
are handed to whoever builds Phase 1.2 (Knowledge Graph / Schema
generation) as a precise worklist instead of a fresh audit.

## Relationship Health

Every reference in the model resolves. Checked and confirmed clean:
`SEOPage.relatedEntityIds` (against the combined pool of product/service/
solution/article ids), `relatedServices`/`relatedSolutions`/
`relatedProducts` in every direction across Products/Services/Solutions,
`SEOArticle.authorId` and `.categoryIds`, `SEOCategory.parentCategory`
(both namespaces — blog and research — including a cycle detector; 0
cycles exist because no category currently sets `parentCategory` at all),
and `SEOOrganization.brand`.

**5 entities are never referenced by anything** (no page's
`relatedEntityIds`, no other entity's `relatedX` field):
`iso_tool`, `zt_tool` (solutions — the same two flagged as Knowledge
Graph orphans above, from the raw-config side rather than the derived
graph), and all 3 blog articles (`dirtyclone-cve-2026-43503`,
`bluekit-aitm-phishing-report`, `owasp-llm-prompt-injection-mapping`). The
`research` page links only to the `blog` *product*, never to individual
articles — a real, evidenced gap for Phase 1.3's internal-linking engine
to close, not a defect in this phase.

**7 pages are orphaned from navigation** — see Migration Readiness.

## Configuration Health

- **1 cross-collection id collision**: `"vciso"` is used as both a
  `PAGES` id and a `SERVICES` id. This is harmless today —
  `SEOPage.relatedEntityIds: ["vciso"]` on the `vciso` page correctly
  resolves to the `vciso` *service* (verified by `validateRelationships`
  finding zero unresolved references) — but it is a real, latent
  ambiguity: `relatedEntityIds` is a flat `string[]` with no
  collection/type discriminator, so any future code walking that array
  cannot distinguish "the vciso page" from "the vciso service" by id
  alone. Worth a naming convention (e.g. collection-prefixed ids) before
  Phase 1.3 builds an internal-linking engine on top of this field.
- **0 empty collections** (excluding `RESEARCH_ARTICLES`, which is
  deliberately and documentedly empty).
- **5/5 checked `SERVICES.url` values resolve to a real, matching
  `PAGES.path`** (`soc`, `dpdp`, `owasp`, `vciso`, `pentest`) — a pass
  worth stating explicitly: it confirms the static-page ↔ SPA-view
  reconciliation documented in `SEO_ARCHITECTURE.md`'s Decision Record
  actually holds true in the data, not just on paper.

## Future Recommendations

1. **Phase 1.4 (Commercial SEO)**: populate `businessObjective` and
   `commercialPriority` first — they are the only two fields at a clean,
   universal 0%, making them the least ambiguous starting point.
2. **Before Phase 1.3 (Internal Linking)**: resolve the `"vciso"` id
   collision (Configuration Health) and decide how individual blog
   articles should be linked from pages, since none are today
   (Relationship Health).
3. **Before or during Phase 1.2 (Knowledge Graph / Schema generation)**:
   extend `knowledge-graph.config.ts`'s relationship builder to also read
   `SOLUTIONS[].relatedProducts`/`relatedServices` and
   `PRODUCTS[].relatedSolutions`, and add `Category` entities/edges for
   the 7 existing `SEOCategory` records — both are pre-existing gaps in
   Phase 1.0's derivation logic that this validation pass surfaced but,
   per this phase's own scope, did not fix.
4. **Content backlog**: an image for every product/service/solution/
   author/article (18 entities, currently 0 covered), and
   page-specific OpenGraph images to replace the one shared default.
5. **Fix the `/index.html` vs `/` string mismatch** in
   `navigation.config.ts`'s gateway link whenever that file is next
   edited — harmless today only because this validator normalizes it
   explicitly (see Migration Readiness).
6. Re-run `generateValidationReport()` after each future phase — it costs
   nothing (pure, in-memory, sub-second) and turns "did we regress
   anything" into a mechanical check instead of a manual review.
